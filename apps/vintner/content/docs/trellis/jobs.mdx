# Job System

Every infrastructure operation in Trellis is executed as a **job**. Jobs are queued in PostgreSQL and claimed by Tendrils for execution. The web UI shows real-time progress via Supabase Realtime.

## Job Types

| Type | Triggered By | What It Does |
|------|-------------|--------------|
| `CONNECTION_TEST` | Connecting a cloud provider | Verifies credentials, discovers existing resources |
| `FETCH_RESOURCES` | Refreshing cached resources | Enumerates VPCs, subnets, hosted zones, etc. |
| `BOOTSTRAP` | Initial workspace setup | Creates infrastructure workspace |
| `PLAN` | Clicking "Plan" on a vine | Runs `terraform plan` + optional Infracost analysis |
| `DEPLOY` | Clicking "Apply" on a vine | Runs `terraform apply` + installs ArgoCD |
| `DESTROY` | Clicking "Destroy" on a vine | Runs `terraform destroy` with cleanup |
| `DEPLOY_WORKER` | Adding a new Tendril | Provisions ECS task definition, IAM roles, CloudWatch |
| `DESTROY_WORKER` | Removing a Tendril | Tears down Tendril infrastructure |

## Job Lifecycle

```
QUEUED ──► CLAIMED ──► PROCESSING ──► SUCCESS
                          │
                          ├──► FAILED
                          │
                          └──► CANCELLED
```

| Status | Meaning |
|--------|---------|
| `QUEUED` | Waiting for a Tendril to pick it up |
| `CLAIMED` | A Tendril has atomically claimed it (prevents double-execution) |
| `PROCESSING` | Tendril is actively running (Terraform in progress) |
| `SUCCESS` | Completed successfully |
| `FAILED` | Failed with error (message attached) |
| `CANCELLED` | User cancelled before completion |

## Jobs List Page

The jobs page (`/dashboard/jobs`) shows all jobs with:

- Job type badge (color-coded)
- Status badge (green = SUCCESS, red = FAILED, yellow = PROCESSING, gray = QUEUED)
- Target vine name
- Assigned Tendril
- Timestamps (created, started, completed)
- Duration

Filtering by status and type is available.

## Job Detail Page

Clicking a job opens the detail page (`/dashboard/jobs/[id]`) with:

### Real-Time Log Streaming

The log viewer shows Terraform output as it happens:

1. Tendril batches log lines and POSTs them to `/api/jobs/[id]/logs`
2. Lines are inserted into `provision_job_logs`
3. The browser subscribes to INSERT events on this table (filtered by job_id) via Supabase Realtime
4. New log batches appear instantly in the viewer

Logs are separated by stream type:
- **STDOUT** — normal Terraform output (plan details, resource creation progress)
- **STDERR** — errors and warnings
- **SYSTEM** — platform messages (credential assumption, phase transitions)

### Plan Results

For PLAN jobs, the detail page additionally shows:
- **Resource changes** — count of resources to add, change, destroy
- **Cost breakdown** — Infracost analysis (if available)
- **Apply button** — creates a DEPLOY job referencing this plan

### Deploy Results

For DEPLOY jobs, after completion:
- **Cluster metadata** — name, endpoint, ARN
- **ArgoCD URL** — link to the installed ArgoCD dashboard
- **Execution timing** — breakdown of phases

## Tendril Select Popover

When triggering a Plan, Apply, or Destroy action, the **Tendril Select Popover** appears:

- Lists all registered Tendrils with their current status
- **Green dot** = ONLINE (heartbeat received within 60 seconds)
- **Yellow dot** = DRAINING (finishing current job, accepting no new ones)
- **Red dot** = OFFLINE (missed heartbeat)
- "Any available" option for automatic assignment

The user selects a Tendril (or "Any available") and confirms. The job is created with the selected `worker_id` (or `null` for automatic assignment).

## Plan Hash Validation

When applying a plan, the DEPLOY job references the PLAN job's artifact. Before execution, the Tendril validates that the vine's configuration hash matches the plan's configuration hash. This prevents applying a stale plan after the vine configuration has been modified.

If the hash doesn't match, the job fails with an error message asking the user to re-plan.

## Job Cancellation

Running jobs can be cancelled from the UI or CLI. Cancellation:
1. Sets the job status to `CANCELLED`
2. The Tendril detects the status change on its next status check
3. Terraform process is terminated gracefully
4. Partial state is preserved in the Terraform state file

## Job Retry

Failed jobs can be retried. This creates a new job with the same configuration, rather than retrying the original job. The new job starts fresh from QUEUED.
