# Device Authorization Grant

## What the standard is

The **OAuth 2.0 Device Authorization Grant** ([RFC 8628](https://www.rfc-editor.org/rfc/rfc8628))
authenticates a client that cannot easily host a browser or receive a redirect — a CLI, a TV, an IoT
device. The device asks the authorization server for a **device code** plus a human-readable
**user code**, tells the user to open a **verification URI** and enter the code, then **polls** a
token endpoint until the user approves (`authorization_pending` / `slow_down`) or the code expires.
The user authenticates in a full browser on another device; the CLI never handles their password.

## How Alethia implements it

`alethia login` (and the guided `alethia init`) run a browser-approval flow in this spirit, with a
couple of deliberate deviations from the literal RFC — documented below so the behaviour is
auditable. The CLI code lives in `apps/cli/cmd/login.go`; the control-plane routes are under
`apps/console/app/api/auth/cli/`.

<Steps>

### Start
`alethia login` generates a `device_code` (a client-side **UUID**, `uuid.New()`), opens
`${web-origin}/cli/login?device_code=<uuid>` in the browser, and starts polling.

### Approve in the browser
The `/cli/login` page (`apps/console/app/(private)/cli/login/page.tsx`) requires an authenticated
Better Auth session and `POST`s to `/api/auth/cli/generate`. That route upserts a `cli_logins` row
binding the `device_code` to `session.user.id`, and best-effort stashes the user's first linked git
provider token (held transiently in `verification_code`).

### Poll to exchange
The CLI polls `POST /api/auth/cli/exchange` with `{ device_code }` every 2 seconds. A **404** means
"pending" (`pollForToken` keeps waiting); any other non-2xx is fatal.

### Mint and store
Once the row shows a `profile_id`, `exchange` **deletes the record** (single use) and mints two JWTs
with the `jose` library, signed HS256 with `CLI_JWT_SECRET`:

- `access_token` — `type: "access"`, **1 hour** TTL.
- `refresh_token` — `type: "refresh"`, **90 day** TTL.

Both are written to `~/.config/alethia/credentials.json` (`os.UserConfigDir()/alethia/credentials.json`,
mode `0755` dir). The git provider token rides back as `provider_token`.

### Refresh
Before each authed call, `getAuthTokenInternal` (`apps/cli/cmd/auth_utils.go`) parses the access
token's `exp`; if it is within 1 minute of expiry it calls `POST /api/auth/cli/refresh` with the
refresh token, which verifies `type === "refresh"` and issues a fresh 1-hour access token. When the
refresh token is gone or invalid, the CLI re-runs the browser flow.

</Steps>

<Callout type="warn">
  **Deviations from RFC 8628 (be precise).** Alethia's flow is *modeled on* the device grant but is
  not a literal implementation:
  <br />• There is **no separate short `user_code` the human types** — the `device_code` is a
  client-generated UUID passed directly in the browser URL, and approval is a click, not a code entry.
  <br />• Pending status is signalled with **HTTP 404**, not the RFC's `authorization_pending` /
  `slow_down` token-endpoint errors; the poll interval is a fixed 2s.
  <br />• The CLI tokens are **HS256 JWTs** with placeholder `iss`/`aud` (`urn:example:issuer` /
  `urn:example:audience`), distinct from the Better Auth web session. See [JWT / JWS](/standards/jose).
</Callout>
