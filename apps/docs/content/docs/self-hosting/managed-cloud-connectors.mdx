# Managed cloud connectors — operator setup

Your customers connect their cloud accounts from **Connectors** in the console. For that to work, your
instance needs a **platform identity** the control plane authenticates as to assume/federate into the
customer account. This page is the operator side; the per-cloud *customer* flows live under
[Connectors](/console/connectors).

Until you set it, managed cloud tiles read **"Not enabled on this instance."** Token clouds
(Hetzner / DigitalOcean / Civo) need nothing here — they use the customer's own API token.

## The model: everything hubs through one AWS identity

There is a single root of trust — a **platform AWS identity** — and every managed cloud federates from
it:

| Cloud | How it uses the platform AWS identity |
|---|---|
| **AWS** | assumes the customer role: `sts:AssumeRole` on `AlethiaProvisionerRole-<externalId>` (ExternalId-gated) |
| **GCP** | the customer Workload-Identity pool trusts an **AWS provider** for your account; the console signs the WIF subject token with the AWS creds (google-auth's `--aws` source). **No separate GCP secret.** |
| **Azure** | the customer app's federated credential trusts `sts.amazonaws.com` with your AWS account as subject |

So **setting the AWS identity lights up AWS and GCP together.** (This is why, in development, a single
AWS credential was enough to connect AWS *and* GCP — the dev machine's ambient AWS creds signed the GCP
token.)

## AWS + GCP — set the platform AWS identity

### 1. Create the identity (IaC)

```bash
cd infra/connector-platform/aws
cp backend.hcl.example backend.hcl
tofu init -backend-config=backend.hcl
tofu apply        # creates alethia-connector-assumer + a least-priv sts:AssumeRole policy
```

The module creates an IAM **user** allowed **only** `sts:AssumeRole` on
`arn:aws:iam::*:role/AlethiaProvisionerRole-*`. It does **not** create an access key (so no secret lands
in state) — you make it in the next step. A `check` block refuses an apply in the wrong account.

*(Manual alternative: create an IAM user, attach a policy allowing `sts:AssumeRole` on
`arn:aws:iam::*:role/AlethiaProvisionerRole-*`.)*

### 2. Wire the key

```bash
aws iam create-access-key --user-name alethia-connector-assumer
```

Put it in `deploy/prod/secrets.local.env`:

```bash
ALETHIA_AWS_ACCESS_KEY_ID=<AccessKeyId>
ALETHIA_AWS_SECRET_ACCESS_KEY=<SecretAccessKey>
# ALETHIA_AWS_ACCOUNT_ID defaults to 270587882865 — the account customers trust. Override only if you
# run the platform in a different AWS account (it flows to the customer setup templates + GCP/Azure trust).
```

Then push it to the vault and redeploy:

```bash
./scripts/bootstrap-secrets.sh
gh workflow run deploy-console.yml
```

`deploy-console` also exports the creds under the standard `AWS_*` names so google-auth can mint the GCP
subject token — this is the step that was missing in production.

### 3. Verify

In the console → **Connectors**, the **AWS** and **GCP** tiles now show **Connect**. Connect a test AWS
account and a GCP project (customer side: [AWS](/console/connectors/aws) ·
[GCP](/console/connectors/gcp)); both should verify to **Connected** (a server-side probe — no runner).

Check each hop if a tile stays "Not enabled":

```bash
# vault has the key?
aws secretsmanager get-secret-value --secret-id alethia/prod/env --region eu-central-1 \
  --query SecretString --output text | jq -r '.ALETHIA_AWS_ACCESS_KEY_ID'
# box .env has AWS_* (not just ALETHIA_AWS_*)?
ssh root@<DEPLOY_HOST> "grep -c '^AWS_ACCESS_KEY_ID=' /opt/alethia/.env"
```

## The platform account id

One account is the hub for all three clouds. It defaults to Alethia's live account **270587882865** and
is **configurable** — set `ALETHIA_AWS_ACCOUNT_ID` (self-hosters running their own platform account).
It flows to the console UI (`NEXT_PUBLIC_ALETHIA_AWS_ACCOUNT_ID`), the customer AWS CloudFormation trust,
the GCP WIF provider (`create-aws --account-id=…`), and the Azure federated-credential subject — so they
must all agree. The `infra/connector/{aws,gcp,azure}` scripts read the same variable.

## Azure and Alibaba — status

- **Azure**: the console health probe currently authenticates as a **client-secret** app
  (`ALETHIA_AZURE_TENANT_ID/CLIENT_ID/CLIENT_SECRET`), but the customer setup script grants a
  *federated* app — so in-console Azure **verification** is a known follow-up (provisioning via the
  runner works). Set the client-secret app creds if you want the console to attempt it; a fix to
  federate the console through the AWS hub (no Azure secret) is planned.
- **Alibaba**: mirrors AWS (a platform RAM user with `sts:AssumeRole`) — set
  `ALETHIA_ALIBABA_ACCESS_KEY_ID/SECRET`. Needs an Alibaba Cloud account.
