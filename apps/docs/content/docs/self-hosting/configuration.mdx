# Configuration

All configuration is environment variables in `.env` (copy from `.env.example`).
The installer generates the secret values; the rest you set. Internal service
URLs (Postgres, SeaweedFS) are wired by compose — you don't set those.

## Ingress

| Key | Notes |
|---|---|
| `ALETHIA_DOMAIN` | Your domain; Caddy gets TLS for it. Empty → plain HTTP on `:80`. |
| `ALETHIA_ACME_EMAIL` | Let's Encrypt contact email. |
| `NEXT_PUBLIC_APP_URL` | Public origin, e.g. `https://alethia.example.com`. |

## Secrets (generated by the installer)

| Key | Generate with |
|---|---|
| `CLI_JWT_SECRET` | `openssl rand -hex 32` — signs CLI/runner tokens (shared with the runner). |
| `BETTER_AUTH_SECRET` | `openssl rand -hex 32` — signs sessions/cookies. |
| `ALETHIA_DB_PASSWORD` / `ALETHIA_APP_DB_PASSWORD` | strong random — bundled Postgres roles. |
| `ALETHIA_CRED_ENCRYPTION_KEY` | `openssl rand -base64 32` — AES-256-GCM key encrypting api-key [connector](/console/connectors/pluggable) secrets (Cloudflare/Vault/Docker Hub/…). Required to connect pluggable connectors. |

## Database & storage

Defaults target the bundled `postgres` + `seaweedfs`. To use a managed Postgres or
S3/R2 instead, point `ALETHIA_DATABASE_URL` / `ALETHIA_STORAGE_*` at them and set
`ALETHIA_STORAGE_AUTO_CREATE_BUCKETS=false`.

| Key | Notes |
|---|---|
| `ALETHIA_DB_USER` / `ALETHIA_DB_NAME` | bundled Postgres user/db |
| `ALETHIA_STORAGE_ENDPOINT` | S3 endpoint (bundled SeaweedFS by default) |
| `ALETHIA_STORAGE_ACCESS_KEY_ID` / `_SECRET_ACCESS_KEY` | must match `deploy/seaweedfs/s3.json` for the bundle |

## Email (AWS SES)

Set `ALETHIA_SES_REGION` to enable sending; with none, sign-in/account emails are
logged to the console instead of sent. From-addresses must be verified SES identities.

| Key | Notes |
|---|---|
| `ALETHIA_SES_REGION` | SES region, e.g. `eu-central-1`. Enables sending. |
| `ALETHIA_SES_ACCESS_KEY_ID` / `_SECRET_ACCESS_KEY` | Optional; omit to use the AWS default credential chain (IAM role / `AWS_*`). |
| `AUTH_EMAIL_FROM` | Auth/security from address, e.g. `Alethia <no-reply@auth.yourdomain>` (quote it). |
| `EMAIL_FROM` | Product/general from address; falls back to `AUTH_EMAIL_FROM`. |

## OAuth providers (optional)

Only the providers you configure appear on the sign-in screen.

| Key |
|---|
| `GITHUB_CLIENT_ID` / `GITHUB_CLIENT_SECRET` |
| `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` |
| `GITLAB_APPLICATION_ID` / `GITLAB_SECRET` |
| `BITBUCKET_KEY` / `BITBUCKET_SECRET` |

Set each provider's callback to `https://<your-domain>/api/auth/callback/<provider>`.

## Cloud connectors (optional)

| Key | Default | Notes |
|---|---|---|
| `NEXT_PUBLIC_CONNECTOR_ASSETS_ORIGIN` | hosted bucket | Origin serving the connector setup artifacts the AWS/GCP/Azure connect flows hand out (`alethia-bootstrap.yaml`, `alethia-{gcp,azure}-setup.sh`). Point it at your own console origin — the files ship in `apps/console/public/`, e.g. `https://alethia.example.com`. The AWS CloudFormation quick-create flow needs an S3-hosted template, so AWS uses the bucket default or the Terraform method. |

## Managed runner fleet (optional)

By default, runners are static — you register them yourself (see [Runners](/console/runners)).
Set `FLEET_PROVIDER=hcloud` to let the in-app scaler provision warm runner VMs on Hetzner Cloud and
scale them by queue depth. Unset / `manual` → no managed fleet (the scaler is a no-op).

| Key | Default | Notes |
|---|---|---|
| `FLEET_PROVIDER` | `manual` | `hcloud` provisions real Hetzner VMs; `manual` = no-op. |
| `HCLOUD_TOKEN` | — | Hetzner Cloud API token. Required when `FLEET_PROVIDER=hcloud`. |
| `ALETHIA_RUNNER_BOOTSTRAP_TOKEN` | — | Low-privilege shared token fleet VMs present to self-register. |
| `ALETHIA_WEB_ORIGIN` | — | Console URL the VMs call back to (must be reachable from them). |
| `HCLOUD_SERVER_TYPE` | `cax21` | Hetzner server type (ARM CAX recommended for runner workloads). |
| `HCLOUD_IMAGE` | `ubuntu-24.04` | Base OS image for fleet VMs. |
| `HCLOUD_SSH_KEYS` | — | Comma-separated Hetzner SSH key IDs. |
| `FLEET_RUNNER_IMAGE_TAG` | `latest` | Runner image tag pulled from GHCR. |
| `FLEET_RUNNER_SLOTS` | `1` | Concurrent job slots per fleet runner. |
| `FLEET_BOOT_GRACE_SECONDS` | `180` | Seconds a freshly-booted VM is exempt from scale-down (cloud-init grace). |

Warm pools are managed in the console **Fleet** page (DB-backed). `FLEET_POOLS` is **deprecated** —
imported once into an empty pools table on first boot, then ignored.

> A complete production template lives at `deploy/prod/.env.production.example`.
