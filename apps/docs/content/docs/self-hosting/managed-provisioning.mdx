# Managed provisioning — operator runbook

With the **managed** operator, your instance runs the provisioning runners itself: a cheap Hetzner VM fleet claims each
customer's plan/deploy job, assumes into the customer's cloud (keyless OIDC, or a token for Hetzner), runs
OpenTofu against Alethia's own project templates, and streams state + logs back through the console. The
customer never runs a runner.

All five clouds — **AWS, GCP, Azure, Alibaba, Hetzner** — provision end-to-end. This page is the operator
checklist to enable it and verify each cloud. The per-cloud *customer* connect flow and the OIDC-issuer
platform setup live in [Managed cloud connectors](/self-hosting/managed-cloud-connectors); this page assumes
you've done that and covers the **fleet** side.

## Architecture (what happens on a deploy)

1. A customer connects a cloud (keyless OIDC for AWS/GCP/Azure/Alibaba; an API token for Hetzner) and
   creates a project, then queues a **plan** or **deploy**.
2. The in-app **scaler** (`lib/fleet`, 60s tick) sizes a per-provider warm pool and creates a Hetzner VM.
   Its cloud-init runs the per-cloud runner image `ghcr.io/alethialabs-io/runner-<cloud>` and the VM
   **self-registers** with a per-VM, short-TTL, instance-bound bootstrap token.
3. The runner claims the job, mints a **per-job cloud token** (OIDC) and a **per-job tofu-state token**, and
   runs `tofu init/plan/apply` against the baked project template. Tofu **state** goes to the console HTTP
   state proxy (`/api/jobs/[id]/state`) — no storage credential ever touches the runner.
4. The kubeconfig comes from the cloud (EKS/GKE/AKS API) or a tofu output (ACK/Talos), post-apply Helm/ArgoCD
   runs, and the job completes. The VM is disposable (SLOTS=1).

The fleet holds **no** storage master key and **no** shared bootstrap token — both were drained from the
cloud-init (E0 2b/0b). Untrusted **BYO** execution additionally requires the container sandbox — see
[Untrusted BYO](#untrusted-byo-container-sandbox-3b) at the end.

## Prerequisites

- A working self-hosted deploy (console + migrate + storage reachable). See [Configuration](/self-hosting/configuration).
- The **OIDC issuer** signing key `ALETHIA_OIDC_SIGNING_KEY` — auto-generated by `scripts/bootstrap-secrets.sh`.
  Its discovery + JWKS routes (`/api/oidc/...`) must be **publicly reachable** (Azure/Alibaba validate the
  assertion). This is set up in [Managed cloud connectors](/self-hosting/managed-cloud-connectors).
- Object storage configured (`ALETHIA_STORAGE_*` on the **console** — NOT the fleet).

## 1. Per-cloud connector setup (one-time)

Do this once per cloud you want to offer. Full detail in
[Managed cloud connectors](/self-hosting/managed-cloud-connectors); summary:

| Cloud | Platform operator step | Customer step |
| --- | --- | --- |
| **AWS** | none (issuer key only) | `tofu apply infra/connector/aws` → paste **role ARN** |
| **GCP** | none (issuer key only) | `tofu apply infra/connector/gcp` → paste **credential config** |
| **Alibaba** | none (issuer key only) | `tofu apply infra/connector/alibaba` → paste **role ARN** |
| **Azure** | register one multi-tenant Entra app + federated credential; set `ALETHIA_AZURE_CLIENT_ID` | `tofu apply infra/connector/azure` |
| **Hetzner** | none | paste a Hetzner **API token** (token auth — no OIDC) |

The connect UI for each provider only lights up once its platform prerequisite is satisfied
(`computePlatformConfigured`): AWS/GCP/Alibaba track the issuer signing key, Azure additionally needs
`ALETHIA_AZURE_CLIENT_ID`, Hetzner is always available.

## 2. Runner images

Managed provisioning of a cloud requires its **per-cloud runner image** to be published to GHCR. The deploy
pipeline (`deploy-console.yml`) builds and pushes all five on a push to `main`:

```
ghcr.io/alethialabs-io/runner-aws       runner-gcp     runner-azure
ghcr.io/alethialabs-io/runner-alibaba   runner-hetzner
```

`runner-alibaba` and `runner-hetzner` carry no cloud CLI (Alibaba is keyless OIDC with a static kubeconfig
from the ACK output; Hetzner is Talos with a static kubeconfig from the talos output). Confirm the tags exist
for the tag your fleet pins (`FLEET_RUNNER_IMAGE_TAG`, default `latest`):

```bash
for c in aws gcp azure alibaba hetzner; do
  docker manifest inspect ghcr.io/alethialabs-io/runner-$c:latest >/dev/null && echo "runner-$c OK"
done
```

## 3. Fleet enablement

The fleet is off until you select the Hetzner provider and give it a token. Set on the **console** env:

| Var | Required | Notes |
| --- | --- | --- |
| `FLEET_PROVIDER=hcloud` | yes | selects the Hetzner backend (default `manual` = no-op) |
| `HCLOUD_TOKEN` | yes | Hetzner API token for the **fleet** project (distinct from a customer connector token) |
| `HCLOUD_SSH_KEYS` | recommended | comma-separated Hetzner SSH key IDs (break-glass access) |
| `ALETHIA_RUNNER_BOOTSTRAP_TOKEN` | auto | seeded by the runner script on first run; the scaler mints per-VM tokens from it |
| `HCLOUD_SERVER_TYPE` | no | default `cax21` (arm64) |
| `FLEET_RUNNER_IMAGE_TAG` | no | default `latest` |
| `FLEET_RUNNER_SLOTS` | no | default `1` (one job per VM) |

Then create **one fleet pool per cloud** you want to provision. Pools are DB rows (`fleet_pools`), created
from the console fleet UI (self-managed deployments) — the `createFleetPool` action, gated on the `fleet`
resource. Each pool sets `provider`, `warm_min`, `max`, `locations`, etc.; the scaler re-reads enabled pools
every tick (no restart). A paused (`enabled=false`) pool is skipped.

> The fleet UI + pool reads/writes are only available on a **self-managed** deployment
> (`isFleetOperatorDeployment` / `deploymentMode() === "self-managed"`). Hosted SaaS hides it.

## 4. Per-cloud verification

For each cloud, end to end:

1. **Connect** the cloud in the console (paste the role ARN / credential config / token). The cloud identity
   verifies server-side (no runner needed).
2. **Create a project** targeting that cloud and a small cluster.
3. **Plan** — a VM spins up in that cloud's pool, claims the job, and returns a plan + a `verify.Report`
   (elench) in the console Plan tab. Confirm:
   - a `fleet-<cloud>-…` server appears in Hetzner and self-registers as a runner,
   - the plan streams logs and completes,
   - no `ALETHIA_STORAGE_*` / bootstrap secret is in the job env (they're not injected).
4. **Deploy** — apply provisions the cluster; the kubeconfig is captured (EKS/GKE/AKS API call, or the
   ACK/Talos tofu output) and post-apply Helm/ArgoCD runs.
5. **Destroy** — tears the cluster down and purges the tofu state (`DELETE …/state`).

Repeat for AWS, GCP, Azure, Alibaba, Hetzner. Alibaba provisions a real managed **ACK** cluster; Hetzner
provisions a self-managed **Talos** cluster.

### Troubleshooting

- **VM boots but never registers** — check the per-VM bootstrap token TTL (`FLEET_BOOTSTRAP_TOKEN_TTL_SECONDS`,
  default 3600) vs. a slow multi-GB image pull, and the boot grace (`FLEET_BOOT_GRACE_SECONDS`).
- **`runner-<cloud>` image pull fails** — the tag isn't published for that cloud; re-check step 2.
- **Cloud auth fails on plan** — the connector's platform prerequisite (esp. Azure `ALETHIA_AZURE_CLIENT_ID`)
  or the customer's connector `tofu apply` is missing / the issuer isn't publicly reachable.

## Untrusted BYO (container sandbox, 3b)

Everything above provisions Alethia's **own** (trusted) templates — safe to run in-process (Passthrough).
Running **untrusted** customer code (BYO Helm/IaC) on the managed fleet additionally requires the per-job
**container sandbox**: a nested rootless-podman container per job with an allowlisted env, its own PID
namespace, and a default-deny domain-allowlist egress net (so it cannot reach the `169.254.169.254` metadata
service, which serves the VM's cloud-init).

The Go backend and the fleet plumbing ship **inert**. Turn-on is **config-only** on the fleet — but the
isolation must be **proven on a real VM first**, because nested podman + `/dev/fuse` + IMDS egress cannot be
tested in CI.

### Turn-on sequence

1. **Enable the sandbox on a canary pool** — set on the fleet cloud-init env (`renderCloudInit`, gated by
   `FLEET_SANDBOX_CONTAINER=1`). This renders the default-deny egress net + squid forward proxy and starts
   the runner with the container backend + proxy env. **Do NOT set `FLEET_SANDBOX_EGRESS_ENFORCED` yet** — with
   it unset the container backend **fail-closes on managed** (jobs refuse), which proves the runtime works
   before you trust egress.

   ```
   FLEET_SANDBOX_CONTAINER=1
   FLEET_EGRESS_EXTRA_DOMAINS=<any extra chart/registry domains>   # optional
   FLEET_EGRESS_PROXY_IMAGE=ubuntu/squid@sha256:<pin>              # pin a digest in prod
   ```

2. **Run the real-VM canary** on a live Hetzner VM. All must pass:
   - `podman run --rm <runner-image> true` (as `runner`) succeeds — proves subuid/subgid + fuse-overlayfs +
     crun + the `--device /dev/fuse` + seccomp/apparmor/systempaths relaxations.
   - From inside the nested child: it **cannot** read `/proc/1/environ`, `$ALETHIA_RUNNER_TOKEN`, or the
     storage/bootstrap secrets, and sees only its own PID tree.
   - `curl --max-time 3 http://169.254.169.254/...` from the child **times out** — with the proxy env set
     AND unset (the `--internal` net + iptables belt + non-allowlisted domain all deny it).
   - Through the proxy: an allowlisted domain (console origin, `registry.opentofu.org`) succeeds; a
     non-allowlisted domain (`example.com`) is **denied 403**; a direct bypass (unset `HTTPS_PROXY`) **fails**
     (no route on the internal net).
   - A real managed **plan + deploy** on the canary completes green once egress is enforced.

3. **Flip egress enforcement** on the canary pool — add `FLEET_SANDBOX_EGRESS_ENFORCED=1`. Managed jobs now
   run in the child with proxy-only egress.

4. **Roll out** to all pools. Optionally set `FLEET_SANDBOX_ENFORCE_MANAGED=1` fleet-wide so any managed pool
   **without** the container backend refuses jobs rather than silently running unsandboxed.

5. **Enable BYO** — `ALETHIA_BYO_HELM_ENABLED=true`.

### Fleet env — 3b knobs

| Var | Meaning |
| --- | --- |
| `FLEET_SANDBOX_CONTAINER` | render the egress net + proxy + container-backend runner (master switch) |
| `FLEET_SANDBOX_EGRESS_ENFORCED` | set `ALETHIA_SANDBOX_EGRESS_ENFORCED=1` — **only after** the canary passes |
| `FLEET_SANDBOX_ENFORCE_MANAGED` | fleet-wide kill-switch: a managed pool lacking the container backend refuses |
| `FLEET_EGRESS_EXTRA_DOMAINS` | extra allowlist domains (comma-separated) |
| `FLEET_EGRESS_PROXY_IMAGE` | the forward-proxy image (pin a digest) |

### If nested podman is fragile

Nested rootless podman + fuse-overlayfs on the arm64 Hetzner kernel is the fragile layer. Fallbacks, in order:
switch the containers `storage.conf` driver to `vfs` (slower but robust) → add `--cap-add SETUID --cap-add
SETGID` to the outer run for `newuidmap` → escalate to a microVM SKU (sysbox/Kata) — a larger fleet change.

Until the canary passes, managed **untrusted** BYO stays flag-gated trusted-only; the trusted 5-cloud
provisioning above is unaffected.
