# Audit log

Every Alethia deployment keeps an **authorization audit log**, and you can browse it in **Settings → Audit
Log**. This is <Community>open-source</Community> — it works in the free edition.

## What it records

The log is a **security record, not a full activity feed**. The Policy Decision Point writes a row for:

- **Every denied authorization** — someone tried something they weren't allowed to do.
- **Sensitive allowed actions** — `destroy`, and the `manage_*` operations (identities, members, integrations,
  billing) and `export_audit`.

That keeps the log meaningful — the things worth auditing — without a row for every read.

Each entry shows the **actor**, the **action**, the **resource**, the **decision** (Allowed / **Denied**), the
**reason**, and **when**. Denied attempts are highlighted — they're the signal you usually care about. Filter
by All / Denied / Allowed.

<Callout type="info">
  The log is engine-agnostic: whether the deployment uses the Postgres engine or the
  [OpenFGA engine](/access-control/openfga), decisions are recorded the same way, so the Audit Log looks
  identical.
</Callout>

## Export

<Enterprise />

Exporting the log to **CSV** (for a SIEM, a spreadsheet, or a compliance archive) is an Enterprise feature,
gated by the `auditExport` entitlement. The **Export CSV** button appears in the header when the entitlement is
held; the export is also enforced server-side.
