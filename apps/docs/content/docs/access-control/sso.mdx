# Single Sign-On

<Enterprise />

SSO lets members sign in through **your** identity provider (IdP) instead of (or alongside) email and social
login. Alethia is the **Service Provider** that *consumes* your IdP — it never runs an identity provider of its
own.

You bring the IdP. Supported via standard protocols:

- **OIDC** — Okta · Microsoft Entra ID · Google Workspace · **AWS IAM Identity Center** · OneLogin · Ping · …
- **SAML 2.0** — the same IdPs over SAML.

<Callout type="info">
  There's nothing to run on Alethia's side — no Hydra, no Alethia-as-IdP. Each organization registers its own
  provider, so different orgs can use different IdPs.
</Callout>

## Registering a provider

In **Settings → Single Sign-On → Register provider**, choose the protocol and fill in the IdP's details:

<Tabs items={["OIDC", "SAML"]}>
<Tab value="OIDC">

| Field | What it is |
|---|---|
| Provider ID | A short slug for the provider (e.g. `okta`) |
| Email domain | The domain whose users route to this provider (e.g. `company.com`) |
| Issuer URL | Your IdP's issuer / discovery URL |
| Client ID / Client secret | The OIDC application credentials from your IdP |

</Tab>
<Tab value="SAML">

| Field | What it is |
|---|---|
| Provider ID | A short slug (e.g. `entra`) |
| Email domain | The domain whose users route to this provider |
| IdP Entity ID | Your IdP's SAML entity / issuer ID |
| SSO URL (entry point) | The IdP's SAML sign-in endpoint |
| X.509 certificate | The IdP's signing certificate |

</Tab>
</Tabs>

## Provisioning into the organization

When a member signs in through SSO for the first time, Alethia **provisions** them into the organization with a
role (least-privileged by default, or mapped from your IdP's group/role claim). From that point they're a
normal [member](/access-control/members-and-invitations) — the [PDP](/access-control/architecture) authorizes
them exactly as it would anyone else.
