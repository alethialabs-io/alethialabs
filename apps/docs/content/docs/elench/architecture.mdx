# Architecture

<Community />

Alethia is **one product**: connect your repos and get a **verified GitOps Kubernetes cluster on your own
cloud**. Two operators ‚Äî **managed** (Alethia's runner fleet assumes a role into your account) and
**self** (a runner you run) ‚Äî and two entry flows that share the same design surface, provisioner, and
proof layer:

- **"A" ‚Äî infer & generate.** Scan repos ‚Üí infer the stack ‚Üí propose a project ‚Üí provision the cluster +
  backing services + ArgoCD ‚Üí generate app manifests ‚Üí GitOps deploy ‚Üí **verify**.
- **"B" ‚Äî bring your own + audit.** Point at existing OpenTofu/Terraform or Kubernetes manifests ‚Üí
  **audit** them with the same engine ‚Üí get a signed report.

<Callout type="info">
  **Core vs Enterprise.** The generate/provision **"A" flow is core** (open-source, self-hostable). The
  **elench proof layer ‚Äî verify ¬∑ audit ¬∑ drift ¬∑ agent ¬∑ MCP ‚Äî is Enterprise.** This page is the map;
  the proof-layer pages carry the <Enterprise /> badge.
</Callout>

## The "A" pipeline

Each stage names its source-of-truth code so the docs stay honest to the implementation.

<Mermaid chart={`
flowchart LR
  repos[repos<br/>connect 1:N] --> scan[scan<br/>Go scanner ¬∑ per-svc]
  scan --> infer[infer + merge<br/>lib/scanner ‚Üí to-project]
  infer --> propose[propose<br/>canvas design]
  propose --> provision[provision<br/>RunDeployV2 ¬∑ tofu]
  provision --> generate[generate<br/>manifests .FromServices]
  generate --> gitops[GitOps<br/>ArgoCD apps ¬∑ path .]
  gitops --> verify[VERIFY<br/>elench gate + receipt]
  verify --> drift[day-2 drift<br/>DETECT_DRIFT ‚Üí Posture loop]
`} />

1. **Connect repos (1:N, monorepo-aware).** `project_source_repos` (migration `0050`) holds every scanned
   repo + its detected services.
2. **Scan.** `packages/core/scanner` clones + statically reads a repo (never executes it) into a
   `RepoDigest`, and detects per-directory **services** (Dockerfile / workspace manifests) ‚Äî so a monorepo
   yields several deployable services, not one blob.
3. **Infer + merge.** `apps/console/lib/scanner/infer.ts` turns the digest into an `InferredStack` (LLM,
   structured output); `to-project.ts` `mergeScansToFormData` folds **N repos / services** into one
   guaranteed-valid `ProjectFormData` (union backing needs, de-duped names, one `source_repos` row/repo).
4. **Propose / design.** The proposal seeds the canvas; the user reviews and edits before anything is
   persisted.
5. **Provision.** `packages/core/provisioner.RunDeployV2` runs OpenTofu, installs ArgoCD, and wires the
   GitOps repo credentials.
6. **Generate manifests.** `packages/core/manifests` renders hardened Deployment/Service/Ingress for each
   detected service and scaffolds them into the GitOps repo ‚Äî **only when that repo has no manifests
   yet** (never clobbers a bring-your-own repo). See [Multi-cloud](/elench/multi-cloud).
7. **GitOps.** ArgoCD's `apps` Application syncs the manifests repo (`path: .`, auto-sync).
8. **Verify.** Between `tofu plan` and `tofu apply`, the [verification engine](/elench/engine-internals)
   runs keyless / least-privilege / OIDC controls and seals a signed receipt ‚Äî **fail-closed**.
9. **Day-2 drift.** After apply, a scheduled `DETECT_DRIFT` job keeps re-checking (below).

## The "B" pipeline ‚Äî bring your own + audit

The verification engine is **plan-agnostic**, so auditing infrastructure Alethia *didn't* generate needs
no new controls:

<Mermaid chart={`
flowchart LR
  A[existing plan / manifests] --> B[AUDIT job]
  B --> C[verify<br/>same controls]
  C --> D[signed report]
  D --> E[VerifyBlock]
`} />

- **OpenTofu / Terraform plan:** `verify.ParseCustomerPlan(planJSON)` ‚Üí `Evaluate` ‚Äî the same AWS/GCP/Azure controls.
- **Kubernetes:** `verify.EvaluateManifests(yaml)` ‚Üí the k8s [control set](/elench/control-catalog).
- The `AUDIT` job (migration `0052`) runs on the runner (`executeAudit`) and posts the report to
  `jobs.execution_metadata.verify_result`, surfaced by the existing `get_plan_result` tool + the
  `VerifyBlock`. Queue it from the assistant (`audit_infrastructure` tool) or `queueAudit`.

## The data model

| Table / field | What it holds |
|---|---|
| `project_source_repos` (0050) | Scanned source repos (1:N) + detected services (JSONB). |
| `environment_drift` (0051) | Latest day-2 `Posture` per environment. |
| `jobs.job_type` | `ANALYZE_REPO` ¬∑ `PLAN` ¬∑ `DEPLOY` ¬∑ `DESTROY` ¬∑ `DETECT_DRIFT` ¬∑ `AUDIT` ¬∑ `*_RUNNER`. |
| `jobs.execution_metadata` | `repo_digest` ¬∑ `verify_result` (the signed report) ¬∑ `drift_posture`. |
