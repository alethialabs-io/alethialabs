# Control catalog

<Enterprise />

The current catalog is **`elench-controls-0.4.0`**. `Evaluate` detects the plan's cloud and runs that
provider's control set (a mixed or unknown plan runs all of them, so nothing is silently skipped). Every
control records its `provider`, so a uniform posture UI can never imply coverage a control does not have.

## AWS

| ID | Title | Hard-fail on |
|----|-------|--------------|
| `KEYLESS-001` | No static IAM access keys | creating an `aws_iam_access_key` (use OIDC federation instead) |
| `OIDC-001` | Federated trust bound to a specific subject | a federated (`sts:AssumeRoleWithWebIdentity`) role whose trust **lacks a `:sub` condition**, or binds `sub` only with a **`StringLike` wildcard** (the "any repo can assume" footgun) |
| `LEASTPRIV-001` | No over-broad IAM grants (named patterns) | `Action:"*"` + `Resource:"*"`; `Allow` + `NotAction`; attaching `AdministratorAccess` / `PowerUserAccess` / `IAMFullAccess`. **Warns** on sensitive service wildcards (`iam:*`, `kms:*`, ‚Ä¶) with `Resource:"*"`. **Not-evaluable** when the policy body is computed or referenced only by ARN. |
| `ACCESS-ANALYZER-001` | No sensitive actions grantable | (opt-in) a policy AWS IAM Access Analyzer proves *could* grant any action in a sensitive denylist. |

## GCP

| ID | Title | Hard-fail on |
|----|-------|--------------|
| `GCP-KEYLESS-001` | No static service-account keys | creating a `google_service_account_key` (use Workload Identity) |
| `GCP-WIF-001` | WIF providers are attribute-conditioned | a `google_iam_workload_identity_pool_provider` with an empty `attribute_condition` |
| `GCP-LEASTPRIV-001` | No project-wide primitive roles | binding `roles/owner` (fail) ‚Äî **warns** on `roles/editor` ‚Äî via `google_project_iam_member`/`_binding` |

## Azure

| ID | Title | Hard-fail on |
|----|-------|--------------|
| `AZURE-KEYLESS-001` | No static app/SP secrets | creating an `azuread_application_password` / `azuread_service_principal_password` (use a federated identity credential) |
| `AZURE-FED-001` | Federated credentials bind a subject | an `azuread_application_federated_identity_credential` with an empty or wildcard `subject` |
| `AZURE-LEASTPRIV-001` | No Owner/Contributor assignments | an `azurerm_role_assignment` of `Owner` (fail) ‚Äî **warns** on `Contributor` ‚Äî scope annotated |

## Alibaba Cloud

Alibaba has a real RAM/OIDC authority surface ‚Äî **RRSA** (RAM Roles for Service Accounts) is its
workload-identity mechanism, the analogue of AWS IRSA / GCP WIF ‚Äî so its control set mirrors the AWS
one. RAM trust and policy documents share AWS IAM's statement shape; the one Alibaba-specific
difference is the trust action (`sts:AssumeRole`, not `sts:AssumeRoleWithWebIdentity`).

| ID | Title | Hard-fail on |
|----|-------|--------------|
| `ALI-KEYLESS-001` | No static RAM access keys | creating an `alicloud_ram_access_key` (use RRSA / AssumeRoleWithOIDC instead) |
| `ALI-OIDC-001` | RRSA trust bound to a specific subject | an `alicloud_ram_role` whose Federated (RRSA) trust **lacks an `oidc:sub` condition**, or binds it only with a **`StringLike` wildcard** (any pod from the issuer can assume). Service-principal roles are out of scope; a computed trust doc ‚Üí **not_evaluable**. |
| `ALI-LEASTPRIV-001` | No over-broad RAM grants (named patterns) | an `alicloud_ram_policy` granting `Action:"*"` on `Resource:"*"` or using `Allow` + `NotAction`; or a **role/user/group** policy attachment of an admin-family System policy (`AdministratorAccess`, `AliyunRAMFullAccess` ‚Äî control of RAM itself is admin one hop away). **Warns** on a service-level wildcard (`ecs:*`, ‚Ä¶) with `Resource:"*"`. **Not-evaluable** when the policy body is computed, or for a non-admin System attachment (its body is never in the plan). |

## Hetzner

Hetzner is **token-auth** ‚Äî the API token is the ceiling, with no OIDC/federation/IAM surface to bind ‚Äî
so its controls assert **network/firewall posture**, the attack surface a Hetzner plan can actually
misconfigure. They are tuned so the shipped Talos template evaluates to **warns only** (never a
hard block of the nightly apply).

| ID | Title | Hard-fail on |
|----|-------|--------------|
| `HCLOUD-FW-001` | Every server is behind a firewall | an `hcloud_server` with **no firewall anywhere** ‚Äî no known `firewall_ids`, no configuration reference to an `hcloud_firewall`, and no `hcloud_firewall_attachment` covering it (a bare public node). **Not-evaluable** (never a false brick) for unresolvable expressions, `label_selectors` / `apply_to` selection, or a plan without a configuration section. |
| `HCLOUD-NET-001` | No world-open SSH; management ports flagged | a firewall rule opening **tcp/22** to the whole internet ‚Äî including split-CIDR spellings that union to full coverage (Talos runs no SSH daemon; world-open 22 is always wrong). **Warns** on world-open Kubernetes API `6443` / Talos apid `50000`/`50001` (open by design today), any other world-open inbound rule, and very broad partial sources on 22. Computed source/port ‚Üí **not_evaluable**. |

## Kubernetes

A separate input path ‚Äî `EvaluateManifests(yaml)` ‚Äî decodes (multi-doc) manifests and runs the k8s set
(`provider` = `k8s`). It backs the [bring-your-own audit](/elench/engine-internals) and grades the
manifests Alethia *generates* (which pass by construction).

| ID | Title | Hard-fail on |
|----|-------|--------------|
| `CONTAINERSECURITY-001` | Containers run non-root, unprivileged, on pinned images | a `privileged` container, `runAsUser: 0`, or a mutable `:latest` / untagged image. **Warns** when `runAsNonRoot` is unset (may run as root). |
| `RBAC-001` | No wildcard RBAC or anonymous bindings | a `ClusterRole`/`Role` rule granting `"*"` verbs on `"*"` resources in `"*"` apiGroups; a binding to `system:anonymous` / `system:unauthenticated` |
| `HOSTACCESS-001` | Pods don't reach the host namespace or filesystem | `hostNetwork` / `hostPID` / `hostIPC: true`, or a `hostPath` volume |
| `RESOURCES-001` | Containers declare CPU + memory limits | a container missing CPU and/or memory `limits` (**warn** ‚Äî reliability, not a breach) |

Coverage is honest the same way the cloud controls are: a control with no in-scope resources is a vacuous
pass, and workload kinds without a decodable pod spec are simply not counted.

## Blind spots

<Callout type="warn">
  The gate inspects what the plan JSON can actually show. It is honest about the rest rather than
  guessing:

  - **Computed bodies** (`after_unknown`) ‚Äî interpolated IAM/trust/bucket policies ‚Üí `not_evaluable`.
  - **Managed-policy bodies** ‚Äî AWS-managed (non-admin) policies attached by ARN, and Alibaba's
    non-admin System policies, have no body in the plan ‚Üí coverage gap, not a pass.
  - **Cross-state / cross-module** ‚Äî resources created in another state or module, permission
    boundaries, and SCPs are out of view.
  - **AWS is deepest; Alibaba mirrors it.** GCP/Azure cover the static-credential, federation-binding,
    and primitive-role cases; Hetzner is network **posture** (token-auth has no authority surface to
    audit); the per-cloud `provider` on each control + the report makes coverage explicit.
</Callout>

## Framework mapping

Each control carries a small, conservative set of framework references (CIS / SOC 2 criteria), surfaced
in the posture and the receipt.

<Callout type="info">
  A wrong control-to-framework mapping (e.g. claiming a control satisfies "SOC 2 CC6.1" when it does
  not) is **worse than no mapping** ‚Äî it manufactures false assurance. Mappings are kept deliberately
  small, owned, and reviewed, never auto-generated.
</Callout>

## Catalog versioning

`CatalogVersion` is part of the evidence. **Bump it whenever a control's logic changes in a way that
could alter a verdict** ‚Äî an old receipt is only meaningful against the catalog (and provider schema)
that produced it. The version is recorded on every report and sealed into the receipt.
