# Elench

<Enterprise />

**Elench** (Greek _elenchus_ — proof by cross-examination) is Alethia's verification layer and agent
harness. The thesis is simple: don't just generate infrastructure — **scrutinise it and produce
evidence**. Every change, human- or agent-initiated, runs through a deterministic gate between
`tofu plan` and `tofu apply`, and leaves a signed evidence receipt; drift re-checks keep proving it
after apply.

<Callout type="info">
  **Honest framing.** Elench produces *evidence that named, versioned controls passed on a specific
  plan* — not a claim of "provably compliant" or "guaranteed least-privilege". A verdict is
  reproducible **given the same plan**; compliance is an auditor's judgment, not a property of one
  apply. The docs here are deliberately precise about what the engine can and cannot see.
</Callout>

## The load-bearing principle

> The **deterministic engine is the trusted verdict. The LLM only proposes** — and every proposal is
> re-validated by that engine before it is trusted.

This division of labor runs through the whole system. The policy gate over the OpenTofu plan JSON is
pure, reproducible code. An LLM may explain a failing control or suggest a fix, but a suggested fix is
offered only if re-running the gate confirms it resolves the failure with no regression. The model is
never the gate and has no write path.

## Layered architecture

```
        ┌────────────────────────── consumers ──────────────────────────┐
        │  in-app agent UI    Claude Code   ChatGPT   Codex   Gemini/Zed │
        └──────┬──────────────────┴───────────┴────────┴────────┬───────┘
               │ native (AI SDK)         all via remote MCP      │ via MCP
        ┌──────▼──────────────────────────────────────────────────▼──────┐
        │  EXPOSURE — one adapter: remote MCP server (READ-ONLY at launch) │
        └───────────────────────────────┬─────────────────────────────────┘
                                         │
        ┌────────────────────────────────▼────────────────────────────────┐
        │  ELENCH CORE                                                     │
        │   • Tool Registry (SSOT): audience-scoped, PDP-gated             │
        │   • Agent runtime: agent = DATA (persona·mission·scope·memory)   │
        │   • Supervisor: ledger orchestrator; sub-agents-as-tools          │
        └──────┬───────────────────────────────────────┬──────────────────┘
               │ calls                                  │ calls
        ┌──────▼─────────────┐              ┌───────────▼────────────────────┐
        │ Platform actions   │              │ VERIFICATION ENGINE             │
        │ (PDP-gated)        │              │  • deterministic plan-JSON gate │
        └────────────────────┘              │  • override / exception         │
                                            │  • signed evidence receipt      │
                                            │  • continuous drift re-check    │
                                            └───────────┬─────────────────────┘
                                                        │ new `verify` stage
                                            ┌───────────▼──────────┐
                                            │ Go runner (OpenTofu)  │
                                            └──────────────────────┘
```

Two halves, joined at the runner:

- **The verification engine** (`packages/core/verify`, `packages/core/drift`,
  `packages/core/accessanalyzer`) — the deterministic gate, the signed receipt, and continuous drift.
  This is the headline.
- **The agent harness** (`apps/console/lib/ai/tools`, `lib/agent/*`, `app/api/mcp`) — one tool registry
  exposed read-only to external agents, scoped per-agent identities with isolated memory, and a
  supervisor that delegates to a colony of sub-agents.
