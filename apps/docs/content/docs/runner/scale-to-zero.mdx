# Scale-to-Zero

Cloud-hosted Runners support automatic scale-to-zero. When no jobs are queued, the Lambda scaler scales ECS services down to 0 desired tasks. When a job arrives, it scales back to 1. This eliminates Fargate costs during idle periods.

## How It Works

A Lambda function runs on a 1-minute schedule via EventBridge. For each runner it asks the console how many jobs are queued, then adjusts that runner's ECS service desired count accordingly. The Lambda never talks to the database directly — it calls a console API endpoint, which owns the queue logic.

### Scale-Up Logic

```
Every 1 minute, for each entry in RUNNERS:
  1. POST {alethia_url}/api/platform/queue
       Authorization: Bearer ${ALETHIA_API_SECRET}
     → Console runs recover_stale_jobs()
     → Console returns { recovered, queued }
  2. If queued > 0 AND current desired = 0:
       → Scale UP to 1
       → Reset idle counter
```

### Scale-Down Logic

```
  3. For each runner:
     - If queued = 0 AND current desired > 0:
       → Increment idle counter
       → If idle counter >= 5 (5 consecutive checks = 5 minutes):
         → Scale DOWN to 0
     - If queued > 0:
       → Reset idle counter
```

The 5-check threshold (5 minutes) prevents flapping — a brief gap between jobs won't cause a scale-down followed by an immediate scale-up.

## Infrastructure

### Lambda Function

| Property | Value |
|----------|-------|
| Runtime | Python 3.12 |
| Architecture | ARM64 |
| Memory | 128 MB |
| Timeout | 30 seconds |
| Trigger | EventBridge rule (every 1 minute) |

### IAM Permissions

The Lambda function needs:
- `ecs:DescribeServices` — read current desired count
- `ecs:UpdateService` — change desired count
- Network access to each runner's console URL (for the queue check)

### Environment Variables

| Variable | Purpose |
|----------|---------|
| `ALETHIA_API_SECRET` | Bearer secret sent to the console's `/api/platform/queue` endpoint |
| `RUNNERS` | JSON array of runner configurations (region, cluster, service, alethia_url) |

### Runners Configuration

```json
[
  {
    "region": "eu-west-1",
    "cluster": "alethia-prod-runner-eu",
    "service": "alethia-prod-runner-eu-service",
    "alethia_url": "https://eu.console.example.com"
  },
  {
    "region": "us-east-1",
    "cluster": "alethia-prod-runner-us",
    "service": "alethia-prod-runner-us-service",
    "alethia_url": "https://us.console.example.com"
  }
]
```

Each entry maps to one ECS service running a Runner. The scaler handles multiple regions independently.

## Queue Check

The Lambda POSTs to the console's platform endpoint instead of querying the database:

```python
req = urllib.request.Request(
    f"{runner['alethia_url']}/api/platform/queue",
    method="POST",
    headers={
        'Authorization': f'Bearer {ALETHIA_API_SECRET}',
        'Content-Type': 'application/json',
    },
)
# Response: { "recovered": <n>, "queued": <n> }
```

The endpoint runs `recover_stale_jobs()` (re-queuing jobs orphaned by a crashed Runner) and returns the current `queued` count. The Lambda scales the runner's ECS service up to 1 when `queued > 0`, and down to 0 after 5 consecutive idle checks.

## Cold Start Behavior

When a Runner is scaled from 0 → 1:

1. **ECS task launch** — ~30 seconds (Fargate cold start, image pull)
2. **Runner boot** — ~5 seconds (binary startup, API registration)
3. **First heartbeat** — ~5 seconds after boot
4. **First job claim** — next poll cycle (≤ 10 seconds)

**Total cold start: ~50 seconds** from job queue to job start.

During this time, the job remains in QUEUED status. The user sees "Waiting for Runner..." in the UI.

## Cost Savings

Without scale-to-zero, a single Runner running 24/7 on Fargate costs:
- 1 vCPU + 4 GiB memory ≈ **$40/month** per Runner

With scale-to-zero, you pay only for:
- **Lambda invocations** — 43,200/month (1/min) × 128MB × 1s ≈ **$0.02/month**
- **Fargate** — only when jobs are running

For workloads with sporadic provisioning (a few deploys per week), this reduces Runner costs by **~99%**.

## Terraform Configuration

The scaler is defined in `infra/platform/scaler/`:

```hcl
module "scaler" {
  source = "./scaler"

  name_prefix        = local.name_prefix
  alethia_api_secret = var.alethia_api_secret

  runners = [
    for name, w in module.runner : {
      region      = var.runners[name].region
      cluster     = w.cluster_name
      service     = w.service_name
      alethia_url = var.runners[name].alethia_url
    }
  ]
}
```

The scaler automatically discovers all Runner deployments from the `module.runner` outputs.
