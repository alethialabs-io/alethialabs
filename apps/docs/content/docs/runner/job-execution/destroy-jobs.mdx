# Destroy Jobs

`DESTROY` jobs tear down all infrastructure provisioned for an environment.

## Execution flow

<Steps>

### Mint cloud credentials
Same keyless per-job minting as other job types. See [Cloud Credentials](/runner/cloud-credentials).

### tofu init
Initializes against the Console HTTP state proxy (`backend "http"`, per-job `TF_HTTP_PASSWORD`) to
read the existing state — no cloud storage credential on the runner.

### tofu destroy
```bash
tofu destroy -auto-approve
```
Removes all resources tracked in the tofu state. No confirmation prompt — the user already confirmed
via the Console or CLI.

### State cleanup
The state managed by the proxy is left in place (empty or minimal) for audit purposes.

### Report results
Job status → SUCCESS. The environment's `status` is set to `DESTROYED`.

</Steps>

## Partial destroy recovery

If the destroy fails mid-way (resource dependency, timeout):

1. tofu state reflects which resources were already destroyed.
2. The job is marked FAILED with the error message.
3. The user retries — the next destroy only attempts the remaining resources.
4. This continues until all resources are removed.

## What gets destroyed

Everything provisioned for the environment:
- VPC/VNet and all networking resources (subnets, NAT gateways, security groups)
- Kubernetes cluster (EKS/GKE/AKS) and node groups
- Databases (Aurora/Cloud SQL/Azure Database instances)
- Caches (ElastiCache/Memorystore/Azure Cache)
- NoSQL tables, queues, topics
- Container registries
- Secrets
- DNS records and certificates
- WAF rules
- IAM roles (IRSA)

<Callout type="warn">
Destroy is permanent for stateful resources. Database data, cache contents, and secret values are **lost**. tofu state is preserved so re-apply is possible, but the data is gone.
</Callout>
