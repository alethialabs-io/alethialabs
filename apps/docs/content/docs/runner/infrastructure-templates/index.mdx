# Infrastructure Templates

The runner provisions infrastructure using pre-built OpenTofu modules (HCL). These templates are
baked into the runner image at build time — each per-cloud image pre-runs `tofu init` on its
template so no provider/module download happens at job time.

## Template Categories

| Category | Path | Purpose |
|----------|------|---------|
| AWS project | `infra/templates/project/aws/` | User infrastructure on AWS |
| GCP project | `infra/templates/project/gcp/` | User infrastructure on GCP |
| Azure project | `infra/templates/project/azure/` | User infrastructure on Azure |
| Alibaba project | `infra/templates/project/alibaba/` | User infrastructure on Alibaba Cloud |
| Hetzner project | `infra/templates/project/hetzner/` | Self-managed Talos K8s on Hetzner VMs |
| Runner | `infra/templates/runner/aws/` | Self-deployed runner infrastructure (ECS Fargate) |
| ArgoCD | `infra/templates/argocd/` | ArgoCD applications and components |

## Template staging (image build)

During the [runner release pipeline](/concepts/release-system), templates are `COPY`d into the image
and pre-initialized. The full `runner` image carries all five project templates; each per-cloud image
(`runner-aws`, `runner-hetzner`, …) carries only its own:

```
infra/templates/project/<cloud>  → /home/runner/project-templates/<cloud>   (+ tofu init)
infra/templates/runner/          → /home/runner/runner-templates/
infra/templates/argocd/          → /home/runner/argocd-templates/
```

## Variable generation

When executing a job, the runner materializes a tfvars file from the environment's config snapshot
(`tofu.OverrideTfvarsFromMap`). The provisioner in `packages/core/provisioner/` maps each component
to specific tofu variables (`ProviderTfvars`). See
[Cloud Provider Abstraction](/concepts/cloud-abstraction) for the mapping layer.

## tofu state

The runner holds **no object-storage credentials**. State for every template is written through the
**Console HTTP state proxy** — the templates declare `backend "http" {}`, and the runner fetches a
per-job token it exposes as `TF_HTTP_PASSWORD`, so state I/O flows through
`/api/jobs/{id}/state`. See [tofu state](/concepts/terraform-state).
