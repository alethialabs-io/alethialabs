# Internal Architecture

Runner is a single Go binary with a straightforward architecture: a main poll loop, a heartbeat goroutine, and shared libraries from the `core` package.

## Main Loop

<Mermaid chart={`
flowchart TD
  Start([runner start]) --> Reg[Register with Alethia API]
  Reg --> HB[Start heartbeat goroutine · every 30s]
  Reg --> Wake[Start wake-stream goroutine · SSE /api/runners/wake]
  Reg --> Poll{Claim loop · on wake or 30s safety poll<br/>POST /api/jobs/claim}
  Wake -->|wake| Poll
  Poll -->|no job| Wait[await next wake / safety tick]
  Wait --> Poll
  Poll -->|job claimed| Mint[Mint per-job cloud creds · keyless OIDC]
  Mint --> Exec[Execute via sandbox seam · tofu, ArgoCD]
  Exec --> Logs[Stream logs]
  Logs --> Report[Report result]
  Report --> Poll
`} />

The claim loop runs on the main goroutine; the wake-stream listener and the heartbeat each run as a
background goroutine. All share the runner configuration.

Provisioning work runs through the **sandbox seam** (`packages/core/sandbox`), not by invoking the provisioner
directly. For trusted Alethia templates the default `Passthrough` backend runs it in-process; on the managed
fleet, untrusted bring-your-own IaC/Helm runs in a per-job **container** with an allowlisted environment and a
locked-down network. See the [`sandbox` package README](https://github.com/alethialabs-io/alethialabs/blob/main/packages/core/sandbox/README.md)
and [Security Architecture → Execution isolation](/concepts/security#execution-isolation-managed-untrusted-byo).

### Job types

The dispatch switch (`internal/agent/runner.go`) routes each claimed job to its handler:

| Job type | Handler | Does |
|---|---|---|
| `PLAN` / `DEPLOY` / `DESTROY` | `executePlan` / `executeDeploy` / … | OpenTofu plan/apply/destroy (+ the verify gate, ArgoCD, and manifest scaffold on deploy). Routed through the **sandbox seam**. |
| `CHART_SCAN` | `executeChartScan` | Render a bring-your-own Helm chart to manifests + scan it (zero secrets, deny-all egress) — always sandboxed. |
| `ANALYZE_REPO` | `executeAnalyzeRepo` | Clone + statically scan a repo into a `RepoDigest`. |
| `DETECT_DRIFT` | `executeDriftDetection` | `tofu plan -refresh-only` → `drift.Analyze` → post a `Posture`. |
| `AUDIT` | `executeAudit` | Run the elench verify engine over a **bring-your-own** plan or k8s manifests → post `verify_result`. Provisions nothing. |
| `IAC_SCAN` | `executeIacScan` | Static scan of bring-your-own IaC — always sandboxed. |
| `STATE_SURGERY` | `executeStateSurgery` | Operator-driven tofu state repair (ships **inert** / fail-closed). |
| `PROBE_CLUSTER` | `executeProbeCluster` | Live cluster-alive probe → writes `environment_probes`. |
| `BUILD` | `executeBuild` | Image build & push — schedules kaniko Jobs in the customer's cluster → `execution_metadata.build_result`. |
| `DEPLOY_RUNNER` / `UPDATE_RUNNER` / `DESTROY_RUNNER` | `executeDeployRunner` / … | Provision, update, or tear down a self-deployed runner. |

## Configuration

Runner receives its configuration via environment variables:

| Variable | Description |
|----------|-------------|
| `ALETHIA_WEB_ORIGIN` | Base URL of the Alethia instance |
| `ALETHIA_RUNNER_ID` | Unique Runner identifier (assigned at registration) |
| `ALETHIA_RUNNER_TOKEN` | Authentication token for API calls |
| `ALETHIA_RUNNER_OPERATOR` | `managed` or `self` (the legacy `ALETHIA_RUNNER_MODE` with `cloud-hosted`/`self-hosted` still works as a back-compat fallback) |
| `INFRACOST_API_KEY` | Optional, for cost estimation |

The runner holds **no object-storage credentials**. tofu state goes through the console **state proxy**
(`backend "http"` → `/api/jobs/{id}/state`) authenticated by a per-job token in `TF_HTTP_PASSWORD`, and cloud
auth is minted per job via keyless OIDC — so a managed fleet VM's cloud-init carries no long-lived secret.
Managed fleet VMs additionally self-register with a per-VM, short-TTL bootstrap token rather than any static
credential.

## Heartbeat Mechanism

The heartbeat goroutine sends `POST /api/runners/heartbeat` every 30 seconds. The payload carries:

- The runner version
- The cloud providers this runner can claim (per-cloud routing)

The runner's identity travels in the `X-Runner-ID` / `X-Runner-Token` headers, not the body. The
response returns any server-side-cancelled job IDs (`cancelled_job_ids`) — the fallback cancel path
when a wake-stream cancel was missed.

Alethia uses the heartbeat to:
- Mark Runners as ONLINE (heartbeat within 60s) or OFFLINE (heartbeat missed)
- Detect dead Runners and requeue their jobs
- Track which Runner is running which job

If a Runner crashes mid-job:
1. Heartbeat stops arriving
2. After 60 seconds, Alethia marks the Runner OFFLINE
3. If no log updates for 5+ minutes, the job is marked FAILED
4. User can retry the job — tofu state is preserved in the console state proxy (`project-tofu-state`, with fenced locks so a reclaimed lock can't clobber a slow writer)

## Graceful Shutdown

When Runner receives SIGINT or SIGTERM:

1. Enters a local **draining** state — stops claiming new jobs
2. Lets the in-flight job (if any) run to completion, up to a 10-minute grace period, after which the
   context is cancelled to force shutdown
3. Exits

In the managed fleet, a warm-pool VM claims jobs until the scaler decides it is surplus. Scale-down
is a two-step **drain then destroy**: the controller marks the runner `DRAINING` (it stops claiming
but finishes its in-flight job), and the next tick destroys it once idle — so the SIGTERM the VM
sends always lands on a runner that already drained gracefully. See
[Warm-pool scaling](/runner/scale-to-zero). (Self-operated runners get the same on any orchestrator's
stop signal.)

## API Client

The `RunnerAPIClient` handles all communication with Alethia:

| Method | Endpoint | Purpose |
|--------|----------|---------|
| `ClaimJob()` | `POST /api/jobs/claim` | Atomic job claiming |
| `Heartbeat()` | `POST /api/runners/heartbeat` | Keepalive |
| `UpdateJobStatus()` | `PUT /api/jobs/{id}/status` | Status transitions |
| `SendLog()` | `POST /api/jobs/{id}/logs` | Log batch delivery |
| `UploadPlanArtifact()` | `POST /api/jobs/{id}/plan-artifact` | Plan JSON upload |
| `DownloadPlanArtifact()` | `GET /api/jobs/{id}/plan-artifact` | Plan JSON download |

All requests include `X-Runner-ID` and `X-Runner-Token` headers for authentication.

## Shared core Package

Runner and Alethia CLI both import from `packages/core/`:

| Package | Purpose |
|---------|---------|
| `provisioner/` | Bootstrap, deploy, destroy orchestration |
| `tofu/` | OpenTofu CLI wrapper (init, plan, apply, destroy) — drives the `tofu` binary via `terraform-exec` |
| `cloud/aws/` | AWS SDK for provisioning (keyless `AssumeRoleWithWebIdentity`, EKS kubeconfig) |
| `cloud/` (`http_backend.go`) | The console HTTP state backend (per-job token via `TF_HTTP_PASSWORD`) — no storage credential on the runner |
| `sandbox/` | The per-job isolation seam (Passthrough / Container backends) |
| `cloud/gcp_provider.go` | GCP Workload Identity Federation provider tfvars + kubeconfig |
| `cloud/azure_provider.go` | Azure federated-auth provider tfvars + kubeconfig |
| `argocd/` | ArgoCD installation and configuration (App-of-Apps rendering) |
| `git/` | Git clone, branch, commit, push |
| `k8s/` | kubectl operations |
| `infracost/` | Infracost CLI wrapper and output parsing |

This shared package ensures both the CLI and Runner use the same provisioning logic.
