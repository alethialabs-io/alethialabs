# Job Execution

All jobs share the same claiming mechanism and credential handling, but differ in what they execute. For the queue mechanics, see [Job Queue Pattern](/concepts/job-queue).

## Claiming

Every 10 seconds, the Tendril calls `POST /api/jobs/claim`. The server runs `claim_next_job()` — a PostgreSQL function using `FOR UPDATE SKIP LOCKED` for atomic, exactly-once claiming.

If a job is returned, the Tendril:
1. Updates status to PROCESSING
2. Reads the vine configuration and cloud identity from the job
3. Assumes cloud credentials ([details](/tendril/cloud-credentials))
4. Executes job-type-specific logic
5. Updates status to SUCCESS or FAILED

## Error Handling

When any step fails:
1. Error logged to STDERR stream
2. Job status set to FAILED with error message
3. Terraform state preserved (partial apply state saved to [Supabase S3](/concepts/terraform-state))
4. Cloud credentials cleared (deferred cleanup always runs)
5. Tendril returns to the poll loop for the next job

Users can retry failed jobs — since state is preserved, retries continue from the last successful resource rather than starting over.
