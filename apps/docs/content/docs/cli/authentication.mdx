# Authentication

alethia uses the **Device Authorization Grant** (RFC 8628) for authentication. You never type a password in the terminal. For the full technical details, see [Authentication Flows](/concepts/authentication).

## Login

```bash
alethia login
```

<Steps>

### CLI generates device code
A UUID identifying this login attempt.

### Browser opens
Navigates to `{console_url}/cli/login?device_code={code}`.

### Approve in browser
Sign in with any OAuth provider (GitHub, GitLab, Bitbucket, Google) or email OTP. Console binds the pending
`device_code` to your account and approves it — nothing is typed back into the terminal.

### CLI polls for the token
While you approve in the browser, the CLI shows a spinner and polls `POST /api/auth/cli/exchange` with the
`device_code`. Console returns HTTP 404 while pending; once approved it returns a JWT pair (access + refresh).

### Tokens stored
Saved to `~/.config/alethia/credentials.json`.

</Steps>

### Force Re-authentication

```bash
alethia login --force
```

Re-authenticates even if a valid token already exists. Useful when switching accounts.

## Logout

```bash
alethia logout
```

Removes `~/.config/alethia/credentials.json`.

## Token Lifecycle

| Token | TTL | Purpose |
|-------|-----|---------|
| Access token | 1 hour | `Authorization: Bearer` header on API calls |
| Refresh token | 90 days | Obtain new access tokens transparently |

Before each API call, alethia decodes the access token's `exp` claim. If it has expired (or expires within a minute), the CLI refreshes automatically via `POST /api/auth/cli/refresh` and rewrites the stored access token. The user never notices.

## Identity & Active Org

After logging in, confirm who you are signed in as and which organization the CLI is scoped to:

```bash
alethia whoami   # user, active org, role, plan, default runner
alethia config   # control-plane URL, active org, on-disk file paths
```

The active organization is sent as the `X-Alethia-Org` header on every request. Set or change it
with `alethia org switch`. See [Identity & Global Options](/cli/identity) and
[Organizations, Members & Teams](/cli/commands/organizations).

## Environment Variables

| Variable | Default | Purpose |
|----------|---------|---------|
| `ALETHIA_WEB_ORIGIN` | Production URL (baked at build) | Override the Alethia API URL |

```bash
export ALETHIA_WEB_ORIGIN=http://localhost:3000
alethia login  # authenticates against local dev server
```

## Token Storage

```
~/.config/alethia/credentials.json
```

```json
{
  "access_token": "eyJ...",
  "refresh_token": "eyJ...",
  "user_email": "you@example.com"
}
```

The access-token expiry is read from the JWT itself (the `exp` claim), so no separate timestamp is
stored. The active-org context is kept separately in `~/.config/alethia/config.json` (see
[Configuration](/cli/configuration)).

<Callout type="warn">
This file contains authentication tokens. It is not encrypted — protect it with filesystem permissions.
</Callout>
