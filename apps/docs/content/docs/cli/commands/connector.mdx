# Connector Commands

The `connector` command group connects your cloud accounts to Alethia so the platform can
provision infrastructure into them. It drives the **same connection flow as the web console's
[Connectors](/console/connectors) page**, but end-to-end from the CLI — there is no need to
open a browser for the default flows.

<Callout type="info">
Each connection is a **[cloud identity](/concepts/glossary)** — the credential anchor a
[project](/concepts/glossary) links to (`cloud_identity_id`). "Connector" is the catalog of provider
*types*; the record `alethia connector` creates for your account is a cloud identity. A project
points at exactly one. Every cloud-identity query is filtered by provider, so accounts never leak
across providers.
</Callout>

Every connection uses **short-lived, keyless credentials** — AWS cross-account roles, GCP
Workload Identity Federation, and Azure federated identity. No access keys or client secrets are
ever created or stored. For the security model and how the [Runner](/runner) consumes these
credentials at provision time, see:

- [AWS Connector](/console/connectors/aws), [GCP Connector](/console/connectors/gcp), [Azure Connector](/console/connectors/azure) — per-provider concepts
- [Runner Cloud Credentials](/runner/cloud-credentials) — how credentials are exchanged at runtime
- [Security Architecture](/concepts/security) — the overall trust model

<Callout type="info">
Connections made from the CLI and the web console are the same records. An account connected
with `alethia connector gcp` immediately shows up in the console Connectors page, and vice
versa.
</Callout>

## How It Works

Each `connector` subcommand runs the same three-stage pipeline, verified server-side:

<Mermaid chart={`
flowchart LR
  A[alethia connector<br/>aws · gcp · azure · alibaba] --> B[Init pending<br/>cloud identity]
  B --> C[Run provider setup<br/>IAM OIDC role · WIF pool · federated cred · RAM OIDC]
  C --> D[Submit captured<br/>credentials]
  D --> E[Server-side verify<br/>auth + permission probe]
  E --> F{Verdict}
  F --> G[Connected]
  F --> H[Degraded]
  F --> I[Disconnected]
`} />

<Steps>

### Initialize
The CLI creates (or reuses) a pending cloud identity for the provider.

### Run provider setup
The CLI drives the provider's own tooling to create the resources Alethia needs (an IAM OIDC provider +
role, a workload identity pool, or an app registration) and captures the resulting credential
material.

### Submit & verify
The captured credentials are sent to Alethia, which verifies them **instantly, server-side** — an
auth + provisioning-permission probe (no runner). The identity is marked `Connected`, `Degraded`, or
`Disconnected` and is ready to provision into. Existing resources are discovered separately by the
server-side [asset inventory](/console/connectors/health-and-inventory).

</Steps>

## Prerequisites

- You must be authenticated: run [`alethia login`](/cli/authentication) first.
- For the **default** flows, the matching provider CLI must be installed and authenticated
  locally:
  - GCP → `gcloud` (used to drive Google Cloud Shell)
  - AWS → `aws`
  - Azure → `az`
  - Alibaba → `aliyun`
- The `--manual` flows require **none** of the above — you run a one-liner in the provider's
  browser shell and paste the result back.

## Execution Models

| Provider | Default | Why |
|----------|---------|-----|
| GCP | Runs the installer inside your **Google Cloud Shell** (via local `gcloud`) | Cloud Shell is already authenticated to your GCP org and has every tool preinstalled |
| AWS | Deploys CloudFormation with the **local `aws` CLI** | AWS CloudShell is browser-only — there is no remote-exec CLI |
| Azure | Runs the setup script with the **local `az` CLI** | Azure Cloud Shell is browser-only — there is no remote-exec CLI |
| Alibaba | Runs the setup script with the **local `aliyun` CLI** | Alibaba Cloud Shell is browser-only — there is no remote-exec CLI (as with AWS and Azure) |

All four providers support `--manual`, which prints a link + command to run in the browser shell and
prompts you to paste the result. Use it when you cannot or prefer not to install the local CLI.
Alibaba additionally supports `--terraform`, which writes an OpenTofu module you apply yourself (see
[`alethia connector alibaba`](#alethia-connector-alibaba)).

## `alethia connector gcp`

Connect a Google Cloud project using Workload Identity Federation.

```bash
alethia connector gcp --project my-gcp-project
```

| Flag | Required | Description |
|------|----------|-------------|
| `--project` | No | GCP project ID. Prompted interactively if omitted. |
| `--manual` | No | Run the installer in the browser Cloud Shell and paste the result instead of using local `gcloud`. |

By default, the CLI runs an installer **inside your Google Cloud Shell**. Cloud Shell is
authenticated as you, so the provisioner service account and a workload identity pool that trusts
Alethia are created directly in your project — and the CLI captures the generated WIF credential
config automatically. No service account keys are ever created.

<Steps>

### Enter the project
Provide `--project`, or type it at the prompt.

### Cloud Shell setup
The CLI invokes `gcloud cloud-shell ssh`, pushes the installer, and runs it. The first run can take
a minute while Cloud Shell provisions.

### Connection test
The captured WIF config is submitted, verified instantly server-side, and the identity is marked Connected (or Degraded/Disconnected).

</Steps>

Sample run:

```
◆ Project ── ◇ Cloud Shell setup ── ◇ Connection test

Running installer via gcloud Cloud Shell (the first run can take a minute)...
✔ Cloud Shell ready.
✔ Installer running in Cloud Shell.
...
--- START CONFIG (copy everything below until END CONFIG) ---
{ ...WIF credential config... }
--- END CONFIG ---

✓ Verifying connection (server-side)...
✓ Connected

✓ GCP project "my-gcp-project" connected
```

<Callout type="warn">
If `gcloud` is not logged in, Cloud Shell cannot authenticate non-interactively and you will see a
reauth error. Run `gcloud auth login`, then retry — or use `alethia connector gcp --manual`.

```
✗ gcloud is not authenticated
Run this, then retry:

  gcloud auth login
```
</Callout>

### Manual GCP setup

With `--manual`, the CLI prints a Cloud Shell link and a command, then waits for you to paste the
WIF config:

```bash
alethia connector gcp --project my-gcp-project --manual
```

1. Open [Google Cloud Shell](https://shell.cloud.google.com/cloudshell/open?shellonly=true&show=terminal).
2. Run the printed command:
   ```bash
   curl -sO https://alethia-connector-assets.s3.eu-west-1.amazonaws.com/alethia-gcp-setup.sh \
     && bash alethia-gcp-setup.sh my-gcp-project
   ```
3. Paste everything between `--- START CONFIG ---` and `--- END CONFIG ---` at the prompt.

## `alethia connector aws`

Connect an AWS account using a keyless, issuer-direct IAM role (an IAM OIDC provider trusting the Alethia
issuer — no access keys, no external id).

```bash
alethia connector aws
```

| Flag | Required | Description |
|------|----------|-------------|
| `--region` | No | AWS region for the CloudFormation stack. |
| `--role-name` | No | Name for the IAM role. Defaults to `AlethiaProvisionerRole`. |
| `--manual` | No | Deploy from the AWS console and paste the role ARN instead of using local `aws`. |
| `--script` | No | Use the `aws`-CLI setup script instead of the CloudFormation stack. |

<Tabs items={["Local aws CLI (default)", "Manual (console)"]}>
  <Tab value="Local aws CLI (default)">

The CLI deploys the bundled CloudFormation template with your local `aws` credentials, then reads
the created role ARN from the stack outputs and submits it automatically.

<Steps>

### Initialize
Alethia creates a pending cloud identity for the connection.

### Create the IAM role
The CLI runs `aws cloudformation deploy` (stack `AlethiaConnect`), creating an IAM OIDC provider for the
Alethia issuer + a role that trusts it via `AssumeRoleWithWebIdentity` (pinned to the fixed audience and
subject).

### Connection test
The role ARN is submitted, the connection is verified instantly server-side, and the identity is marked Connected (or Degraded/Disconnected).

</Steps>

```bash
alethia connector aws --region eu-west-1
```

  </Tab>
  <Tab value="Manual (console)">

With `--manual`, the CLI prints a CloudFormation **quick-create** link pre-filled with the Alethia issuer
URL and prompts for the resulting role ARN:

```bash
alethia connector aws --manual
```

1. Open the printed quick-create link (it pre-fills the template and issuer URL).
2. Acknowledge the IAM capabilities checkbox and create the stack.
3. Copy the `RoleArn` from the stack **Outputs** tab and paste it at the prompt:
   ```
   ? Role ARN: arn:aws:iam::123456789012:role/AlethiaProvisionerRole
   ```

  </Tab>
</Tabs>

## `alethia connector azure`

Connect an Azure subscription using federated identity (no client secret).

```bash
alethia connector azure --subscription <subscription-id>
```

| Flag | Required | Description |
|------|----------|-------------|
| `--subscription` | No | Azure subscription ID. Prompted interactively if omitted. |
| `--manual` | No | Run setup in Azure Cloud Shell and paste the result instead of using local `az`. |

<Tabs items={["Local az CLI (default)", "Manual (Cloud Shell)"]}>
  <Tab value="Local az CLI (default)">

The CLI runs the setup script with your local `az` login. It creates a user-assigned managed
identity (a plain ARM resource — no app registration, no client secret), a federated credential
trusting Alethia's OIDC issuer, and a least-privilege role assignment, then captures the tenant,
client, and subscription IDs automatically.

<Steps>

### Enter the subscription
Provide `--subscription`, or type it at the prompt.

### Create the managed identity
The CLI runs the setup script via `az`, creating the managed identity, federated credential, and
role assignment.

### Connection test
The captured IDs are submitted, the connection is verified instantly server-side, and the identity is marked Connected (or Degraded/Disconnected).

</Steps>

  </Tab>
  <Tab value="Manual (Cloud Shell)">

With `--manual`, the CLI prints a command to run in Azure Cloud Shell and prompts for the three
IDs the script outputs:

```bash
alethia connector azure --subscription <subscription-id> --manual
```

1. Open [Azure Cloud Shell](https://shell.azure.com).
2. Run the printed command:
   ```bash
   curl -sO https://alethia-connector-assets.s3.eu-west-1.amazonaws.com/alethia-azure-setup.sh \
     && bash alethia-azure-setup.sh <subscription-id>
   ```
3. Paste the **Tenant ID**, **Client ID**, and **Subscription ID** it prints at the prompts.

  </Tab>
</Tabs>

## `alethia connector alibaba`

Connect an Alibaba Cloud account using keyless `AssumeRoleWithOIDC`. By default the CLI runs a setup
script with your local `aliyun` CLI: it creates a RAM OIDC provider trusting the Alethia issuer plus
a RAM role, and captures the resulting role ARN automatically. Use `--manual` to run the same script
in the Alibaba Cloud Shell, or `--terraform` to apply an OpenTofu module instead. Alethia stores
nothing but the role ARN — no AccessKey.

```bash
alethia connector alibaba
```

| Flag | Required | Description |
|------|----------|-------------|
| `--region` | No | Alibaba region for the RAM provider. Defaults to `cn-hangzhou`. |
| `--manual` | No | Run the setup script in the Alibaba Cloud Shell and paste the role ARN. |
| `--terraform` | No | Apply the OpenTofu module instead of running the setup script. |
| `--dir` | No | Directory to write the OpenTofu module into (with `--terraform`). Defaults to `./alethia-alibaba-connector`. |

<Steps>

### Initialize
Alethia creates a pending [cloud identity](/concepts/glossary) for the connection.

### Create the RAM role
The CLI runs the setup script with your local `aliyun` CLI, creating a RAM OIDC provider trusting the
Alethia issuer (`<control-plane>/api/oidc`) plus a RAM role, and captures the role ARN from its
output. With `--manual`, run the same script in the Alibaba Cloud Shell and paste the ARN back.

### Connection test
The role ARN is submitted, verified instantly server-side, and the identity is marked Connected (or
Degraded/Disconnected). Alethia stores nothing but the role ARN — no AccessKey.

</Steps>

### OpenTofu module (`--terraform`)

With `--terraform`, the CLI writes `main.tf` into the target directory (`--dir`) instead of running
the setup script. It provisions the same RAM OIDC provider + role. Authenticate to Alibaba (the
`aliyun` CLI, or `ALICLOUD_ACCESS_KEY` / `ALICLOUD_SECRET_KEY`), apply it, then paste the `role_arn`
output at the prompt:

```bash
cd alethia-alibaba-connector
tofu init
tofu apply -var "region=cn-hangzhou" -var "alethia_issuer_url=https://alethialabs.io/api/oidc"
```

## `alethia connector list`

List all connected (verified) cloud accounts.

```bash
alethia connector list
```

Output:

```
PROVIDER   ACCOUNT                              CONNECTED
AWS        AWS (123456789012)                   2 hours ago
GCP        GCP (my-gcp-project)                 5 days ago
AZURE      Azure (3f9a1c2b-...)                 5 days ago
```

The table is interactive — sort and navigate with the keyboard (see [TUI Components](/cli/tui)).

## `alethia connector remove`

Disconnect a cloud account.

```bash
alethia connector remove gcp
```

If you omit the provider, the CLI shows a picker of connected accounts.

| Flag | Required | Description |
|------|----------|-------------|
| `-y`, `--yes` | No | Skip the confirmation prompt. |

<Callout type="warn">
Disconnecting resets the identity to a pending state and **orphans any projects** that referenced it
(their `cloud_identity_id` is cleared). It does not destroy already-provisioned cloud resources —
run [`alethia project destroy`](/cli/commands/plan-and-apply) first to tear those down. You can
reconnect the same provider later.
</Callout>

## Troubleshooting

| Symptom | Cause & fix |
|---------|-------------|
| `gcloud is not authenticated` / reauth error | Cloud Shell cannot prompt non-interactively. Run `gcloud auth login`, then retry — or use `--manual`. |
| `gcloud CLI not found on PATH` | Install the [Google Cloud CLI](https://cloud.google.com/sdk/docs/install), or use `--manual`. |
| `aws CLI not found on PATH` / `az CLI not found on PATH` | Install the provider CLI, or use `--manual` to set the connection up from the browser. |
| `could not find WIF config in installer output` | The Cloud Shell installer did not finish. Re-run, or use `--manual` and paste the config yourself. |
| `Captured WIF config is not valid JSON` | The pasted block was truncated. Copy everything between `--- START CONFIG ---` and `--- END CONFIG ---`. |
| Connection test fails | Alethia could not assume the credentials. Confirm the role trust policy pins the Alethia issuer + sub/aud (AWS), the workload identity binding (GCP), or the federated credential + least-privilege role (Azure). The verify runs inline server-side and returns the reason with the result — re-run the connector command to see it. |
| `Invalid format. Expected: arn:aws:iam::…` | The pasted role ARN is malformed — copy the exact `RoleArn` from the stack Outputs. |
