# Command Reference

alethia organizes its commands into groups (`connector`, `project`, `runner`, `jobs`, `cluster`,
`org`, `members`, `teams`, `channels`, `alerts`, `roles`, `grants`, `sso`, `fleet`,
`classification`, `ops`), a set of read-only inspection groups (`repo`, `cloud`, `agent`, `addon`,
`chart`, `iac`, `cost`, `drift`, `probes`), plus top-level auth (`login`/`logout`/`init`), identity
(`whoami`/`config`/`open`), `billing`, `usage`, `activity`, and utility commands. All commands
communicate with the Alethia [Console](/concepts/glossary) API — they never run
[OpenTofu](/concepts/glossary) (the `tofu` binary) locally. Provisioning (`tofu plan`/`apply`/
`destroy`) happens on a [Runner](/runner); the CLI only queues and observes it.

Every command also accepts the [global flags](/cli/identity#output-formats) `-o, --output <table|json|csv>` and `--no-input`.

<Callout type="info">
`members`, `teams`, and the org-scoped commands operate on the **active organization** (set via
`alethia org switch`). Override per-invocation with `--org <id>`. The `cluster` command is singular
with a `clusters` alias, so both `alethia cluster list` and `alethia clusters list` work.
</Callout>

<Callout type="info">
New in the reference above: [`init`](/cli/commands/init) (first-run setup), and the
[`classification`](/cli/commands/classification) group (governed taxonomy — dimensions, values,
and assignments). Both hit the same Console API as the [web console](/console/classification).
</Callout>

<Callout type="warn">
**`alethia ops` is a privileged, staff-only surface** — terminal-first break-glass incident
recovery, not a day-to-day command. Every verb hits the same audited `/api/breakglass/*` endpoints
as the operator UI, requires a `--reason` recorded to an append-only audit *before* the act, and
typed-confirms the resource id server-side. The whole group is gated behind
`ALETHIA_BREAKGLASS_ENABLED` + a `BREAKGLASS_OPERATORS` allowlist; a non-operator (or a deployment
that hasn't armed it) is refused with `403`/`404`. High-blast verbs (`force-release-lock`,
`state-surgery`, `orphan-clean`) additionally require a two-person `--approval` token minted by a
**different** operator via `alethia ops approve`, and `state-surgery`/`orphan-clean` ship **INERT**
(fail-closed — they refuse rather than perform an unscoped mutation).
</Callout>

## Interactive Selection

When `--project-id` (or `--runner-id`) is omitted from `project plan`, `project apply`, or
`project destroy`, alethia launches an interactive TUI selector — provided stdin is a TTY and
`--no-input` is off. See [TUI Components](/cli/tui) for how this works.

## Common Patterns

### Connect a Cloud Account

```bash
# Connect a GCP project (runs the installer in your Cloud Shell)
alethia connector gcp --project my-gcp-project

# List connected accounts
alethia connector list
```

See [Connector Commands](/cli/commands/connector) for AWS and Azure flows.

### Plan-Review-Apply

```bash
# 1. Plan
alethia project plan --project-id <id>
# Note the job ID from output

# 2. Stream logs to review
alethia jobs logs <plan-job-id> --follow

# 3. Apply
alethia project apply --project-id <id> --plan-job-id <plan-job-id>
```

### Scripting with --wait

```bash
# Block until plan completes, then apply
alethia project plan --project-id <id> --wait
alethia project apply --project-id <id> --plan-job-id <id> --wait
```

### Log Tailing

```bash
# Stream logs like tail -f
alethia jobs logs <job-id> --follow
```
