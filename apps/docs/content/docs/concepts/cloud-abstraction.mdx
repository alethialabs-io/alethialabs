# Cloud Provider Abstraction

Alethia provisions across **five providers** — AWS, GCP, Azure, and Alibaba Cloud (full managed clouds) plus Hetzner (a lean, self-managed Talos Kubernetes cluster on cheap VMs). A single form, a single CLI, and a single Runner agent work across all of them, through a multi-layered abstraction.

<img src="/docs/diagrams/concepts/cloud-abstraction.svg" alt="Cloud Abstraction" />

## What Alethia can provision

Alethia models a cloud-indifferent set of **capability categories** (the `ProjectConfig` service model in `packages/core/types/project_config.go`). Each maps to a concrete resource per provider and is gated by a `create_*`/`provision_*` toggle the Runner emits from your Project config:

Kubernetes cluster · VPC/network · managed SQL database · managed Redis/cache · message queue · pub/sub topic · object storage · NoSQL table · container registry · DNS zone · secrets manager · WAF · CDN / managed TLS certificate — plus the cluster autoscaler and the [pluggable-connector seam](/concepts/connectors) (swap the cloud-native DNS/secrets/registry for a tool you already run).

## Provider capability matrix

The registry (`lib/cloud-providers/registry.ts`) maps each capability to the provider's cloud-native service. The **four managed clouds are at feature parity** across every category; **Hetzner is intentionally lean** — it provisions a cluster + private network only, and managed-service categories are covered **in-cluster** (bring-your-own via the [add-on marketplace](/console/add-ons), e.g. a Postgres/Redis/MinIO Helm chart) rather than as managed cloud services.

| Capability | AWS | GCP | Azure | Alibaba | Hetzner |
|-----------|-----|-----|-------|---------|---------|
| Kubernetes | EKS | GKE | AKS | ACK | Talos (self-managed) |
| Network | VPC | VPC Network | VNet | VPC | Private Network |
| Database | Aurora | Cloud SQL | Azure Database | ApsaraDB RDS | in-cluster (BYO) |
| Cache | ElastiCache | Memorystore | Azure Cache for Redis | ApsaraDB for Redis | in-cluster (BYO) |
| NoSQL | DynamoDB | Firestore | Cosmos DB | Tablestore | in-cluster (BYO) |
| Queue | SQS | Pub/Sub | Service Bus | MNS | in-cluster (BYO) |
| Topic | SNS | Pub/Sub | Service Bus | MNS | in-cluster (BYO) |
| Object storage | S3 | Cloud Storage | Blob Storage | OSS | in-cluster (BYO) |
| Container registry | ECR | Artifact Registry | ACR | ACR | external |
| DNS | Route 53 | Cloud DNS | Azure DNS | AliDNS | external |
| Secrets | Secrets Manager | Secret Manager | Key Vault | KMS | external |
| WAF | AWS WAFv2 | Cloud Armor | Azure WAF | Alibaba WAF | — |
| Certificate / CDN | ACM + CloudFront | Managed Certificate | App Service Certificate | Managed Certificate | — |

## Infra services (post-apply)

After `tofu apply`, the Runner wires a set of **in-cluster infra services** over ArgoCD (external-dns, the external-secrets store, ingress, the default storage class, the ArgoCD URL). Not every service maps cleanly to every cloud — some are cloud-inherent gaps, not bugs. Rather than silently omit a service (or ship it broken), the pipeline records a **machine-readable per-service decision** with an honest reason — the same `not_evaluable`/"honest N/A" discipline the [verify gate](/concepts/security) applies to the plan.

The decisions come from `argocd.InfraServiceDecisions` (`packages/core/argocd/decisions.go`), computed from the **same gates** the templates render against (`InfraFacts.DNSProvider()`, the per-cloud ClusterSecretStore conditions), so a recorded decision can never drift from what actually shipped. The Runner forwards them on `execution_metadata["infra_services"]` (status + reason only — non-sensitive), and the console renders them alongside the add-on status.

| Infra service | AWS | GCP | Azure | Alibaba | Hetzner |
|---------------|-----|-----|-------|---------|---------|
| external-dns | installed (Route 53, IRSA) | installed when Workload Identity SA present | installed when managed-identity present | **N/A** — alibabacloud provider has no RRSA support upstream ([external-dns#5019](https://github.com/kubernetes-sigs/external-dns/issues/5019)); manage AliDNS outside the cluster | installed (Hetzner webhook) when a Cloud API token is connected |
| external-secrets store | installed (Secrets Manager, IRSA) | installed (Secret Manager, WI) | installed (Key Vault, WI) | installed (KMS Secrets Manager, RRSA) | **N/A** — no cloud secret store; use the [Vault connector](/concepts/connectors) |
| ingress | installed (ALB controller) | skipped — cloud LB available; install the ingress-nginx add-on | skipped — install the ingress-nginx add-on | skipped — install the ingress-nginx add-on | skipped — install the ingress-nginx add-on |
| storage class | installed (gp3, default) | built-in default (standard-rwo) | built-in default (managed-csi) | verify a default-annotated alicloud-disk class | installed (hcloud-volumes, default) |
| ArgoCD URL | installed (ALB ingress, ACM cert) | skipped — no managed ingress yet; port-forward + admin password | skipped — port-forward + admin password | skipped — port-forward + admin password | skipped — port-forward + admin password |

Each cell above is exactly what the pipeline records per deploy: an `installed`/`skipped` status plus a human-readable reason. A skip always states **why** and the **alternative** — so an operator never has to guess whether a missing service is a gap or a deliberate cloud-inherent N/A.

## Region Mapping

Regions are mapped across providers for cross-cloud Project duplication:

```
AWS us-east-1  ←→  GCP us-east4    ←→  Azure eastus
AWS eu-west-1  ←→  GCP europe-west1 ←→  Azure westeurope
AWS ap-south-1 ←→  GCP asia-south1  ←→  Azure centralindia
```

Each provider has a default region: `us-east-1` (AWS), `us-east4` (GCP), `eastus` (Azure).

## Instance Type Mapping

Compute instance types are mapped for Project conversion:

| AWS | GCP | Azure | Profile |
|-----|-----|-------|---------|
| t3.medium | e2-medium | Standard_B2s | General (2 vCPU, 4 GiB) |
| t3.large | e2-standard-2 | Standard_D2s_v5 | General (2 vCPU, 8 GiB) |
| m5a.large | n2-standard-2 | Standard_D2as_v5 | Compute (2 vCPU, 8 GiB) |
| c5.large | c2-standard-4 | Standard_F2s_v2 | Compute-optimized |
| r5.large | n2-highmem-2 | Standard_E2s_v5 | Memory-optimized |

## Database Engine Mapping

| AWS | GCP | Azure |
|-----|-----|-------|
| aurora-postgresql | cloudsql-postgresql | azure-postgresql |
| aurora-mysql | cloudsql-mysql | azure-mysql |

Capacity units differ: AWS uses **ACUs** (Aurora Capacity Units, 0.5–128), GCP uses **vCPUs** (1–96), Azure uses **vCores** (1–64). When converting across providers, the platform maps equivalent capacity ranges and warns when units change.

## The `provider_config` JSONB Column

Provider-specific configuration that doesn't fit the shared schema is stored in a `provider_config` JSONB column on component tables. For example:

- **Cluster**: `{ "enable_karpenter": true }` (AWS) vs `{ "enable_autopilot": true }` (GCP) vs `{ "enable_cluster_autoscaler": true }` (Azure)
- **DNS**: `{ "waf_cloudfront": true, "waf_alb": false }` (AWS) vs `{ "cloud_armor": true }` (GCP) vs `{ "azure_waf": true }` (Azure)
- **NoSQL**: `{ "global_table": true }` (AWS DynamoDB) vs `{ "multi_region": true }` (Azure Cosmos DB)

This keeps the shared schema clean while allowing provider-specific features.

## How the Form Adapts

When the user switches providers via the Provider Ribbon in the Design a Project form:

1. **Instance types** update to the target provider's catalog
2. **Kubernetes versions** update (e.g., 1.32 for AWS EKS, 1.31 for GCP GKE)
3. **Database engines** switch (Aurora → Cloud SQL → Azure Database)
4. **Cache node types** switch (cache.r6g.large → M2 → C2)
5. **Network options** adapt (VPC → VPC Network → VNet)
6. **DNS/WAF options** change (CloudFront WAF → Cloud Armor → Azure WAF)
7. **Autoscaler** toggles change (Karpenter → GKE Autopilot → Cluster Autoscaler)
8. **Region dropdown** repopulates with the provider's regions

All form fields reset to provider-appropriate defaults. See [Multi-Cloud Conversion](/concepts/multi-cloud-conversion) for how existing Projects are converted across providers.

## OpenTofu Templates

Each provider has its own set of OpenTofu templates:

- `infra/templates/project/aws/` — EKS, Aurora, ElastiCache, DynamoDB, S3, ECR, etc.
- `infra/templates/project/gcp/` — GKE, Cloud SQL, Memorystore, Firestore, GCS, Artifact Registry, etc.
- `infra/templates/project/azure/` — AKS, Azure Database, Azure Cache, Cosmos DB, Blob Storage, ACR, etc.
- `infra/templates/project/alibaba/` — ACK, ApsaraDB RDS, KVStore, Tablestore, OSS, CR, etc.
- `infra/templates/project/hetzner/` — Talos Kubernetes + private network (Cilium CNI, hcloud CCM/CSI).

The runner selects the correct template directory based on the project's cloud provider and generates `terraform.tfvars.json` (the tfvars file `tofu` reads) from the project configuration (`packages/core/cloud/<cloud>_provider.go` `ProviderTfvars`).

## Pluggable connectors

The same registry pattern extends one level down to **per-category connectors** — letting a Project swap the cloud-native default for a tool you already run (Cloudflare DNS, Vault secrets, Docker Hub registry, Datadog/Grafana Cloud observability). See [Connectors Architecture](/concepts/connectors).
