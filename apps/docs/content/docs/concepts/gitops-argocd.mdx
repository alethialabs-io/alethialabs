# GitOps & ArgoCD

After a successful `tofu apply`, the runner automatically installs **ArgoCD** on the provisioned Kubernetes cluster. ArgoCD manages all subsequent application deployments via GitOps — Git is the source of truth.

<img src="/docs/diagrams/concepts/gitops-argocd.svg" alt="GitOps ArgoCD" />

## What Gets Installed

Every provisioned cluster comes with ArgoCD and a set of supporting components:

| Component | Purpose |
|-----------|---------|
| **ArgoCD** | GitOps continuous delivery engine |
| **AWS Load Balancer Controller** | Kubernetes ingress via ALB/NLB (AWS) |
| **External DNS** | Automatic DNS record management from K8s resources |
| **External Secrets Operator** | Sync secrets from cloud stores to K8s Secrets |
| **Karpenter** | Just-in-time node provisioning (AWS) |
| **Metrics Server** | Resource metrics for Horizontal Pod Autoscaler |
| **Storage Class** | EBS / Persistent Disk / Managed Disk provisioner |

## Installation Flow

<Steps>

### `tofu apply` completes
The runner provisions all infrastructure (VPC, cluster, databases, etc.) via `tofu`. Outputs include cluster name, endpoint, and ARN.

### Exec-plugin kubeconfig (CLI-free)
The runner is **CLI-free** — it does not shell out to `aws eks update-kubeconfig`, `gcloud`, or `az`.
It writes an **exec-credential** kubeconfig (`packages/core/cloud.writeExecKubeconfig`) built from the
`tofu` cluster outputs (endpoint + CA), whose exec block invokes the runner itself as the credential
plugin: `runner kube-token --provider <aws|gcp|azure>` (`apps/runner/internal/agent/kube_token.go`).
So kubectl/helm mint a fresh short-lived token in-process on demand, exchanging the keyless cloud
credential for cluster access — no cluster CLIs, no long-lived kubeconfig secret.

### ArgoCD installed via Helm
The Runner runs `helm upgrade --install argocd` with the ArgoCD Helm chart. It waits for all ArgoCD pods to become Ready.

### Admin password stays in the cluster
The ArgoCD admin password lives in the `argocd-initial-admin-secret` Kubernetes Secret and is **never extracted or stored by Alethia** — retrieve it on demand:

```bash
kubectl -n argocd get secret argocd-initial-admin-secret \
  -o jsonpath='{.data.password}' | base64 -d
```

### Infrastructure facts rendered
`tofu` outputs (cluster name, VPC ID, RDS endpoints, ElastiCache endpoints, etc.) are rendered into a **Helm values file** called "infra facts." This file provides concrete resource identifiers to ArgoCD applications.

### App of Apps configured
ArgoCD Application manifests are rendered from templates and applied to the cluster.

### Metadata reported
The ArgoCD URL and cluster metadata are stored in the job's execution metadata and written to the Project's component tables. Credentials (the admin password, kubeconfigs, generated secret values) are scrubbed and never persisted.

</Steps>

## App of Apps Pattern

ArgoCD uses the **App of Apps** pattern: a single "parent" Application watches a directory of YAML files, each defining a child Application.

<Files>
  <Folder name="manifests/applications" defaultOpen>
    <File name="aws-load-balancer-controller.yaml" />
    <File name="external-dns.yaml" />
    <File name="external-secrets-operator.yaml" />
    <File name="karpenter.yaml" />
    <File name="metrics-server.yaml" />
    <File name="storage-class.yaml" />
  </Folder>
</Files>

The parent Application points to this directory. When a new YAML file is added (via Git push), ArgoCD automatically deploys the new application.

## Infrastructure Facts

`tofu` outputs are injected into ArgoCD applications as Helm values. This bridges the gap between infrastructure provisioning (OpenTofu) and application deployment (ArgoCD).

Example infra facts:

```yaml
cluster:
  name: api-backend-production
  endpoint: https://ABC123.gr7.eu-west-1.eks.amazonaws.com
  oidcProviderArn: arn:aws:iam::123456789012:oidc-provider/...

database:
  endpoint: api-backend-production.cluster-abc123.eu-west-1.rds.amazonaws.com
  port: 5432

cache:
  endpoint: api-backend-production.abc123.euw1.cache.amazonaws.com
  port: 6379

dns:
  domain: api.example.com
  zoneId: Z1234567890
```

ArgoCD applications reference these values in their Helm charts. For example, an application's database connection string uses `{{ .Values.database.endpoint }}`.

## ArgoCD Templates

Templates are stored in `infra/templates/argocd/` and staged into the Runner Docker image during the release build. Each template is a Kubernetes manifest with Helm-style placeholders:

| Template | What It Deploys |
|----------|----------------|
| `aws-load-balancer-controller.yaml` | ALB/NLB ingress controller |
| `external-dns.yaml` | Route53/Cloud DNS/Azure DNS sync (renders only when DNS is enabled with a domain and the cloud has a supported backend + workload identity) |
| `external-secrets-operator.yaml` | Secrets Manager → K8s Secrets |
| `karpenter.yaml` | Node auto-provisioning |
| `metrics-server.yaml` | CPU/memory metrics for HPA |
| `storage-class-gp3.yaml` | GP3 EBS storage class |
| `project-infra.yaml` | ArgoCD AppProject boundary |

## Post-Installation

After ArgoCD is installed:

1. **All infrastructure components are managed by ArgoCD** — changes to Git repos are automatically synced
2. **The ArgoCD URL is displayed** in the Console Clusters page with a clickable link
3. **Applications can be added** by pushing new YAML files to the `manifests/applications/` directory
4. **Drift detection** — ArgoCD continuously compares the live cluster state against Git and alerts on drift
