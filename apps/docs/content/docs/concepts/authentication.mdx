# Authentication Flows

The platform has three distinct authentication mechanisms — one for each component.

<img src="/docs/diagrams/concepts/authentication.svg" alt="Authentication Flows" />

## Web Authentication (Console)

Console uses **Better Auth** (in-process, with the Drizzle adapter) for web sessions. Two sign-in paths are supported — social OAuth and email OTP:

| Provider | Method |
|----------|--------|
| GitHub | OAuth 2.0 (native) |
| Google | OAuth 2.0 (native) |
| GitLab | OAuth 2.0 (genericOAuth) |
| Bitbucket | OAuth 2.0 (genericOAuth) |
| Email | 6-digit OTP (email OTP plugin) |

### OAuth Flow

<Steps>

### User clicks provider button
The sign-in page shows buttons for each OAuth provider.

### Redirect to provider
Better Auth redirects to the provider's authorization endpoint (e.g., `github.com/login/oauth/authorize`).

### User authorizes
The user grants the application access to their account.

### Callback with code
The provider redirects back to `/api/auth/[...all]` with an authorization code. Better Auth exchanges it for access and refresh tokens.

### Session created
Better Auth creates a session with a JWT. The JWT is stored in an HTTP-only cookie and sent with every request.

</Steps>

### Email OTP Flow

For password-less email sign-in, the user enters their email, receives a **6-digit one-time code**, and submits it back. Better Auth verifies the code and issues the same cookie-backed session as the OAuth path.

### Session Management

- Sessions are managed by Better Auth (creation, refresh, revocation)
- The session JWT lives in an HTTP-only cookie; the Next.js middleware validates it on every request
- Data access runs through the app's Policy Decision Point (PDP), which sets the active org via `set_config('app.current_org', …)` before queries
- Coarse Postgres RLS keyed on `org_id` enforces per-organization data isolation (not `auth.uid()`)

Authentication is *who you are*; **authorization** is *what you can do*. Once you're signed in, the PDP decides
every action — see [**Access Control**](/access-control) for roles, permissions, grants (allow/deny), and the
org/team model.

## CLI Authentication (alethia)

alethia uses the **Device Authorization Grant** (RFC 8628) — the same pattern used by GitHub CLI, Azure CLI, and AWS SSO.

### Why Device Code?

- Users never type passwords in the terminal
- Works with any OAuth provider (the browser handles the actual login)
- Refresh tokens are long-lived (90 days), so users rarely re-authenticate

### Flow

<Steps>

### CLI generates device code
`alethia login` generates a UUID device code and opens the browser to `{console_url}/cli/login?device_code={code}`.

### User authenticates in browser
The user is already logged into Console (or logs in via OAuth). The CLI login page shows a 6-character verification code.

### User enters code in CLI
The user types the verification code at the Bubble Tea TUI prompt.

### Token exchange
The CLI sends `POST /api/auth/cli/exchange` with the device code and verification code. Console validates the pair and returns a CLI-specific JWT.

### Token stored locally
The refresh token is saved to `~/.config/alethia/auth.json`. The access token (1-hour TTL) is used for API calls. When it expires, the CLI transparently refreshes via `POST /api/auth/cli/refresh`.

</Steps>

### JWT Structure

CLI tokens are signed with HS256 using `CLI_JWT_SECRET`:

```json
{
  "sub": "user-uuid",
  "email": "user@example.com",
  "type": "access",
  "iss": "urn:example:issuer",
  "aud": "urn:example:audience",
  "exp": 1719936000
}
```

- **Access token**: 1-hour TTL, used in `Authorization: Bearer` header
- **Refresh token**: 90-day TTL, used only to obtain new access tokens

## Runner Authentication

Runners authenticate with Console using **token-based auth** via HTTP headers.

### Registration

When a Runner is registered (via UI or CLI), the server:

1. Generates a random token
2. Hashes it with SHA-256
3. Stores the **hash** in `runners.token_hash`
4. Returns the **plaintext** token to the user (shown once, never stored)

### Request Authentication

Every Runner API call includes two headers:

```
X-Runner-ID: {runner-uuid}
X-Runner-Token: {plaintext-token}
```

The server:
1. Looks up the runner by `X-Runner-ID`
2. Hashes the provided `X-Runner-Token` with SHA-256
3. Compares against `runners.token_hash`
4. Rejects if mismatch (401)

<Callout type="warn">
The plaintext token is shown only once at registration. If lost, the Runner must be re-registered. This is intentional — it prevents token reuse and ensures tokens are not stored in a recoverable location.
</Callout>

## Authentication Summary

| Component | Method | Token Lifetime | Storage |
|-----------|--------|---------------|---------|
| **Console** (web) | Better Auth (OAuth + email OTP) | Session-based (auto-refresh) | HTTP-only cookie |
| **alethia** (CLI) | Device Code Grant | Access: 1h, Refresh: 90d | `~/.config/alethia/auth.json` |
| **Runner** (agent) | SHA-256 token hash | Permanent (until re-registration) | Environment variable |
