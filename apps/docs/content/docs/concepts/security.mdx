# Security Architecture

The platform's security model is built on one principle: **never store static cloud credentials**. Every cloud access is temporary, scoped, and revocable.

<img src="/docs/diagrams/concepts/security.svg" alt="Security Architecture" />

## Zero-Credential Model

Traditional infrastructure tools require users to provide long-lived access keys (AWS Access Key ID, GCP service account JSON, Azure client secret). These keys are stored in configuration files, environment variables, or vault systems — creating a persistent attack surface.

Console takes a different approach. The platform never stores cloud credentials. Instead, the control plane is **its own OIDC identity provider** (see [the Alethia OIDC issuer](#the-alethia-oidc-issuer-workload-identity) below): it mints short-lived, audience-scoped assertions that each cloud federates against. **All four managed clouds are keyless** — no access key, service-account JSON, or client secret anywhere:

| Provider | Mechanism | Session |
|----------|-----------|---------|
| AWS | `AssumeRoleWithWebIdentity` directly into the customer role (IAM OIDC provider — no platform account, no ExternalId) | ~1 hour, auto-refreshed |
| GCP | Workload Identity Federation (OIDC token exchange) | ~1 hour, auto-refreshed |
| Azure | Client assertion into the customer's user-assigned managed identity (federated credential — no platform Entra app) | ~1 hour, auto-refreshed |
| Alibaba | `AssumeRoleWithOIDC` (anonymous — no AccessKey, no Alethia Alibaba account) | ~1 hour, auto-refreshed |

The only long-lived secret is the issuer's **signing key** (`ALETHIA_OIDC_SIGNING_KEY`), held on the control plane and never shared — clouds verify assertions against the published JWKS. If the user wants to revoke access, they delete the IAM role / WIF config / managed identity / RAM role in their cloud account. There is nothing to leak from the Console side.

## The Alethia OIDC issuer (workload identity)

Rather than assume into each cloud with a static platform key, the control plane runs a minimal,
single-purpose **workload-identity OIDC issuer** at `${APP_URL}/api/oidc` (`lib/oidc/issuer.ts`). It is
the trust root every managed cloud federates against:

- **Mint.** `mintWorkloadToken({ audience })` signs a short-lived (≤10 min) RS256 assertion with a fixed
  subject (`alethia-connector`) and the cloud's expected **audience** (e.g. `api://AzureADTokenExchange`,
  `sts.amazonaws.com`, `sts.aliyuncs.com`). The audience scopes the token so it can't be replayed at a
  different cloud.
- **Publish.** The public half is served as a JWKS at `${APP_URL}/api/oidc/jwks` (plus an OIDC discovery
  document). The customer's cloud fetches it to verify signatures — the private key never leaves the vault.
- **Verify.** Each cloud validates the assertion (issuer + subject + audience + signature) and exchanges it
  for its own ~1 hour credential. A successful exchange **is** the proof of access.

Because the exchanged credential is short-lived, the runner keeps a live assertion available for the whole
job — it re-mints into a token **file** (or self-refreshing recipe) that the provider re-reads, so a
`tofu apply` longer than the ~1 hour session keeps authenticating without any static key. Key material is
rotatable with zero downtime — see [Key rotation](#key-rotation).

**This one model powers all four managed clouds.** The sections below are per-cloud deep dives — they differ
only in the federation primitive each cloud exposes (AWS `AssumeRoleWithWebIdentity`, GCP Workload Identity
Federation, Azure federated identity, Alibaba `AssumeRoleWithOIDC`) and in what the customer creates to trust
the issuer. Each ends with a **Permissions scope** — exactly what Alethia provisions in that cloud, and how
tightly the role is (being) scoped.

## AWS: Keyless, issuer-direct IAM role

### Setup

The user deploys a CloudFormation template (or OpenTofu module) in their AWS account. This creates:

1. **An IAM OIDC provider** for the Alethia issuer (`${APP_URL}/api/oidc`) — the account's trust root for Alethia
2. **`AlethiaProvisionerRole`** — an IAM role whose trust policy allows **only** `sts:AssumeRoleWithWebIdentity` from that provider, pinned to the fixed audience (`sts.amazonaws.com`) and subject (`alethia-connector`)

There is **no ExternalId and no trust in an Alethia AWS account** — the cryptographic sub/aud pin replaces the
confused-deputy shared secret entirely, and no other identity (not even a different Alethia token) can assume
the role.

### Runtime Flow

The control plane (for server-side connection health / inventory) and managed runners (for provisioning)
federate **directly** into the customer role with a minted assertion — a single hop, no platform account:

<Mermaid chart={`
sequenceDiagram
  participant R as Console / Runner
  participant STS as AWS STS
  R->>STS: AssumeRoleWithWebIdentity(customer RoleArn, OIDC assertion) — no access key
  STS-->>R: Temporary customer credentials (~1h)
  Note over R: tofu runs; the assertion auto-refreshes from the re-read token file
`} />

A managed runner has no ambient AWS identity, so it writes the assertion to a web-identity token file and an
AWS profile (`AWS_CONFIG_FILE`/`AWS_PROFILE`); the SDK re-reads the file and re-assumes on expiry, so an apply
past the ~1 hour session survives. (Self-operated runners run in the customer's own cloud with their own
credentials and assume the customer role directly.)

### Permissions scope

The OIDC trust above controls *who* may assume the role. This section is *what* the role is allowed to do —
a separate, independent axis.

**Exactly what Alethia provisions in AWS** (every service the project templates can touch — feature-gated,
so a given Project uses a subset): EC2/VPC (subnets, NAT, route tables, security groups, VPC endpoints),
EKS (cluster, managed node groups, add-ons, IRSA OIDC provider + roles), RDS (Aurora), ElastiCache
(Redis/Valkey), Route 53, ACM, Secrets Manager, DynamoDB, SQS/SNS, ECR, S3, KMS (envelope keys), CloudWatch
Logs + Kinesis Firehose (log delivery), EventBridge (Karpenter interruption), Application Auto Scaling, and
the IAM roles for the in-cluster controllers (external-dns, the AWS Load Balancer Controller, EBS CSI,
Karpenter, …).

<Callout type="info">
**Least-privilege by default — no account admin, fully action-enumerated.** The setup attaches **five
scoped policies** instead of `AdministratorAccess`: `ComputeNet`, `ComputeScale`, `Data`, `Edge` (the
data-plane, split into net/scale to stay under AWS's 6144-char managed-policy limit — each lists the
**specific `ec2:`/`eks:`/`rds:`/… actions** the templates issue, not `service:*`), and an action-enumerated
`IAM` policy that lets the templates create the roles they need but **explicitly denies** attaching
admin-grade managed policies (`AdministratorAccess`/`PowerUserAccess`/`IAMFullAccess`), scopes `PassRole`
to the provisioned services, and denies `organizations:*`/`account:*`/billing — so the connector **cannot
escalate to admin**. (Reads stay `Describe*`/`Get*`/`List*` so a template/module bump doesn't break a real
apply.) A permissions boundary on the created roles is the next hardening increment. See the
[AWS connector guide](/console/connectors/aws).
</Callout>

## GCP: Workload Identity Federation

### Setup

The user configures a Workload Identity Pool and Provider in their GCP project:

1. Creates a Workload Identity Pool
2. Adds an OIDC Provider with the Console issuer URL and audience
3. Binds a GCP service account to the pool with `roles/iam.workloadIdentityUser`

### Runtime Flow

<Mermaid chart={`
sequenceDiagram
  participant R as Console / Runner
  participant STS as GCP STS
  R->>STS: Present minted OIDC token (Console issuer)
  STS-->>R: Short-lived GCP access token (~1h)
  Note over R: tofu runs with the exchanged credentials
`} />

No service account JSON key files are created or stored. The OIDC token exchange happens at runtime for each job.

### Permissions scope

**Exactly what Alethia provisions in GCP:** Compute/VPC (networks, subnets, Cloud Router + NAT, firewall
rules, Cloud Armor), GKE (cluster + node pools), Cloud SQL, Memorystore (Redis), Cloud DNS, Artifact
Registry, Secret Manager, Cloud Storage, Firestore, Pub/Sub, and the workload-identity service accounts +
bindings for in-cluster controllers.

<Callout type="info">
**Least-privilege by default — no `roles/editor`.** The service account gets **10 predefined roles + 5
custom roles**. The predefined roles are kept for services whose Google-maintained admin set is already
tightly scoped and churns with new features (container / compute.network / compute.security /
servicenetworking / cloudsql / redis / dns / artifactregistry / secretmanager, plus `iam.serviceAccountUser`)
— hand-enumerating those buys nothing. The **5 custom roles replace the predefined admin roles that bundled
data-plane access a provisioner must never hold**, granting management verbs only: GCS bucket mgmt *without
object data*, Firestore database/index mgmt *without entity data*, Pub/Sub topic/sub mgmt *without
publish/consume*, add-on SA mgmt (dropping undelete/enable/disable), and a project-metadata reader
(replacing `roles/browser`, no folder/org hierarchy reads). The templates also use **resource-scoped IAM**
(zone-scoped external-dns; per-secret accessor; per-GSA workloadIdentityUser), so the provisioner needs **no
project `setIamPolicy`**. `secretmanager.admin` is kept predefined on purpose — dropping
`secretmanager.versions.access` breaks the provider's secret-version refresh. See the
[GCP connector guide](/console/connectors/gcp).
</Callout>

## Azure: Federated Identity

Azure is keyless and **customer-side** — there is **no platform Entra app** and no `ALETHIA_AZURE_CLIENT_ID`.
The customer creates a **user-assigned managed identity** in their own subscription (a plain ARM resource —
no App Registration, no directory-admin rights) with a **federated-identity credential** trusting the
Alethia OIDC issuer (subject `alethia-connector`, audience `api://AzureADTokenExchange`). The console and the
runner authenticate **as that customer identity** (its `client_id` is stored per-connection) via a client
assertion — **no client secret**. This is parity with the GCP Workload Identity Federation model.

### Setup

The customer runs the connect script (Azure Cloud Shell or OpenTofu), which creates everything in **their**
subscription and stores nothing but three public identifiers:

1. `az identity create` — a user-assigned managed identity in the customer's subscription
2. `az identity federated-credential create` — a credential trusting the Alethia issuer (subject
   `alethia-connector`, audience `api://AzureADTokenExchange`)
3. `az role definition/assignment create` — the least-privilege **Alethia Provisioner** custom role (below)

The customer pastes back their **tenant id**, **subscription id**, and the **managed identity's client id**.

### Runtime Flow

<Mermaid chart={`
sequenceDiagram
  participant R as Console / Runner
  participant AZ as Azure (Entra / ARM)
  R->>AZ: Present minted OIDC assertion as a client assertion (ARM_OIDC_TOKEN)
  AZ-->>R: Access token for Azure Resource Manager (~1h)
  Note over R: tofu runs; the runner re-mints into the token file so long applies refresh
`} />

No client secret is ever generated or stored. Access revocation is instant by deleting the managed
identity (or its role assignment) in the customer's subscription.

### Permissions scope

**Exactly what Alethia provisions in Azure:** Resource Group, AKS (+ node pools), VNet (subnets, NSGs, NAT
gateway, public IPs), PostgreSQL Flexible Server (+ private DNS zone), Redis Cache, DNS zones, Container
Registry, Key Vault (+ secrets), Storage, Service Bus, Cosmos DB, WAF policy, and the managed identities +
role assignments for in-cluster workload identity.

<Callout type="info">
**Least-privilege by default — a custom role, not `Contributor`.** The customer's managed identity is granted a
**custom role definition** ("Alethia Provisioner") enumerating only the actions the services above need —
including the AKS credential-list and per-resource-provider registration that `Contributor` omits (so this
also *fixes* a gap: pure `Contributor` cannot create the external-dns role assignment the templates require).
The one escalation-grade permission it needs — `roleAssignments/write` — is **constrained by an ABAC
condition** to a single built-in role (*DNS Zone Contributor*), so the connector can grant that to the
external-dns identity and **nothing else** (no path to Owner). See the
[Azure connector guide](/console/connectors/azure).
</Callout>

## Alibaba: RAM Role via OIDC (account-free)

Alibaba is keyless **and account-free** — Alethia holds no Alibaba account and no AccessKey. The customer
applies an OpenTofu module that creates a **RAM OIDC provider** (fixed name `alethia`) trusting the Alethia
issuer plus a **RAM role**, then pastes the role ARN (Alethia derives the provider ARN from it).

### Runtime Flow

<Mermaid chart={`
sequenceDiagram
  participant R as Console / Runner
  participant STS as Alibaba STS
  R->>STS: AssumeRoleWithOIDC(RoleArn, OIDCProviderArn, minted token) — anonymous, no AccessKey
  STS-->>R: Temporary STS credentials (~1h)
  Note over R: tofu runs; the runner re-mints into the token file so long applies refresh
`} />

`AssumeRoleWithOIDC` is an anonymous STS action — the OIDC token itself authenticates the call, so there is
no AccessKey and no request signature. On the runner, the `alicloud` provider performs the exchange itself
from `ALIBABA_CLOUD_OIDC_TOKEN_FILE` (re-read on refresh). Revocation is instant by deleting the RAM role.

### Permissions scope

**Exactly what Alethia provisions in Alibaba Cloud:** ACK (managed cluster + node pool), VPC (vSwitches, NAT,
SNAT), EIP, RDS, Container Registry EE, KVStore (Redis), AliDNS, KMS, Message Service (MNS), OSS, Tablestore
(OTS), and WAF.

<Callout type="info">
**Least-privilege by enumerated custom policy — never account admin.** The Alibaba RAM role attaches **4
enumerated custom policies** (replacing the per-service `*FullAccess` System policies): `ComputeCluster`
(ACK + its transitive node-pool ECS + API-server/Service SLB), `Network` (VPC/vSwitch/NAT/SNAT + EIP),
`Data` (RDS/KVStore/OSS/Tablestore/KMS), and `EdgeReg` (Container Registry/AliDNS/MNS/WAF). Each lists the
**specific create/read/modify/delete/tag actions** the templates issue (reads stay `Describe*`/`Get*`/`List*`
so a provider bump doesn't break a real apply) — never `service:*`, `AliyunRAMFullAccess`, or an admin
policy. The service-linked roles ACK/NAT need are covered by a **narrow custom policy granting only
`ram:CreateServiceLinkedRole`** (service-linked roles are cloud-managed with fixed permissions, so this
can't escalate). See the [Alibaba connector guide](/console/connectors/alibaba).
</Callout>

## Key rotation

The issuer's signing key is the trust root for all four clouds, so it rotates with **zero downtime** via an
overlap: the JWKS can publish two keys at once, and each assertion's `kid` header selects the right one.

1. `scripts/rotate-oidc-key.sh` mints a new primary key and parks the outgoing key in a published-but-not-
   signing slot (`ALETHIA_OIDC_SIGNING_KEY_PREVIOUS`); redeploy. New assertions are signed by the new key;
   in-flight assertions signed by the old key still verify (both are published).
2. Wait ≥24 h — long enough for every cloud's cached JWKS to refresh and all ≤10-min old assertions to
   expire.
3. `scripts/rotate-oidc-key.sh --finalize` drops the old key; redeploy.

See the [rotation runbook](/self-hosting/managed-cloud-connectors#rotating-the-issuer-signing-key).

## Execution isolation (managed untrusted BYO)

Keyless auth controls *who* can touch a cloud; **execution isolation** controls what the *provisioning code
itself* can do. In **managed** mode Alethia runs the provisioning fleet, so the trust question is: whose code
runs on it?

- **Trusted templates** — Alethia's own project templates (EKS/GKE/AKS/ACK/Talos + supporting services) run
  in-process on the runner. They are reviewed platform code, so this is safe.
- **Untrusted bring-your-own** — customer OpenTofu / Helm (a provider, an `external` data source, a chart
  hook) is arbitrary code. It runs inside a **per-job container sandbox**: a fresh container per job with its
  own PID namespace, an **allowlisted environment** (no runner token, no storage key, no signing key — only
  the short-lived, job-scoped cloud credentials it needs), read-only credential mounts, and a **default-deny,
  domain-allowlisted network**. A fail-closed guard refuses to start the container if any platform secret
  would leak into it.

Three properties keep the blast radius of a single job to that one job:

- **No storage master key on the runner.** OpenTofu state goes through a Console **state proxy**
  (`backend "http"` → `/api/jobs/[id]/state`) authenticated by a **per-job, key-scoped** token; the state
  object key is derived server-side from the job. The runner never holds the object-storage credential, and
  one job's token can only touch its own state (fenced locking prevents lost updates).
- **Per-job, per-VM credentials.** Cloud tokens are minted per job and bound to it (a stolen runner token
  can't mint another cloud's credentials); each fleet VM self-registers with its own short-TTL, instance-bound
  bootstrap token rather than a shared fleet secret.
- **Metadata + egress locked down.** The untrusted child has no network route of its own — it reaches only a
  domain-allowlist forward proxy, so the cloud metadata service (and arbitrary exfiltration endpoints) are
  unreachable by construction; only the registries, Git hosts, and the target cloud's API can be reached.

The container boundary is what actually contains a hostile provider (env-scrubbing alone doesn't — a child
sharing the worker's PID namespace can read the parent's `/proc`). Until it is enabled and verified on the
fleet, managed **untrusted** BYO stays gated off; trusted provisioning is unaffected. Operators turn it on per
the [managed provisioning runbook](/self-hosting/managed-provisioning#untrusted-byo-container-sandbox-3b). See
[Isolated Execution](/isolated-execution) for the full picture; the engineering detail lives in the
`packages/core/sandbox` package README.

## CLI Authentication (RFC 8628)

The alethia CLI uses the **Device Authorization Grant** flow (RFC 8628), the same pattern used by GitHub CLI, Azure CLI, and AWS SSO:

<Mermaid chart={`
sequenceDiagram
  participant CLI as alethia CLI
  participant B as Browser (Console)
  participant API as Console API
  CLI->>CLI: generate device_code
  CLI->>B: open {console}/cli/login?device_code=...
  B->>B: user signs in, approves the device_code
  CLI->>API: poll POST /api/auth/cli/exchange (device_code)
  API-->>CLI: 404 while pending, then JWT pair once approved
  CLI->>CLI: store ~/.config/alethia/credentials.json
`} />

The user never types a password in the terminal. The refresh token is stored locally and used to obtain short-lived access tokens for API calls. Automatic refresh happens transparently via `POST /api/auth/cli/refresh`.

## Web Authentication

The web application uses **Better Auth** with OAuth providers (GitHub and Google native; GitLab and Bitbucket via genericOAuth) plus email sign-in via 6-digit OTP:

- GitHub
- GitLab
- Bitbucket
- Google

The user clicks a provider button, is redirected to the OAuth flow, and returns with a session JWT stored in an HTTP-only cookie. Sessions are managed entirely by Better Auth.

## Row Level Security (RLS)

Every org-scoped table in the database has a coarse Row Level Security policy keyed on `org_id`. The current org is set per request via `set_config('app.current_org', …)`, driven by a Policy Decision Point (PDP) in the app:

```sql
CREATE POLICY "Org members can only access own org data"
  ON projects FOR ALL
  USING (org_id = current_setting('app.current_org')::uuid);
```

This means even if there is a bug in application code that constructs a query without a `WHERE org_id = ?` clause, PostgreSQL will filter the results to only the current org's data. A raw `SELECT * FROM projects` returns only the current org's Projects.

RLS is the coarse, unbypassable backstop; fine-grained role/permission decisions are made by the PDP. See [Access Control](/access-control) for the full authorization model (roles, permissions, grants, and the org/tenancy model).

Cloud identities are additionally filtered by provider — an AWS identity will not appear in a GCP context.

## Git Token Management

OAuth tokens for Git providers (GitHub, GitLab, Bitbucket) are stored on the Better Auth `account` table:

- Better Auth persists each linked provider's `accessToken` / `refreshToken` / expiry on `account` at login
- The `getValidProviderToken()` server action checks token expiry before use
- Expired tokens are automatically refreshed via the provider's refresh flow
- Access is scoped to the owning user (the `org_id` RLS backstop + the PDP)
