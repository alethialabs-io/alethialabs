# Security Architecture

The platform's security model is built on one principle: **never store static cloud credentials**. Every cloud access is temporary, scoped, and revocable.

<img src="/docs/diagrams/concepts/security.svg" alt="Security Architecture" />

## Zero-Credential Model

Traditional infrastructure tools require users to provide long-lived access keys (AWS Access Key ID, GCP service account JSON, Azure client secret). These keys are stored in configuration files, environment variables, or vault systems — creating a persistent attack surface.

Console takes a different approach. The platform never stores cloud credentials. Instead, each cloud provider offers a federation mechanism that grants temporary credentials at runtime:

| Provider | Mechanism | Session Duration |
|----------|-----------|-----------------|
| AWS | Cross-Account IAM Role with STS AssumeRole | 1 hour |
| GCP | Workload Identity Federation (OIDC token exchange) | 1 hour |
| Azure | Federated Identity via Entra ID (OIDC credential) | 1 hour |

If the user wants to revoke access, they delete the IAM role / WIF config / App Registration in their cloud account. There is nothing to leak from the Console side.

## AWS: Cross-Account IAM Role

### Setup

The user deploys a CloudFormation template in their AWS account. This creates:

1. **`AlethiaProvisionerRole`** — an IAM role with permissions for EC2, EKS, RDS, ElastiCache, Route53, Secrets Manager, DynamoDB, SQS/SNS, ECR, S3, and IAM (for IRSA)
2. **Trust Policy** — restricts who can assume the role to the Console platform's AWS account
3. **External ID** — a unique identifier generated by Console, stored in `cloud_identities`

### Runtime Flow

```
Runner ──► STS AssumeRole(RoleArn, ExternalID)
        ◄── Temporary credentials (AccessKeyID, SecretKey, SessionToken)
        ──► Terraform runs with temporary credentials
        ◄── Credentials expire after 1 hour
```

### External ID (Confused Deputy Prevention)

Without an External ID, any AWS account that knows the role ARN could attempt to assume it. The External ID acts as a shared secret between Console and the user's AWS account. Only the Console platform (which generated the External ID) can successfully call `AssumeRole`.

### Permissions Scope

The IAM role is not an admin role. It has precisely the permissions needed for Terraform to manage the supported resource types: VPCs, EKS clusters, RDS instances, ElastiCache, DynamoDB, ECR, S3, SQS/SNS, Route53, Secrets Manager, WAF, ACM, and IAM roles for IRSA (IAM Roles for Service Accounts in EKS).

## GCP: Workload Identity Federation

### Setup

The user configures a Workload Identity Pool and Provider in their GCP project:

1. Creates a Workload Identity Pool
2. Adds an OIDC Provider with the Console issuer URL and audience
3. Binds a GCP service account to the pool with `roles/iam.workloadIdentityUser`

### Runtime Flow

```
Runner ──► Present OIDC token from Console
GCP STS ──► Validate token against registered issuer/subject
         ◄── Short-lived GCP access token
         ──► Terraform runs with exchanged credentials
```

No service account JSON key files are created or stored. The OIDC token exchange happens at runtime for each job.

## Azure: Federated Identity

### Setup

The user creates an App Registration in Azure Entra ID (formerly Azure AD):

1. Creates an App Registration
2. Adds a Federated Credential with the Console OIDC issuer URL and subject claim
3. Grants the app appropriate RBAC roles on the target subscription

### Runtime Flow

```
Runner ──► Present OIDC token from Console
Azure  ──► Validate token against registered issuer/subject
        ◄── Access token for Azure Resource Manager
        ──► Terraform runs with federated credentials
```

No client secrets are generated. Access revocation is instant by deleting the App Registration.

## CLI Authentication (RFC 8628)

The alethia CLI uses the **Device Authorization Grant** flow (RFC 8628), the same pattern used by GitHub CLI, Azure CLI, and AWS SSO:

```
1. alethia login
   └── CLI generates UUID device_code
   └── Opens browser to: {console_url}/cli/login?device_code={code}

2. User authenticates in browser
   └── Sees 6-character verification code
   └── Types code into CLI prompt

3. CLI exchanges device_code + verification_code
   └── POST /api/auth/cli/exchange
   └── Receives refresh_token

4. Token stored locally
   └── ~/.config/alethia/auth.json (encrypted)
```

The user never types a password in the terminal. The refresh token is stored locally and used to obtain short-lived access tokens for API calls. Automatic refresh happens transparently via `POST /api/auth/cli/refresh`.

## Web Authentication

The web application uses **Better Auth** with OAuth providers (GitHub and Google native; GitLab and Bitbucket via genericOAuth) plus email sign-in via 6-digit OTP:

- GitHub
- GitLab
- Bitbucket
- Google

The user clicks a provider button, is redirected to the OAuth flow, and returns with a session JWT stored in an HTTP-only cookie. Sessions are managed entirely by Better Auth.

## Row Level Security (RLS)

Every org-scoped table in the database has a coarse Row Level Security policy keyed on `org_id`. The current org is set per request via `set_config('app.current_org', …)`, driven by a Policy Decision Point (PDP) in the app:

```sql
CREATE POLICY "Org members can only access own org data"
  ON specs FOR ALL
  USING (org_id = current_setting('app.current_org')::uuid);
```

This means even if there is a bug in application code that constructs a query without a `WHERE org_id = ?` clause, PostgreSQL will filter the results to only the current org's data. A raw `SELECT * FROM specs` returns only the current org's Specs.

RLS is the coarse, unbypassable backstop; fine-grained role/permission decisions are made by the PDP. See [Access Control](/access-control) for the full authorization model (roles, permissions, grants, and the org/tenancy model).

Cloud identities are additionally filtered by provider — an AWS identity will not appear in a GCP context.

## Git Token Management

OAuth tokens for Git providers (GitHub, GitLab, Bitbucket) are stored on the Better Auth `account` table:

- Better Auth persists each linked provider's `accessToken` / `refreshToken` / expiry on `account` at login
- The `getValidProviderToken()` server action checks token expiry before use
- Expired tokens are automatically refreshed via the provider's refresh flow
- Access is scoped to the owning user (the `org_id` RLS backstop + the PDP)
