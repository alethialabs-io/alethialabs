# Domain Model & Terminology

Every object in Alethia maps to one row in one Postgres table under `apps/console/lib/db/schema/`.
This page describes that graph and the state machines its rows move through. For the canonical name
of each concept — and the words the product deliberately avoids — start with the
[Glossary](/concepts/glossary); this page assumes those terms.

<img src="/docs/diagrams/concepts/domain-model.svg" alt="Domain Model" />

## The core shape

The tenancy and configuration graph is four levels deep:

<Mermaid chart={`
flowchart TD
  Org["organization<br/>(RLS boundary, org_id)"] -->|1:N| Proj["project<br/>(projects)"]
  Proj -->|N:1| CI["cloud identity<br/>(cloud_identities)"]
  Proj -->|1:N| Env["environment<br/>(project_environments)"]
  Env --> Comp["components<br/>(project_* tables)"]
  Env --> Jobs["jobs<br/>(provisioning work)"]
  Jobs --> Logs["job_logs"]
`} />

- An **organization** (`org_id`) is the tenancy boundary. Coarse Postgres RLS keyed on `org_id`
  filters every org-scoped table (see [Security](/concepts/security#row-level-security-rls)).
- A **project** (`projects`) is a declarative infrastructure *app*. It links to **exactly one**
  [cloud identity](/concepts/glossary#connectivity) (`projects.cloud_identity_id`) and carries the
  `region` and `iac_version`. It owns **N** environments.
- An **environment** (`project_environments`) is an independently-provisionable deployment target.
  This is the level most other tools collapse to 1:1 — here it is **1:N**: `staging` and `production`
  are two rows of the same project, each with its own state file, its own components, and its own
  provisioning `status`.
- **Components** are the cloud resources an environment declares (`project_cluster`,
  `project_databases`, …). Every component row carries **both** `project_id` and `environment_id`.

<Callout type="warn">
  **A project owns N environments, not one.** The per-environment provisioning `status` lives on
  `project_environments` (moved off `projects`), and components attach to a `(project, environment)`
  pair — never to the project alone. Treating a project as if it has a single environment is the most
  common stale assumption.
</Callout>

## Environments

`project_environments` is the pivot of the whole model. Each row has:

| Column | Type | Meaning |
|---|---|---|
| `name` | `text` | Slug feeding the OpenTofu state key (`{project}-{name}-{region}/tofu.tfstate`) and the URL. Unique per project. |
| `stage` | `environment_stage` | `development` \| `staging` \| `production`. Drives provider template defaults (instance sizing, HA knobs). Distinct from `name`. |
| `status` | `project_status` | The per-environment provisioning lifecycle (see below). |
| `is_default` | `boolean` | Exactly one row per project is the representative for single-value surfaces — the CLI wire, the project list's "Env"/status column, the project-detail header. Enforced by a partial unique index. |
| `lifecycle` | `environment_lifecycle` | `persistent` (long-lived dev/staging/prod) or `ephemeral` (disposable, TTL'd). |
| `region` | `text` | `NULL` inherits `projects.region`. |

### The ephemeral seam (inert today)

`lifecycle`, `expires_at`, `cost_cap`, and the reaper bookkeeping columns (`reap_attempts`,
`last_reap_at`, `reap_gave_up_at`) exist so a future reaper — a `drift`-style reconciler — and the UI
can build disposable "sandbox" environments on top. **Nothing reads them yet**: every environment is
effectively `persistent`. The column is a forward seam, not a live feature.

## Environment lifecycle

The per-environment `status` (`project_status` enum) is the state machine a provision walks. It is a
property of the **environment**, not the project — provisioning `production` never moves `staging`.

<Mermaid chart={`
stateDiagram-v2
  [*] --> DRAFT
  DRAFT --> QUEUED: submit plan/deploy
  QUEUED --> PROVISIONING: runner claims the job
  PROVISIONING --> ACTIVE: tofu apply succeeds
  PROVISIONING --> FAILED: apply fails
  FAILED --> QUEUED: retry
  ACTIVE --> QUEUED: re-deploy a change
  ACTIVE --> DESTROYING: destroy
  DESTROYING --> DESTROYED
  DESTROYING --> FAILED
  DESTROYED --> [*]
`} />

| Status | Meaning |
|---|---|
| `DRAFT` | Configuration written but never submitted for provisioning |
| `QUEUED` | A provisioning job is waiting for a runner to claim it |
| `PROVISIONING` | A runner is actively executing `tofu` for this environment |
| `ACTIVE` | Infrastructure is provisioned and live |
| `FAILED` | A plan/apply/destroy failed (retryable — re-enters `QUEUED`) |
| `DESTROYING` | A `DESTROY` job is tearing the environment down |
| `DESTROYED` | Infrastructure has been torn down |

## Component status

Each component carries its own `component_status`, independent of its environment's status — a
cluster can be `ACTIVE` while a newly-added database is still `CREATING`.

| Status | Meaning |
|---|---|
| `PENDING` | Declared, awaiting first provisioning |
| `CREATING` | Being created by `tofu apply` |
| `ACTIVE` | Successfully provisioned |
| `UPDATING` | A configuration change is being applied |
| `FAILED` | Creation or update failed |
| `DESTROYING` | Being torn down |
| `DESTROYED` | Successfully removed |

## Jobs

A **job** (`jobs`) is one unit of runner work. It references a `(project_id, environment_id,
cloud_identity_id)` and, once claimed, a `runner_id`. Its `provision_job_status` walks:

<Mermaid chart={`
stateDiagram-v2
  [*] --> QUEUED
  QUEUED --> CLAIMED: claim_next_job / SKIP LOCKED
  CLAIMED --> PROCESSING: runner starts execution
  PROCESSING --> SUCCESS: reported ok
  PROCESSING --> FAILED: error / stale-job sweep
  QUEUED --> CANCELLED: user cancels
  CLAIMED --> CANCELLED
  SUCCESS --> [*]
  FAILED --> [*]
  CANCELLED --> [*]
`} />

See [Job Queue Pattern](/concepts/job-queue) for the atomic-claim mechanics and
[Runner Scaling](/concepts/runner-scaling) for how capacity appears.

### Job types

The `provision_job_type` enum (`apps/console/lib/db/schema/enums.ts`) is the full catalog of work a
runner can claim — not just plan/apply/destroy:

| Type | What it does |
|---|---|
| `PLAN` | `tofu plan` (+ optional Infracost estimate); uploads a plan artifact |
| `DEPLOY` | `tofu plan` → [verify gate](/concepts/security) → `tofu apply` → ArgoCD install + manifest scaffold |
| `DESTROY` | `tofu destroy` with graceful cleanup |
| `DEPLOY_RUNNER` | Provision a self-operated runner's infrastructure |
| `UPDATE_RUNNER` | Roll a self-operated runner to a newer release |
| `DESTROY_RUNNER` | Tear down a self-operated runner's infrastructure |
| `ANALYZE_REPO` | Clone + statically scan a git repo into a `RepoDigest` (repo-first onboarding) |
| `DETECT_DRIFT` | `tofu plan -refresh-only` → a drift `Posture` (`packages/core/drift`) |
| `AUDIT` | Run the Elench `verify` engine over a bring-your-own plan/manifests — **provisions nothing** |
| `CHART_SCAN` | Render + scan a bring-your-own Helm chart (`helm template` → `verify.EvaluateManifests`); zero secrets, deny-all egress |
| `IAC_SCAN` | Scan a bring-your-own IaC module (clone → pin commit → inventory providers/modules → `tofu validate`); its result **gates** that environment's `PLAN`/`DEPLOY`/`DESTROY` |
| `STATE_SURGERY` | Break-glass, two-person-approved, fully-audited repair of a corrupt `tofu` state; ships **inert / fail-closed** (the runner refuses unless explicitly opted in) |

Promotions (`staging → production`) do **not** add a job type — a promotion reuses `PLAN` then
`DEPLOY` under a `promotions` row.
