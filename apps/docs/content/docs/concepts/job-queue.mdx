# Job Queue Pattern

Every provisioning operation in the platform — planning, deploying, destroying, drift detection,
bring-your-own scans — is modeled as a **job** in a PostgreSQL queue. Runners poll for jobs, claim
them atomically, execute them, and report results. (Cloud-account connection is *not* a job; it runs
server-side in the Console — see [Connectors](/concepts/connectors).)

<img src="/docs/diagrams/concepts/job-queue.svg" alt="Job Queue" />

## Why a Job Queue?

Infrastructure provisioning is slow (minutes to hours) and must be reliable. A job queue provides:

- **Decoupling** — the web UI and CLI queue work without waiting for completion
- **Exactly-once execution** — atomic claiming prevents two Runners from running the same job
- **Resumability** — if a runner dies, the job can be retried from persisted OpenTofu state
- **Auditability** — every operation is logged with timestamps, status, and metadata

## Job Types

The common provisioning jobs are below; the full `provision_job_type` catalog (including
`ANALYZE_REPO`, `DETECT_DRIFT`, `AUDIT`, `CHART_SCAN`, `IAC_SCAN`, and `STATE_SURGERY`) is documented
in the [Domain Model](/concepts/domain-model#job-types) and the [Glossary](/concepts/glossary#job-types-at-a-glance).

| Type | Purpose | Triggered By |
|------|---------|-------------|
| `PLAN` | `tofu plan` + optional Infracost analysis | Planning an environment |
| `DEPLOY` | `tofu plan` → verify gate → `tofu apply` → install ArgoCD | Applying an environment |
| `DESTROY` | `tofu destroy` with graceful cleanup | Destroying an environment |
| `DEPLOY_RUNNER` | Provision a self-operated runner's infrastructure | Deploying a self-operated runner |
| `UPDATE_RUNNER` | Roll a self-operated runner to the latest release | Updating a runner |
| `DESTROY_RUNNER` | Tear down a self-operated runner's infrastructure | Removing a deployed runner |

## Job Lifecycle

<Mermaid chart={`

| Status | Meaning |
|--------|---------|
| `QUEUED` | Waiting for a Runner to pick it up |
| `CLAIMED` | A Runner has locked it (prevents double-execution) |
| `PROCESSING` | Runner is actively running `tofu` |
| `SUCCESS` | Completed successfully |
| `FAILED` | Failed with error message |
| `CANCELLED` | User cancelled before completion |

## Atomic Job Claiming

The critical invariant: **exactly one Runner executes each job**. This is enforced by PostgreSQL's `FOR UPDATE SKIP LOCKED`:

```sql
CREATE OR REPLACE FUNCTION claim_next_job(p_runner_id uuid)
RETURNS jobs AS $$
DECLARE
  job jobs;
BEGIN
  SELECT * INTO job
  FROM jobs
  WHERE status = 'QUEUED'
  ORDER BY created_at ASC
  FOR UPDATE SKIP LOCKED
  LIMIT 1;

  IF job IS NOT NULL THEN
    UPDATE jobs
    SET status = 'CLAIMED',
        runner_id = p_runner_id,
        claimed_at = now()
    WHERE id = job.id;
  END IF;

  RETURN job;
END;
$$ LANGUAGE plpgsql;
```

<Callout type="info">
`FOR UPDATE SKIP LOCKED` is the key mechanism. If Runner A has locked a row, Runner B's query **skips** it entirely rather than waiting. B either gets the next queued job or gets nothing. No distributed locks, no coordination — just PostgreSQL.
</Callout>

## Config Snapshot

When a job is created, the current Project configuration is serialized as a **config_snapshot** in the job record. This ensures the Runner executes exactly the configuration that was planned, even if the user modifies the Project after queuing.

For DEPLOY jobs, a **plan hash** is validated: if the Project config changed since the plan was generated, the deploy fails with a hash mismatch error, forcing the user to re-plan.

## Job Execution Flow

<Steps>

### Job Created
User plans / applies / destroys an environment in the Console (or runs `alethia project plan` /
`alethia project apply` / `alethia project destroy` in the CLI). A `jobs` row is inserted with status `QUEUED`.

### Runner Claims Job
A runner's poll loop calls `POST /api/jobs/claim`. The server executes `claim_next_job()`. If a job is returned, the runner proceeds.

### Credential Assumption
The runner reads the `cloud_identity` from the job and assumes temporary, **keyless** credentials — `AssumeRoleWithWebIdentity` (AWS), Workload Identity Federation token exchange (GCP), or federated identity (Azure) — using a minted OIDC assertion, no static key. See [Security](/concepts/security).

### Execution
The runner runs the job-type-specific logic (`tofu` plan/apply/destroy, ArgoCD install, etc.) while streaming logs.

### Result Reporting
On completion, the runner calls `PUT /api/jobs/{id}/status` with SUCCESS or FAILED, plus execution metadata (`tofu` outputs, cost data, cluster info, and the verify report).

### Finalization
For DEPLOY jobs, the Console calls `finalizeDeployment()` to extract cluster metadata from `tofu` outputs and update the environment's component tables (cluster endpoint, ArgoCD URL, database endpoints, etc.).

</Steps>

## Failure Recovery

When a Runner dies mid-job:

1. Heartbeat stops → Console marks Runner OFFLINE after 60 seconds
2. Job stays in PROCESSING with no new log updates
3. After 5 minutes of no log activity, Console marks the job FAILED
4. User can **retry** the job — creates a new job with the same config
5. OpenTofu state is preserved in S3-compatible storage, so the retry continues from the last successful state

## Real-time Status Updates

Job status changes are broadcast via [Postgres LISTEN/NOTIFY → SSE](/concepts/realtime). The UI subscribes to `jobs` UPDATE events and updates the jobs store in real-time. The job detail page also subscribes to `job_logs` INSERT events for live log streaming.
