# Real-time Architecture

The platform uses **Postgres LISTEN/NOTIFY fanned out to the browser over SSE** (Server-Sent Events) to push live updates. Three features depend on this: log streaming, job status updates, and Runner status tracking.

<img src="/docs/diagrams/concepts/realtime.svg" alt="Realtime Architecture" />

## How LISTEN/NOTIFY → SSE Works

When a row changes, a Postgres trigger emits a `NOTIFY` on a named channel with a small JSON payload. The console holds a single `LISTEN` connection per app instance and receives every notification. It fans those notifications out to connected browsers over an SSE stream, filtering each event to the subscriber that should see it.

```
Postgres NOTIFY ──► Console LISTEN (1 per instance) ──► SSE ──► Browser
```

The implementation lives in `apps/console/lib/realtime/index.ts`. A single LISTEN connection avoids one connection per browser tab. For multi-instance scale-out, an optional Redis transport relays notifications between console instances (enterprise).

## Log Streaming

The most latency-sensitive use case. When a Runner executes Terraform, stdout/stderr are streamed to the browser in near real-time.

### Data Flow

<Steps>

### Runner writes logs
The `JobLogger` buffers Terraform output and flushes every 2 seconds (or when the buffer exceeds 10KB) via `POST /api/jobs/{id}/logs`.

### Console inserts into database
The API route inserts the log chunk into `job_logs` with a stream type (STDOUT, STDERR, or SYSTEM).

### Postgres emits NOTIFY
An `AFTER INSERT` trigger on `job_logs` fires `NOTIFY` on the log channel. The console's LISTEN connection receives it immediately.

### Browser receives chunk
The job detail page holds an SSE connection scoped to its `job_id`. The console relays the matching notification down the stream and the chunk appears in the log viewer within milliseconds.

</Steps>

### Client Subscription

```typescript
const events = new EventSource(`/api/jobs/${jobId}/logs/stream`);

events.addEventListener('log', (event) => {
  const { log_chunk, stream_type } = JSON.parse(event.data);
  appendLog(log_chunk, stream_type);
});
```

### End-to-End Latency

| Step | Latency |
|------|---------|
| Terraform → JobLogger buffer | ≤ 2 seconds |
| POST to Console API | ~50ms |
| Database INSERT | ~5ms |
| NOTIFY → SSE delivery | ~50ms |
| **Total** | **~2–3 seconds** |

## Job Status Updates

The dashboard layout opens an SSE stream for job changes to update the jobs store in real-time.

### Subscription (Dashboard Layout)

```typescript
const events = new EventSource('/api/jobs/stream');

events.addEventListener('job', (event) => {
  jobsStore.addOrUpdateJob(JSON.parse(event.data));
});
```

<Callout type="info">
Events are scoped server-side by the caller's `org_id` (the same org the SSE handler resolves from the session) — each browser only receives events for jobs in its own organization, so multi-tenant isolation is enforced before anything reaches the stream.
</Callout>

### What triggers updates:
- Job created (INSERT) → new row appears in job list
- Runner claims job (UPDATE: QUEUED → CLAIMED)
- Execution starts (UPDATE: CLAIMED → PROCESSING)
- Job completes (UPDATE: PROCESSING → SUCCESS/FAILED)

## Runner Status Tracking

The Runners page opens an SSE stream for `runners` changes:

```typescript
const events = new EventSource('/api/runners/stream');

events.addEventListener('runner', (event) => {
  const payload = JSON.parse(event.data);
  if (payload.op === 'DELETE') {
    runnersStore.removeRunner(payload.old.id);
  } else {
    runnersStore.addOrUpdateRunner(payload.new);
  }
});
```

This enables live status indicators:
- **Green** (ONLINE) — heartbeat received within 60 seconds
- **Yellow** (DRAINING) — finishing current job, not accepting new ones
- **Red** (OFFLINE) — heartbeat missed

## Project Status Updates

The projects store listens for Project status changes (e.g., DRAFT → PROVISIONING → ACTIVE). A trigger on the `projects` table emits a NOTIFY on update, which the console relays over SSE:

```typescript
events.addEventListener('project', (event) => {
  const project = JSON.parse(event.data);
  projectsStore.updateProjectInPlace(project.id, {
    status: project.status,
    estimated_monthly_cost: project.estimated_monthly_cost
  });
});
```

## Store Synchronization Pattern

All Zustand stores follow the same pattern for real-time updates:

1. **Initial fetch** on page mount (with 30-second stale threshold to avoid redundant API calls)
2. **SSE subscription** on dashboard layout mount
3. **Optimistic updates** via `addOrUpdate` methods that upsert by ID
4. **Session persistence** for filters and pagination state (via `sessionStorage`)

This means the UI stays current without polling. Job status changes, Runner heartbeats, and Project provisioning progress all appear instantly.
