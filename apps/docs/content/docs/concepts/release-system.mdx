# Build, Release & CI/CD

alethia CLI and Runner (agent) are versioned and released independently using **release-please** with conventional commits. Each component has its own release workflow, tag format, and distribution channel.

<img src="/docs/diagrams/concepts/release-system.svg" alt="Release Pipeline" />

## release-please Configuration

The monorepo uses release-please with `separate-pull-requests: true`, meaning each component gets its own release PR.

```json
// release-please-config.json
{
  "separate-pull-requests": true,
  "packages": {
    "apps/cli": {
      "release-type": "simple",
      "component": "alethia",
      "include-component-in-tag": true,
      "changelog-path": "CHANGELOG.md"
    },
    "apps/runner": {
      "release-type": "simple",
      "component": "runner",
      "include-component-in-tag": true,
      "changelog-path": "CHANGELOG.md"
    }
  }
}
```

**How it works:**

1. Developers write conventional commits: `feat(alethia): add project get command`, `fix(runner): handle nil credential`
2. On push to main, release-please analyzes commits since the last release
3. It creates (or updates) a release PR per component with:
   - Version bump (major/minor/patch based on commit type)
   - Auto-generated CHANGELOG.md entries
4. When merged, release-please creates a git tag (`cli-v0.2.2` or `runner-v0.1.0`)
5. The tag triggers the component's release workflow

## alethia (CLI) Release Pipeline

The CLI's **release home is its own public repo, `alethialabs-io/alethia-cli`** — that's where the
GitHub Release, binaries, and package metadata live. The monorepo stays the **source of truth + the
version brain**: `release-please` (real conventional-commit history) computes the version and cuts a
`cli-v*` tag; `release-cli.yml` then snapshots that exact source onto `alethia-cli` under a clean
semver tag (`vX.Y.Z`) and runs GoReleaser **there**, so every published version string is clean
semver (`0.3.0`, not `cli-v0.3.0`) and users find releases on the CLI repo.

**Trigger:** `release-please` cuts a `cli-v*` tag; `release-please.yml` calls the reusable
`release-cli.yml` (also runnable via `workflow_dispatch` with a `version`).

<Mermaid chart={`
flowchart TD
  RP["release-please (monorepo)<br/>cli-vX.Y.Z tag + changelog"] --> RC["release-cli.yml"]
  RC --> Sync["Snapshot apps/cli + packages/core → alethia-cli<br/>(tidied, standalone) + tag vX.Y.Z"]
  Sync --> GR["GoReleaser in the alethia-cli checkout<br/>(-f apps/cli/.goreleaser.yml, clean vX.Y.Z)"]
  GR --> Rel["GitHub Release on alethia-cli:<br/>archives + .deb/.rpm/.apk + checksums"]
  GR --> Docker["Docker: ghcr.io/alethialabs-io/alethia (amd64+arm64)"]
  GR --> Brew["Homebrew: Formula/alethia.rb → homebrew-tap"]
  GR --> Scoop["Scoop: alethia.json → scoop-bucket"]
`} />

**GoReleaser config** (`apps/cli/.goreleaser.yml`) runs inside the `alethia-cli` checkout at its clean
`vX.Y.Z` tag, so `{{ .Version }}` is clean semver (no `--skip=validate` / `trimprefix` needed). It:
- Builds binaries across macOS/Linux/Windows × amd64/arm64 (`project_name: alethia`)
- Creates the GitHub Release **on alethia-cli** with archives, `.deb`/`.rpm`/`.apk` packages, checksums
- Publishes/updates the Homebrew formula (`homebrew-tap`) and Scoop manifest (`scoop-bucket`), whose
  download URLs point at the alethia-cli release

**Distribution channels:**
- `brew install alethialabs-io/tap/alethia` (macOS/Linux via Homebrew) — or `brew tap alethialabs-io/tap && brew install alethia`
- `scoop bucket add alethia https://github.com/alethialabs-io/scoop-bucket && scoop install alethia` (Windows)
- `curl -fsSL https://get.alethialabs.io | sh` (macOS/Linux) · `irm https://get.alethialabs.io/install.ps1 | iex` (Windows)
- `.deb` / `.rpm` / `.apk` packages attached to each GitHub Release
- `docker run --rm ghcr.io/alethialabs-io/alethia --version`
- Direct binary download from GitHub Releases

### CLI repo (`alethialabs-io/alethia-cli`)

`alethia-cli` is the CLI's public home — both the **release/binary home** (above) and a browsable
source front door + focused issue target. Two workflows keep it current, both building the standalone
tree via the shared `.github/mirror/build-stage.sh` (`apps/cli` + `packages/core` copied verbatim so
`apps/cli/go.mod`'s `replace` stays valid + tidied so it compiles without `go.work`):

- **`mirror-cli.yml`** — continuous **source** sync on every `main` push (so the code stays current
  between releases), gated by a standalone `go build`.
- **`release-cli.yml`** — at release time, snapshots the exact `cli-v*` source, tags `alethia-cli`
  `vX.Y.Z`, and publishes the release there (see the pipeline above).

PRs opened on `alethia-cli` are directed back to the monorepo (`.github/mirror/` seeds its README +
CONTRIBUTING). Both use the same GitHub App.

### Go-live checklist (maintainer)

Everything is wired in the repo; three one-time steps make the install channels live.
The tap/bucket repos (`alethialabs-io/homebrew-tap`, `alethialabs-io/scoop-bucket`) already
exist, and the source repo is already **public** (so anonymous brew/scoop/curl/docker
downloads work the moment a release exists).

1. **Publish credentials (GitHub App).** The cross-repo publishes (formula, scoop manifest,
   CLI mirror) authenticate as a **GitHub App** — CI mints a short-lived token per run via
   `create-github-app-token`, scoped to just the target repo. The App's id + private key live
   in the `TAP_APP_ID` / `TAP_APP_PRIVATE_KEY` repo secrets.

   *This is already provisioned for the hosted alethialabs.io release.* Run the steps below only to
   set up a fresh pipeline (e.g. a fork), or to rotate the key:

   **Easy path** — run the helper once (you must be an `alethialabs-io` org owner, logged into `gh`):
   ```bash
   bash scripts/setup-release-github-app.sh
   ```
   It creates the App with the right permissions (**Contents: Read and write** + **Pull requests:
   Read and write** so release-please can open its release PRs, no webhook) via GitHub's manifest
   flow and sets both secrets automatically. Then click the printed **Install** link → select
   `homebrew-tap`, `scoop-bucket`, `alethia-cli`, **and the monorepo `alethialabs`** (release-please
   opens its release PRs there as the App so the required checks run).

   **Manual path** (fallback): Org → **Settings → Developer settings → GitHub Apps → New GitHub
   App** → permissions **Contents: Read and write** + **Pull requests: Read and write** (+ Metadata:
   read), uncheck the webhook → **Create** → **Generate a private key** → **Install App** on
   `homebrew-tap`, `scoop-bucket`, `alethia-cli`, and `alethialabs` → set `TAP_APP_ID` (the App id)
   and `TAP_APP_PRIVATE_KEY` (the whole `.pem`) as repo secrets.

   *(The App replaces the old `HOMEBREW_TAP_GITHUB_TOKEN` PAT, which can be deleted once the App
   is installed.)*
2. **Wire `get.alethialabs.io`** (for `curl … | sh` / `irm … | iex`): `tofu apply`
   `infra/cp-hetzner` (creates the `get` CNAME + tunnel ingress), then redeploy prod so
   Caddy picks up the `@get` host route in `deploy/prod/Caddyfile.tunnel`.
3. **Cut the release.** Merge a `feat:`/`fix:` commit under `apps/cli/` to `main`, then merge
   the release-please CLI PR. The `cli-v*` tag runs the pipeline end-to-end and the
   archives / formula / scoop manifest / `.deb`/`.rpm`/`.apk` / docker images all publish —
   every install method then works.

The literal tap-free `brew install alethia` (Homebrew core) is a separate, adoption-gated
milestone — see the [Homebrew Core runbook](/cli/homebrew-core).

> The last GitHub Release predates the `alethia` rename (its archives are named
> `grape_*`). The first release after this pipeline lands re-publishes them as
> `alethia_*` and creates the formula, manifest, and Linux packages.

## Runner Release Pipeline

**Trigger:** Push to tag matching `runner-v*`

**Workflow:** `.github/workflows/release-runner.yml`

<Mermaid chart={`
flowchart TD
  Tag["tag push (runner-v0.1.0)"] --> CO["Checkout"]
  CO --> Stage["Stage templates → build context<br/>(project/{aws,gcp,azure,…} + runner + argocd)"]
  Stage --> BX["Docker buildx (multi-arch)"]
  BX --> ECR["Push to AWS ECR"]
  BX --> GHCR["Push to GHCR (ghcr.io)"]
  ECR --> Meta["POST /api/releases/runner { version, release_notes }"]
  GHCR --> Meta
  Meta --> Roll["ECS roll: aws ecs update-service --force-new-deployment"]
`} />

<Callout type="warn">
  The **ECS roll** at the end of this workflow updates the **self-deployed / release** runner service
  — it is *not* how the managed fleet runs. Managed capacity comes from the in-app scaler booting
  Hetzner VMs from the same GHCR image (see [Runner Scaling](/concepts/runner-scaling)). Publishing a
  runner release makes the image available to both paths; only the self-deployed ECS service is rolled
  here.
</Callout>

**Key details:**
- **Template staging:** infrastructure templates are copied into the Docker build context so the final image carries all OpenTofu modules (AWS, GCP, Azure, Alibaba, Hetzner) plus the runner + ArgoCD templates
- **Dual registry:** ECR for the ECS self-deployed path, GHCR for the managed fleet and public pulls
- **API publish:** `POST /api/releases/runner` records a `runner_releases` row so the Console can show version info and detect outdated runners
- **Rolling deployment:** `--force-new-deployment` rolls the ECS service that hosts the self-deployed runner

## Runner Dev Deployment

**Trigger:** Manual (`workflow_dispatch`)

**Workflow:** `.github/workflows/deploy-runner.yml`

For development/hotfix deployments without a version tag. Builds and deploys using the git SHA as the image tag instead of a semver version. Same Docker build and ECS deployment steps as the release workflow.

## release-please Automation

**Workflow:** `.github/workflows/release-please.yml`

**Trigger:** Push to default branch

Runs `googleapis/release-please-action` which:
1. Reads `release-please-config.json` and `.release-please-manifest.json`
2. Analyzes conventional commits since the last release per component
3. Creates/updates release PRs with changelogs
4. On PR merge, creates git tags that trigger release workflows

**Manifest** (`.release-please-manifest.json`) tracks current versions:

```json
{
  "apps/cli": "0.1.7",
  "apps/runner": "0.0.0"
}
```

## Version Tracking in the Platform

When a Runner release is published, the Console API receives the version and release notes via `POST /api/releases/runner`. This creates a `runner_releases` record. Registered Runners reference their `release_id` so the platform can:

- Display current version per Runner in the UI
- Show release notes and changelogs
- Detect outdated Runners
