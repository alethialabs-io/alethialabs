# Build, Release & CI/CD

alethia CLI and Runner (agent) are versioned and released independently using **release-please** with conventional commits. Each component has its own release workflow, tag format, and distribution channel.

<img src="/docs/diagrams/concepts/release-system.svg" alt="Release Pipeline" />

## release-please Configuration

The monorepo uses release-please with `separate-pull-requests: true`, meaning each component gets its own release PR.

```json
// release-please-config.json
{
  "separate-pull-requests": true,
  "packages": {
    "apps/cli": {
      "release-type": "simple",
      "component": "alethia",
      "include-component-in-tag": true,
      "changelog-path": "CHANGELOG.md"
    },
    "apps/runner": {
      "release-type": "simple",
      "component": "runner",
      "include-component-in-tag": true,
      "changelog-path": "CHANGELOG.md"
    }
  }
}
```

**How it works:**

1. Developers write conventional commits: `feat(alethia): add project get command`, `fix(runner): handle nil credential`
2. On push to main, release-please analyzes commits since the last release
3. It creates (or updates) a release PR per component with:
   - Version bump (major/minor/patch based on commit type)
   - Auto-generated CHANGELOG.md entries
4. When merged, release-please creates a git tag (`cli-v0.2.2` or `runner-v0.1.0`)
5. The tag triggers the component's release workflow

## alethia (CLI) Release Pipeline

**Trigger:** `release-please` cuts a `cli-v*` tag; `release-please.yml` then calls the reusable workflow.

**Workflow:** `.github/workflows/release-cli.yml`

```
release-please merges the CLI release PR → tag cli-v0.2.2
  └── Checkout + Go setup
       └── GoReleaser (-f apps/cli/.goreleaser.yml, --skip=validate)
            ├── Build: darwin/amd64, darwin/arm64, linux/{amd64,arm64}, windows/{amd64,arm64}
            ├── Archive: tar.gz per platform (zip on Windows) + checksums
            ├── nfpm: .deb / .rpm / .apk (linux) → attached to the release
            ├── GitHub Release: upload archives + packages + checksums
            ├── Docker: ghcr.io/alethialabs-io/alethia (amd64+arm64 manifest)
            ├── Homebrew: push Formula/alethia.rb to alethialabs-io/homebrew-tap
            └── Scoop: push alethia.json to alethialabs-io/scoop-bucket
```

**GoReleaser config** (`apps/cli/.goreleaser.yml`):
- Builds binaries across macOS/Linux/Windows × amd64/arm64 (`project_name: alethia`)
- Creates a GitHub Release with archives, `.deb`/`.rpm`/`.apk` packages, and checksums
- Publishes/updates the Homebrew formula and Scoop manifest automatically

**Distribution channels:**
- `brew install alethialabs-io/tap/alethia` (macOS/Linux via Homebrew) — or `brew tap alethialabs-io/tap && brew install alethia`
- `scoop bucket add alethia https://github.com/alethialabs-io/scoop-bucket && scoop install alethia` (Windows)
- `curl -fsSL https://get.alethialabs.io | sh` (macOS/Linux) · `irm https://get.alethialabs.io/install.ps1 | iex` (Windows)
- `.deb` / `.rpm` / `.apk` packages attached to each GitHub Release
- `docker run --rm ghcr.io/alethialabs-io/alethia --version`
- Direct binary download from GitHub Releases

### One-time tap/bucket setup (maintainer)

GoReleaser pushes the Homebrew formula and Scoop manifest at release time; the target
repos (`alethialabs-io/homebrew-tap`, `alethialabs-io/scoop-bucket`, both public with a
`main` branch) already exist. The remaining one-time step is the token:

1. **Install the tap GitHub App** on both `homebrew-tap` and `scoop-bucket`, and set the
   `TAP_APP_ID` / `TAP_APP_PRIVATE_KEY` repo secrets — `release-cli.yml` mints a scoped
   token via `create-github-app-token` (`repositories: homebrew-tap`, `scoop-bucket`) and
   passes it as both `HOMEBREW_TAP_GITHUB_TOKEN` and `SCOOP_TAP_GITHUB_TOKEN`. (A classic
   PAT with `public_repo` in those env vars also works.)
2. **Wire `get.alethialabs.io`** (for `curl … | sh`): the `get` DNS record + tunnel
   ingress land via `tofu apply` of `infra/cp-hetzner`, and the Caddy host route is in
   `deploy/prod/Caddyfile.tunnel`. Apply cp-hetzner once and redeploy.
3. **Cut a release**: merge a `feat:`/`fix:` commit under `apps/cli/` to `main`, then
   merge the release-please CLI PR. The `cli-v*` tag runs the pipeline end-to-end and the
   formula/manifest/packages publish — every install method then works.

> The last GitHub Release predates the `alethia` rename (its archives are named
> `grape_*`). The first release after this pipeline lands re-publishes them as
> `alethia_*` and creates the formula, manifest, and Linux packages.

## Runner Release Pipeline

**Trigger:** Push to tag matching `runner-v*`

**Workflow:** `.github/workflows/release-runner.yml`

```
Tag push (runner-v0.1.0)
  └── Checkout
       └── Stage templates
            ├── infra/templates/project/aws → apps/runner/project-templates/aws
            ├── infra/templates/project/gcp → apps/runner/project-templates/gcp
            ├── infra/templates/project/azure → apps/runner/project-templates/azure
            ├── infra/templates/runner → apps/runner/runner-templates
            └── infra/templates/argocd → apps/runner/argocd-templates
       └── Docker buildx (ARM64)
            ├── Push to AWS ECR (eu-west-1)
            └── Push to GitHub Container Registry (ghcr.io)
       └── Publish release metadata
            └── POST /api/releases/runner { version, release_notes }
       └── Force new ECS Fargate deployment
            └── aws ecs update-service --force-new-deployment
```

**Key details:**
- **Template staging:** Infrastructure templates are copied into the Docker build context so the final image contains all Terraform modules for AWS, GCP, and Azure
- **ARM64 only:** Runner runs on Graviton (ARM) Fargate tasks for cost efficiency
- **Dual registry:** ECR for production Fargate deployment, GHCR for public access
- **API publish:** Console is notified of new releases and can display version info in the UI
- **Rolling deployment:** `--force-new-deployment` triggers ECS to launch new tasks with the updated image

## Runner Dev Deployment

**Trigger:** Manual (`workflow_dispatch`)

**Workflow:** `.github/workflows/deploy-runner.yml`

For development/hotfix deployments without a version tag. Builds and deploys using the git SHA as the image tag instead of a semver version. Same Docker build and ECS deployment steps as the release workflow.

## release-please Automation

**Workflow:** `.github/workflows/release-please.yml`

**Trigger:** Push to default branch

Runs `googleapis/release-please-action` which:
1. Reads `release-please-config.json` and `.release-please-manifest.json`
2. Analyzes conventional commits since the last release per component
3. Creates/updates release PRs with changelogs
4. On PR merge, creates git tags that trigger release workflows

**Manifest** (`.release-please-manifest.json`) tracks current versions:

```json
{
  "apps/cli": "0.1.7",
  "apps/runner": "0.0.0"
}
```

## Version Tracking in the Platform

When a Runner release is published, the Console API receives the version and release notes via `POST /api/releases/runner`. This creates a `runner_releases` record. Registered Runners reference their `release_id` so the platform can:

- Display current version per Runner in the UI
- Show release notes and changelogs
- Detect outdated Runners
