# Runner Scaling

Managed runner capacity is driven by an **in-app scaler** — a control loop that lives inside the
Console process (`apps/console/lib/fleet/`), not a separate AWS service. A 60-second tick, plus an
on-demand wake whenever a job is enqueued, sizes a per-provider **warm pool** by queue depth and
converges it through a `FleetProvider`. The default provider is **Hetzner Cloud**
(`FLEET_PROVIDER=hcloud`): cheap VMs that boot a per-cloud runner image from GHCR and self-register.

## The control loop

`apps/console/lib/fleet/` splits into a pure planner and an impure driver:

| File | Role |
|---|---|
| `scaler.ts` | Hosts the 60s tick (`TICK_INTERVAL_MS = 60_000`) and the fire-on-enqueue `wakeFleetScaler`. Serializes passes per process so two overlapping ticks can't over-provision. |
| `plan.ts` | **Pure** `plan(project, observed)` — diffs a pool target against observed reality and returns the minimal, safe create/drain/destroy actions. Reconciles four axes in priority order: **health → count → placement → version**. |
| `controller.ts` | Applies the planned actions idempotently through the provider and records each to the `fleet_actions` ledger. |
| `hcloud.ts` | The Hetzner `FleetProvider` — `list`/`create`/`destroy` over the Hetzner REST API (`fetch`, no SDK). |
| `config.ts` | Parses `FLEET_POOLS` (a JSON array of pool targets). Unset or invalid → `[]` → the controller is a **no-op**. |

## Sizing: warm pools, not scale-to-zero

Each pool target (a `FLEET_POOLS` entry) is a small set of knobs (`config.ts`):

| Knob | Default | Meaning |
|---|---|---|
| `warmMin` | `1` | The warm floor — healthy runners kept ready even at zero backlog. |
| `max` | `10` | Hard ceiling on healthy instances. |
| `buffer` | `1` | N+1 headroom added on top of demand. |
| `slotsPerRunner` | `1` | Concurrent jobs one runner claims. |
| `surge` | `1` | Max new VMs created per tick (bounded ramp). |
| `scaleDownGraceTicks` | `5` | Consecutive idle-surplus ticks before a runner is drained. |
| `locations` | `["fsn1"]` | Hetzner locations to spread across. |

The desired healthy count each tick is:

```
targetCount = clamp( max(warmMin, demand + buffer), 0, max )
demand      = max(recentPeak, ceil(backlog / slotsPerRunner))
```

So a pool holds `warmMin` runners warm, lifts toward `demand + buffer` as the queue grows, and never
plans the claimable (online) count below the warm floor. A non-zero `warmMin` trades a little idle
cost for **zero cold-start latency** on the common case; set `warmMin: 0` to allow scale-to-zero for
a pool that tolerates boot latency.

## VM lifecycle: cloud-init self-registration

When the planner calls for a new instance, the Hetzner provider creates a VM whose **cloud-init**
runs the per-cloud runner image from GHCR (`ghcr.io/alethialabs-io/runner-<cloud>`). The image
self-registers rather than being pre-provisioned with credentials:

<Steps>

### Create
`controller.ts` calls `hcloud.create()`; a VM boots with cloud-init carrying an
`ALETHIA_RUNNER_BOOTSTRAP_TOKEN` — a **short-TTL, instance-bound** token, not a shared fleet secret.

### Self-register
The runner binary exchanges the bootstrap token for its identity via `POST /api/runners/register`,
receiving its own `ALETHIA_RUNNER_ID` / `ALETHIA_RUNNER_TOKEN`.

### Claim & work
The runner enters its poll loop (`POST /api/jobs/claim`, ~10s) and starts claiming jobs for the
clouds it serves. See [Job Queue Pattern](/concepts/job-queue).

### Drain & destroy
When backlog falls and a runner has been surplus for `scaleDownGraceTicks` consecutive ticks, the
planner emits a `drain` (finish the current job, accept no new ones) then a `destroy`. The VM is
deleted; billing stops.

</Steps>

Every action the controller takes is written to the **`fleet_actions`** ledger
(`create` \| `drain` \| `destroy` \| `noop`) so there is a durable record of *why* the scaler
changed the pool.

## Concurrency ceiling

`scheduleTick()` serializes passes **within a process** — a wake arriving mid-pass coalesces into a
single follow-up so a wake-storm can't fan out into N parallel `read → plan → act` passes (each
would read the same stale instance list and over-provision). This is **not** cross-replica: the
serializer state is per-process, so more than one replica could still run concurrent passes against
the same pool. Production runs a single box today, so this is not live exposure; true multi-replica
safety would need a Postgres advisory lock around `reconcilePool`.

## Heartbeats & liveness

While running, a runner heartbeats every **30 seconds** (`POST /api/runners/heartbeat`), carrying
its binary version and resetting its status to `ONLINE`. If heartbeats stop for 60 seconds the
Console marks the runner `OFFLINE`; jobs stuck in `PROCESSING` with no log activity are swept to
`FAILED` — see [Failure Recovery](/concepts/job-queue#failure-recovery).

The runner is **CLI-free**: its images bundle `tofu`, `kubectl`, and `helm` but **not** aws-cli /
gcloud / azure-cli. Cluster tokens are minted in-process via `runner kube-token`, and cloud auth is
keyless OIDC — so the images stay small and hold no static cloud credentials. See
[Security](/concepts/security#execution-isolation-managed-untrusted-byo).
