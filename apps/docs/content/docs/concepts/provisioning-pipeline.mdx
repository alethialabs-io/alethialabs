# Provisioning Pipeline

When a user plans or applies an environment, the platform transforms the project's form configuration into OpenTofu variables through a five-stage pipeline. Each stage has a single responsibility: serialize, freeze, deserialize, map, execute.

<Callout type="info">
  The Project config can be authored by hand **or inferred from repos** â€” the "A" flow scans one or more
  repos (monorepo-aware), infers the stack, and produces the same `ProjectFormData` this pipeline
  consumes. See the [elench Architecture](/elench/architecture) page for the full repos â†’ verified cluster
  loop.
</Callout>

## Stage 1: Form â†’ Database

The project form in Console (or `alethia project plan` in the CLI) calls `createProject()`, which inserts the configuration into normalized component tables:

| Table | Data |
|-------|------|
| `projects` | Project name, region, environment, cloud_identity_id, iac_version |
| `project_network` | CIDR block, provision_network flag, existing network_id |
| `project_cluster` | K8s version, instance types, node min/max/desired, cluster_admins, provider_config |
| `project_dns` | Enabled flag, zone_id, domain_name, WAF settings (in provider_config) |
| `project_repositories` | Apps destination (GitOps) repo URL |
| `project_source_repos` | Scanned source repos (1:N) + detected services â€” used to generate app manifests |
| `project_databases` | Engine, instance class, scaling config (per database) |
| `project_caches` | Node type, cluster size (per cache) |
| `project_queues` | Queue name, config (per queue) |
| `project_topics` | Topic name, subscriptions (per topic) |
| `project_nosql_tables` | Table/collection config with provider_config |
| `project_secrets` | Secret name and metadata |

Each table stores the shared schema. Provider-specific options live in `provider_config` JSONB columns â€” see [Cloud Provider Abstraction](/concepts/cloud-abstraction).

## Stage 2: Database â†’ Config Snapshot

When a job is queued, `buildConfigSnapshot()` in `app/server/actions/projects.ts` queries all component tables in parallel and assembles a single JSON object. This object is stored in `jobs.config_snapshot`.

The snapshot is a **point-in-time freeze**. Even if the user edits the Project after queuing, the job executes the exact configuration that was planned.

```
config_snapshot = {
  ...project fields,
  provider: "aws" | "gcp" | "azure",
  network: { provision_network, cidr_block, network_id, ... },
  cluster: { cluster_version, instance_types, node_min_size, ... },
  dns:     { enabled, zone_id, domain_name, provider_config, ... },
  databases: [...],
  caches:    [...],
  queues:    [...],
  topics:    [...],
  nosql_tables: [...],
  secrets:   [...],
  git_access_token: ""   // fetched at runtime, not stored
}
```

<Callout type="info">
The git access token is intentionally left empty in the snapshot. The Runner fetches it at runtime via `POST /api/jobs/{id}/git-token` to avoid storing secrets in the job record.
</Callout>

## Stage 3: Config Snapshot â†’ ProjectConfig

The Runner claims the job and deserializes the JSON snapshot into a strongly-typed Go struct:

```go
func snapshotToProjectConfig(snapshot map[string]any) (*types.ProjectConfig, error) {
    data, _ := json.Marshal(snapshot)
    var sc types.ProjectConfig
    json.Unmarshal(data, &sc)
    return &sc, nil
}
```

The `ProjectConfig` struct in `packages/core/types/project_config.go` mirrors the snapshot shape with typed fields: `Network`, `Cluster`, `DNS`, `Databases[]`, `Caches[]`, and so on.

## Stage 4: ProjectConfig â†’ terraform.tfvars.json

Each cloud provider implements a `ProviderTfvars()` method that maps `ProjectConfig` fields to provider-specific OpenTofu variable names:

| ProjectConfig Field | AWS | GCP | Azure |
|-----------------|-----|-----|-------|
| `ProjectName` | `project_name` | `project_name` | `project_name` |
| `Region` | `region` | `region` | `location` |
| `CloudAccountID` | `aws_account_id` | `project_id` | `subscription_id` |
| `Network.ProvisionNetwork` | `provision_vpc` | `provision_network` | `provision_vnet` |
| `Network.CIDRBlock` | `vpc_cidr` | `network_cidr` | `vnet_cidr` |
| `Cluster.ClusterVersion` | `eks_cluster_version` | `gke_cluster_version` | `aks_cluster_version` |
| `Cluster.InstanceTypes` | `eks_instance_types` | `gke_instance_types` | `aks_instance_types` |
| `Databases[]` | `rds_config` | `cloud_sql_engine`, ... | `azure_db_engine`, ... |
| `Caches[]` | `redis_instance_type` | `memorystore_instance_type` | `azure_cache_sku` |
| `Queues[]` + `Topics[]` | `sqs_queues`, `sns_topics` | `pubsub_topics` | `service_bus_queues`, `service_bus_topics` |
| `NosqlTables[]` | `ddb_table_configuration` | `firestore_databases` | `cosmos_db_collections` |
| `StorageBuckets[]` | `bucket_configuration` | `cloud_storage_buckets` | `storage_containers` |

The result is a `map[string]interface{}` written to `terraform.tfvars.json` via `OverrideTfvarsFromMap()`.

**No templating engine touches the `.tf` files.** The OpenTofu templates at `infra/templates/project/{aws,gcp,azure,alibaba,hetzner}/` are plain HCL with `variable` declarations. Values are injected entirely through OpenTofu's native tfvars mechanism (`terraform.tfvars.json` â€” the filename `tofu` reads for back-compat).

## Stage 5: OpenTofu Execution

<Steps>

### Template Selection
The Runner resolves the template directory based on `ProjectConfig.Provider` â€” one of `aws/`, `gcp/`, or `azure/` under the templates path.

### Backend Configuration
OpenTofu state is stored in S3-compatible storage. The backend key follows the pattern `{project}-{env}-{region}/tofu.tfstate`. See [OpenTofu State](/concepts/terraform-state).

### Plan
`tofu plan` runs with the generated tfvars. For PLAN jobs, an Infracost analysis is also generated. The plan artifact is uploaded to S3-compatible storage.

### Apply
For DEPLOY jobs, `tofu apply` runs the plan. If a `plan_job_id` is linked, the cached plan artifact is downloaded and applied directly.

### Post-Apply
After apply, the runner extracts `tofu` outputs (cluster endpoint, database endpoints, ARNs), writes an exec-plugin kubeconfig in-process (CLI-free â€” see [GitOps & ArgoCD](/concepts/gitops-argocd)), installs ArgoCD, and renders application templates (per cloud â€” see [Multi-cloud](/elench/multi-cloud)). It then **generates app manifests** for the detected services and scaffolds them into the GitOps repo â€” only when that repo has no manifests yet, so a bring-your-own repo is never overwritten. See [GitOps & ArgoCD](/concepts/gitops-argocd).

Between `plan` and `apply` the [elench verification gate](/elench/engine-internals) runs (fail-closed) and seals a signed receipt.

</Steps>

## Plan Hash Validation

When a DEPLOY job references a prior PLAN job, the platform validates that the configuration hasn't changed between plan and apply. If the `config_snapshot` hash differs, the deploy fails with a hash mismatch error and the user must re-plan. This prevents applying a stale plan against a modified Project.
