# Job Detail & Logs

The job detail page (`/dashboard/jobs/{id}`) shows the full execution state of a job with real-time log streaming. For the underlying streaming architecture, see [Real-time Architecture](/concepts/realtime).

## Layout

### Header
- Job type icon and label
- Status badge (animated for PROCESSING)
- Duration (elapsed time, updates live)
- Job ID (first 8 chars, copyable)
- Assigned Runner name
- Created timestamp
- Action buttons

### Log Viewer

A full-height, monospace, auto-scrolling log area showing the runner's `tofu` (OpenTofu) output in real-time:

- **STDOUT** — normal output (white/light gray)
- **STDERR** — errors and warnings (red)
- **SYSTEM** — platform messages like "Assuming IAM role..." (blue)

Logs are streamed via a Postgres LISTEN/NOTIFY → SSE subscription on `provision_job_logs` INSERT events, filtered by `job_id`. New chunks appear within ~2–3 seconds of the Runner writing them.

Auto-scroll pins to the bottom as new logs arrive. Scrolling up manually disables auto-scroll until the user scrolls back to the bottom.

### Collapsible Details

Three expandable sections below the logs:

- **Job Details** — full metadata (ID, type, status, runner, all timestamps)
- **Config Snapshot** — the project configuration at time of job creation (JSON)
- **Execution Metadata** — OpenTofu outputs, cost data, cluster info, and the **`verify_result`** report (JSON)

For PLAN and DEPLOY jobs, `execution_metadata["verify_result"]` carries the [verify gate](/console/jobs/plan-and-apply#verify-gate) report — the fail-closed policy evaluation over the plan JSON. The Console renders it as the **Verify Report** in the project's Plan tab; see [engine internals](/elench/engine-internals).

## Actions

| Button | When Visible | Effect |
|--------|-------------|--------|
| **Cancel** | QUEUED or PROCESSING | Sets status to CANCELLED, Runner terminates gracefully |
| **Apply** | PLAN job with SUCCESS status | Opens Runner Select Popover → creates DEPLOY job |
| **Re-run** | Any terminal status (SUCCESS, FAILED, CANCELLED) | Creates a new job with the same configuration |

## Status Polling

The page polls `getJobStatus(jobId)` every 3 seconds until the job reaches a terminal state. This is a lightweight query that returns only `status` and `error_message` — the full log streaming happens via the Realtime subscription.

## Status Banner

When the job reaches a terminal state, a banner appears:

- **SUCCESS** — green banner with checkmark
- **FAILED** — red banner with error message
- **CANCELLED** — gray banner with "Cancelled by user"
