# Jobs

Every infrastructure operation is modeled as a **job**. The Jobs page shows all jobs with filtering, search, and real-time status updates. For the technical deep dive on how jobs are queued and claimed, see [Job Queue Pattern](/concepts/job-queue).

## Job List Page

Located at `/dashboard/jobs`, the page shows:

### Filters

**Status filters** (button group):
- All, QUEUED, PROCESSING, SUCCESS, FAILED, CANCELLED

**Type filters** (button group):
- All Types, DEPLOY, PLAN, DESTROY, CONNECTION_TEST, FETCH_RESOURCES, DEPLOY_RUNNER, UPDATE_RUNNER, DESTROY_RUNNER

**Search** — filter by job ID, Runner name, or project name.

### Data Table

| Column | Description |
|--------|-------------|
| Type | Job type with icon |
| Status | Color-coded badge |
| Project | Target project name |
| Runner | Assigned Runner name |
| Started | Timestamp |
| Completed | Timestamp |
| Duration | Elapsed time |

Click any row to navigate to the [job detail page](/console/jobs/job-detail).

### Real-time Updates

The jobs store receives real-time events from the dashboard layout's LISTEN/NOTIFY → SSE subscription. New jobs appear automatically, and status changes update badges in-place without page refresh. See [Real-time Architecture](/concepts/realtime).

## Runner Select Popover

When triggering Plan, Apply, or Destroy from a project, the **Runner Select Popover** appears:

- Lists all registered Runners with status indicators (green = ONLINE, yellow = DRAINING, red = OFFLINE)
- "Any available" option for automatic assignment
- Only ONLINE Runners can accept new jobs

The selected Runner's ID is stored as `assigned_runner_id` on the job.
