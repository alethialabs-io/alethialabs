# Project Basics

The first section of the Design a Project form establishes the project identity and its default target environment. A project owns **N environments** (`project_environments`); this form seeds the default one.

## Fields

| Field | Type | Validation | Description |
|-------|------|-----------|-------------|
| **Project Name** | Text input | Required, lowercase, a-z0-9 + hyphens, max 25 chars | Used in resource naming and OpenTofu state paths |
| **Environment Stage** | Select | Required | `development`, `staging`, or `production` — the environment's `stage` |
| **OpenTofu Version** | Select | Required | The `iac_version` field — default: 1.11.4. Also available: 1.10.5, 1.9.8 |

## Project Name

The project name is used extensively:
- Resource naming in OpenTofu (e.g., `{project}-{env}-eks-cluster`)
- State file path (see [OpenTofu State](/concepts/terraform-state))
- Display in the UI (project cards, job listings)

<Callout type="info">
Names are enforced lowercase with only letters, numbers, and hyphens. This ensures compatibility with cloud resource naming constraints across all three providers.
</Callout>

## Environment Stage

`stage` is a property of the **environment**, not the project. A project can own several environments
(dev / staging / prod), each independently provisionable with its own `stage`. The form's three options:

| Stage | Purpose | Naming Convention |
|-------|---------|------------------|
| `development` | Dev/testing | `{project}-dev-*` |
| `staging` | Pre-production | `{project}-staging-*` |
| `production` | Production workloads | `{project}-prod-*` |

The environment stage affects OpenTofu state isolation — each stage gets its own state file, even within the same project and region.
