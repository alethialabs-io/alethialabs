# Network

The network section configures the virtual network that contains all provisioned resources.

## Create New vs. Use Existing

Two modes controlled by a toggle:

### Create New Network

Provisions a new network from scratch. Fields:

| Field | Description |
|-------|-------------|
| **CIDR Block** | IP address range (e.g., `10.0.0.0/16`) |
| **NAT Gateway** | Single (cost-optimized) or HA (one per availability zone) |

<Tabs items={["AWS", "GCP", "Azure"]}>
  <Tab value="AWS">
    Creates a **VPC** with:
    - Public and private subnets across availability zones
    - Internet Gateway for public subnets
    - NAT Gateway(s) for private subnet internet access
    - Route tables for each subnet type
    - Default security group
  </Tab>
  <Tab value="GCP">
    Creates a **VPC Network** with:
    - Custom-mode subnets in the selected region
    - Cloud NAT with Cloud Router
    - Firewall rules for internal communication
  </Tab>
  <Tab value="Azure">
    Creates a **Virtual Network (VNet)** with:
    - Subnets for cluster nodes and services
    - NAT Gateway for outbound traffic
    - Network Security Groups (NSGs)
  </Tab>
</Tabs>

### Use Existing Network

Instead of creating a network, attach the project to one that **already exists** in the account — its id
comes from Alethia's server-side [asset inventory](/console/connectors/health-and-inventory) (network id,
name, CIDR, region).

**How it works — brownfield attach (AWS · GCP · Azure).** When you pick "existing", Alethia records the
network id and, at deploy time, the [Runner](/runner) does **not** create a network. Instead the OpenTofu
template *reads* your existing network and wires the cluster into it:

- **AWS** — the runner looks up the VPC's subnets and passes them in (public/private split).
- **GCP** — the template `data`-sources the VPC network and resolves its subnetwork in the project's
  region (including that subnet's pod/service secondary ranges) for GKE.
- **Azure** — the template `data`-sources the VNet and uses one of its subnets for AKS.

<Callout type="info">
Existing networks must be in the same region as the project. The list comes from Alethia's server-side
[asset inventory](/console/connectors/health-and-inventory), which stays fresh automatically (a
reconciliation sweep, plus optional event forwarders) — there's no manual refresh.
</Callout>

## Reuse existing resources

"Use existing network" is one case of a deliberately **narrow** idea: reuse the **shared substrate you
deploy _into_**, instead of creating a duplicate. The test for whether reuse is a provisioning concern is
simple — **does the rest of the infrastructure graph attach to it?**

- ✅ **Network (VPC/VNet + subnets)** — the cluster and nodes attach into its subnets. Available today on
  AWS, GCP, and Azure (a provision-new vs use-existing toggle; at deploy time the template `data`-sources
  the existing network rather than creating one). The list comes from the server-side
  [asset inventory](/console/connectors/health-and-inventory).
- 🚧 **Kubernetes cluster** — your workloads deploy _onto_ it (nobody wants a fresh cluster per app). Same
  toggle, on the roadmap.

That's the whole scope. **Databases, caches, queues, storage, etc. are _not_ "reuse" targets** — nothing
downstream attaches to them in the graph. If a database already exists, there's nothing to provision:
the app simply needs its **connection endpoint + secret**, which is a *connection/config* concern (a value
you supply), not a provisioned component. Trying to model that as "provision vs existing" just reintroduces
a credential-handling problem the keyless model otherwise avoids.

<Callout type="info">
The inventory still tracks every resource kind (databases, caches, …) — but for **drift/day-2 awareness,
collision-avoidance, and agent context**, not for reuse. Storing what exists is not the same as attaching
your project to it.
</Callout>

## CIDR Block Validation

The form validates CIDR notation in real-time and shows a breakdown. For `10.0.0.0/16`:

| Field | Value |
|-------|-------|
| Total addresses | 65,536 |
| Usable IPs | 65,531 |
| Range | 10.0.0.0 – 10.0.255.255 |
| Size | Large |

Common ranges:
- `/16` — 65,536 addresses (recommended for production)
- `/20` — 4,096 addresses (suitable for small environments)
- `/24` — 256 addresses (minimal, development only)

## NAT Gateway Cost Impact

NAT gateways have significant cost implications:

| Mode | AWS Cost | Description |
|------|---------|-------------|
| Single | ~$35/mo | One NAT in a single AZ (cost-optimized, single point of failure) |
| HA | ~$105/mo (3 AZ) | One NAT per AZ (high availability, 3x cost) |

The [cost sidebar](/console/design-project/cost-sidebar) reflects the NAT gateway choice immediately.

## Allowed CIDR Blocks

Optional field for restricting inbound access. Enter CIDR ranges that should be allowed to connect to resources within the network. Default: open to the VPC CIDR only.
