# Design a Project

The design **canvas** is where you define infrastructure. You add services as nodes, configure each one in its inspector, and Plan then Deploy the environment. The canvas is the sole design surface — the legacy multi-section form has been removed.

## The canvas

Every project opens on its **Architecture** view. The root **Project node** holds the project's basics — name, environment stage, and cloud region — and pins the stack's core cloud account, which is chosen once when the project is created. Every other service you add is a node wired to the graph.

Under the hood the canvas persists through an abstract Zod schema and a graph ⇄ form round-trip, so a shared `react-hook-form` context still backs a few inspector fields (repository and region pickers). You interact with nodes and inspectors, not a form.

## Adding services

Add services from the **Add** palette or the `⌘K` command palette. The **Add** palette groups services by domain (Workloads, Data, Storage, Messaging, Security, Networking, Compute, DevOps) and matches on each entry's label and subtitle. The `⌘K` command palette groups services into **Core** and **Periphery** and searches across every provisionable service by name — its per-cloud service names (e.g. `rds`, `postgres`, `redis`, `s3`) and synonyms. Selecting one drops its node on the canvas and opens its inspector; kinds with variants (for example a database engine) ask you to pick the variant first.

The palette is filtered by the project's cloud: a service the target cloud can't provision (Hetzner has no managed pub/sub or NoSQL, for instance) isn't offered, and a singleton already on the canvas is disabled. Services not yet backed by a Terraform module show a `Soon` badge.

## Configuring a node

Each node opens its **inspector** — a set of collapsible config sections. The node's effective provider drives its fields, region choices, and provider-specific advanced knobs. The cloud account is not set per node; it is inherited from the Project root.

Existing resources you can reuse (VPCs, subnets, and the like) come from Alethia's server-side
[asset inventory](/console/connectors/health-and-inventory), which stays current automatically
(event-driven + a reconciliation sweep). There's no manual refresh.

<Callout type="info">
The cloud account is chosen once, at project creation — there is no in-canvas provider switch. When you duplicate a project onto a different cloud, the platform converts the config and surfaces conversion warnings. See [Multi-Cloud Conversion](/concepts/multi-cloud-conversion).
</Callout>

## Save, Plan, Deploy

Saving a project writes a `projects` row and its **default environment** (a `project_environments` row with status `DRAFT`), then a component row per node in the relevant tables (`project_network`, `project_cluster`, `project_databases`, and so on). The project then appears in the project listing.

From the Architecture view you drive the environment with the **Run** menu and the staged-changes bar:

- **Plan** (`PLAN`) — dry-run the OpenTofu plan and price it with Infracost.
- **Deploy** — apply the staged changes. **Destroy** lives in Project settings.
- **Audit**, **Detect drift**, and **Probe cluster** run their respective jobs against the live environment.

## Validation

Each service's config is validated by its own Zod schema (`lib/validations/project-form.schema.ts`). For example:

- Project name: free-text display name, max 50 characters; the URL slug is derived via `slugify` (must contain at least one letter or number).
- Bucket names: 3–63 lowercase letters, digits, or hyphens; must start and end with a letter or digit.
- Component names (database, cache, queue, topic, table, secret) are required.

A service the target cloud can't provision, or a service binding to a resource that isn't in the environment, is rejected fail-closed at deploy time rather than silently dropped.
