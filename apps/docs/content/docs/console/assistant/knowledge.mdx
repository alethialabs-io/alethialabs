# Knowledge base

The **Knowledge** panel (in the rail) is where you tell Elench how to behave and what it should always
know for the current [workspace](/console/assistant/workspaces). Everything here rides the system prompt
of **every chat in that scope**, so the first answer is already grounded — you don't repeat yourself
each conversation.

It has three parts.

## Custom instructions

A free-text field for *how* Elench should act here. Examples:

> This is production — always require approval before an apply, and never propose destroy.

> Prefer Hetzner for non-prod. Our region is `eu-central-1` unless a project says otherwise.

Instructions save on blur and apply to every chat in the workspace.

## Knowledge documents

The knowledge itself is a **list of named documents**, not one blob — so each is titled, sized, and
removable on its own, and Elench can tell you *which* document it drew on.

<Steps>

### Add a document

Click **Add knowledge**, give it a title (e.g. *Runbook: production deploys*), and write the body.

### Save it

**Save document** persists it to the workspace. It appears in the list with its size and when it was
last edited.

### Edit or remove

Hover a document for **edit** and **delete**. Deleting it drops it from every future chat immediately.

</Steps>

Because knowledge rides **every** turn's prompt, its size is a real cost — a **capacity meter** shows
how much of the budget (50,000 characters) you've used. Keep documents tight and factual: conventions,
owners, on-call, runbooks, gotchas — the things Elench can't work out on its own.

## Already known (auto-derived)

Inside a **project** workspace, the panel also shows an **Already known** section: the live facts Elench
derives from your account on every turn — the project's identity and region, its environments and their
status, and its recent jobs and outcomes. You don't need to write any of this down; it's read fresh each
time. The section makes it obvious what's *already* grounded versus what's worth pinning by hand.

<Callout type="info">
  The organization workspace has no auto-derived section — it's the general assistant, grounded by the
  org-wide instructions and documents you pin, plus the tools it can call on demand.
</Callout>

## How it reaches the model

Under the hood, a scope's instructions and documents are the `agent_context` row for that
`(organization, project)` pair; they're assembled — org first, project second, then the derived facts —
into the cached system prompt for each turn. The [agent runtime](/elench/agent-runtime) covers the
prompt assembly and the tenant-isolation guards.
