# AWS Connector

AWS integration is **keyless and issuer-direct**: your account registers an **IAM OIDC provider** that trusts
Alethia's OIDC issuer, plus an IAM role Alethia assumes via **`AssumeRoleWithWebIdentity`** — no access keys, no
external id, and no trust in an Alethia AWS account. This is the same model GCP, Azure, and Alibaba use. See
[Security Architecture](/concepts/security) for the full model.

<Callout type="info">
**Is this safe?** You create the trust entirely in **your own** AWS account — Alethia never sees a key.
It signs in with a short-lived token (≤10 min) that AWS verifies and swaps for a ~1-hour credential. The
only thing Alethia stores is the **role ARN** (a public identifier). **Revoke anytime** by deleting the
stack or the role.
</Callout>

## Setup Methods

<Tabs items={["CloudFormation (Recommended)", "Terraform"]}>
  <Tab value="CloudFormation (Recommended)">
    <Steps>

    ### Launch CloudFormation Stack
    Click "Launch Stack in AWS" — this opens the AWS Console with a pre-filled CloudFormation template
    (the issuer URL is filled in for you) that creates:
    - An **IAM OIDC provider** for the Alethia issuer (`https://alethialabs.io/api/oidc`)
    - `AlethiaProvisionerRole` — an IAM role whose trust policy allows **only** `sts:AssumeRoleWithWebIdentity`
      from that provider, pinned to the fixed audience (`sts.amazonaws.com`) and subject (`alethia-connector`)

    ### Acknowledge and Create
    In the AWS Console, acknowledge that CloudFormation will create IAM resources, then click "Create stack."

    ### Copy Role ARN
    After the stack completes (~1 minute), go to the Outputs tab and copy the Role ARN.

    ### Paste into Alethia
    Paste the Role ARN into the Alethia form and click "Connect."

    </Steps>
  </Tab>
  <Tab value="Terraform">
    <Steps>

    ### Apply the Terraform module
    Download the module ("Download module" in the connect sheet) and run `terraform apply`. It creates the
    IAM OIDC provider + role that trust the Alethia issuer. **No variables are required** — the defaults pin
    the issuer, audience (`sts.amazonaws.com`), and subject (`alethia-connector`). Self-hosters override
    `issuer_url` to point at their own console.

    ### Copy Role ARN
    Copy the `role_arn` output from Terraform.

    ### Paste into Alethia
    Paste into the form and click "Connect."

    </Steps>
  </Tab>
</Tabs>

<Callout type="info">
One IAM OIDC provider per issuer URL is allowed per AWS account, so connect an account **once**. To reconnect
after deleting the stack, remove the existing provider first (or reuse the stack).
</Callout>

## Verification

After submitting, Alethia verifies the connection **instantly, server-side** (no runner):

1. The console mints a short-lived assertion and calls `sts:AssumeRoleWithWebIdentity` on your role
2. Calls `sts:GetCallerIdentity` to confirm the assumed identity (proof of access)
3. Runs a small capability probe (representative reads) → the connection is `Connected`, or `Degraded`
   when provisioning permissions are missing, or `Disconnected` when auth fails

If verification fails, the error message explains what went wrong (e.g., trust policy misconfiguration,
missing permissions). Existing resources (VPCs, subnets, …) are discovered separately by the
server-side [asset inventory](/console/connectors/health-and-inventory), not during this probe.

## IAM Role Permissions

The role is created with the AWS-managed `AdministratorAccess` policy so it can provision every
Terraform-managed resource type (EC2/VPC, EKS, RDS, ElastiCache, Route 53, Secrets Manager, DynamoDB, SQS/SNS,
ECR, S3, IAM for IRSA, ACM, WAF). You can attach a tighter policy instead — the trust relationship (who may
assume the role) is independent of what the role is allowed to do.

## What's Stored

Only the **Role ARN** (`arn:aws:iam::123456789012:role/AlethiaProvisionerRole`) is stored. No access keys,
secret keys, session tokens, or external id are ever persisted — every credential is a short-lived assertion
minted at call time.
