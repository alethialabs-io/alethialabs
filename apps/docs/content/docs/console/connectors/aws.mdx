# AWS Connector

Connecting AWS registers a **cloud identity** (`cloud_identities`) that is **keyless and issuer-direct**: your account registers an **IAM OIDC provider** that trusts
Alethia's OIDC issuer, plus an IAM role Alethia assumes via **`AssumeRoleWithWebIdentity`** — no access keys, no
external id, and no trust in an Alethia AWS account. This is the same model GCP, Azure, and Alibaba use. See
[Security Architecture](/concepts/security) for the full model.

<Callout type="info">
**Is this safe?** You create the trust entirely in **your own** AWS account — Alethia never sees a key.
It signs in with a short-lived token (≤10 min) that AWS verifies and swaps for a ~1-hour credential. The
only thing Alethia stores is the **role ARN** (a public identifier). **Revoke anytime** by deleting the
IAM role (and its OIDC provider).
</Callout>

## Setup Methods

Two equivalent, keyless paths — the **AWS CloudShell** script (nothing to install) or the **OpenTofu**
module. Both create the same IAM OIDC provider + `AlethiaProvisionerRole` in your account.

<Tabs items={["AWS CLI / CloudShell", "OpenTofu"]}>
  <Tab value="AWS CLI / CloudShell">
    <Steps>

    ### Open AWS CloudShell
    Open [AWS CloudShell](https://console.aws.amazon.com/cloudshell/home) — the `aws` CLI is
    preinstalled and already authenticated. (Or run the script locally with a configured `aws` CLI.)

    ### Run the setup script
    Paste the one-liner from the connect sheet — `curl -sO …/alethia-aws-setup.sh && bash alethia-aws-setup.sh`.
    It creates the **IAM OIDC provider** for the Alethia issuer + `AlethiaProvisionerRole` (whose trust
    allows **only** `sts:AssumeRoleWithWebIdentity` from that provider, pinned to audience
    `sts.amazonaws.com` and subject `alethia-connector`), attaches the five least-privilege policies, and
    prints `role_arn`. Self-hosters pass their issuer as the first argument.

    ### Copy Role ARN
    Copy the `role_arn` the script prints and paste it into the form, then click "Connect."

    </Steps>
  </Tab>
  <Tab value="OpenTofu">
    <Steps>

    ### Apply the OpenTofu module
    Download the module ("Download module" in the connect sheet) and run `tofu apply`. It creates the
    IAM OIDC provider + role that trust the Alethia issuer. **No variables are required** — the defaults pin
    the issuer, audience (`sts.amazonaws.com`), and subject (`alethia-connector`). Self-hosters override
    `issuer_url` to point at their own console.

    ### Copy Role ARN
    Copy the `role_arn` output from OpenTofu.

    ### Paste into Alethia
    Paste into the form and click "Connect."

    </Steps>
  </Tab>
</Tabs>

<Callout type="info">
One IAM OIDC provider per issuer URL is allowed per AWS account, so connect an account **once**. To reconnect
after deleting the role, re-run the script (it reuses the existing provider) or remove the provider first.
</Callout>

## Verification

After submitting, Alethia verifies the cloud identity **instantly, server-side** (no runner):

1. The console mints a short-lived assertion and calls `sts:AssumeRoleWithWebIdentity` on your role
2. Calls `sts:GetCallerIdentity` to confirm the assumed identity (proof of access)
3. Runs a small capability probe (representative reads) → the cloud identity is `Connected`, or `Degraded`
   when provisioning permissions are missing, or `Disconnected` when auth fails

If verification fails, the error message explains what went wrong (e.g., trust policy misconfiguration,
missing permissions). Existing resources (VPCs, subnets, …) are discovered separately by the
server-side [asset inventory](/console/connectors/health-and-inventory), not during this probe.

## IAM Role Permissions

What the role may *do* is independent of *who may assume it* (the OIDC trust above) — a separate axis.

**Exactly what Alethia provisions in your account** (feature-gated — a given Project uses a subset): EC2/VPC
(subnets, NAT, route tables, security groups, VPC endpoints), EKS (cluster, managed node groups, add-ons, the
IRSA OIDC provider + controller roles), RDS (Aurora), ElastiCache (Redis/Valkey), Route 53, ACM, Secrets
Manager, DynamoDB, SQS/SNS, ECR, S3, KMS envelope keys, CloudWatch Logs + Kinesis Firehose, EventBridge
(Karpenter), Application Auto Scaling, and the in-cluster controller IAM roles (external-dns, AWS Load
Balancer Controller, EBS CSI, Karpenter).

<Callout type="info">
**Least-privilege by default — no account admin, fully action-enumerated.** The setup attaches **five scoped
policies** instead of `AdministratorAccess`: `…-ComputeNet`, `…-ComputeScale`, `…-Data`, `…-Edge` — the
data-plane, each listing the **specific `ec2:`/`eks:`/`rds:`/… actions** the templates issue (not
`service:*`; split into net/scale to fit AWS's 6144-char managed-policy limit) — plus an action-enumerated
`…-IAM` policy. The IAM policy lets the templates create the EKS/IRSA roles they need but **explicitly
denies** attaching `AdministratorAccess`/`PowerUserAccess`/`IAMFullAccess` to anything, scopes `PassRole` to
the provisioned services, and denies all `organizations:*`/`account:*`/billing — so the role **cannot
escalate to admin**. Reads stay `Describe*`/`Get*`/`List*` so a template/module bump doesn't break a real
apply. A defence-in-depth **permissions boundary** on the created roles is the next hardening increment. The
OIDC trust is independent of all this.
</Callout>

## What's Stored

Only the **Role ARN** (`arn:aws:iam::123456789012:role/AlethiaProvisionerRole`) is stored. No access keys,
secret keys, session tokens, or external id are ever persisted — every credential is a short-lived assertion
minted at call time.
