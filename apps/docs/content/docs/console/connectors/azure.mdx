# Azure Connector

Azure integration is **keyless**. Alethia registers **one multi-tenant Entra app** whose Federated
Identity Credential trusts the Alethia OIDC issuer. Both the console (verification) and the runner
(provisioning) authenticate **as that app** by presenting a short-lived OIDC assertion; Azure validates
it and issues an ARM access token. **You register no app and store no client secret** — you only grant
Alethia's app a role in your tenant. See [Security Architecture](/concepts/security).

<Callout type="info">
**Is this safe?** You never create an app or a secret — you only grant Alethia's app a role in **your own**
subscription. Alethia signs in with a short-lived token (≤10 min) that Entra ID verifies and swaps for a
~1-hour credential. The only thing stored is your **tenant + subscription id** (public). **Revoke anytime**
by removing the app's role assignment.
</Callout>

## Setup

The connect dialog gives you a one-line `az` command (or a Terraform module). It does two things in your
tenant — no app registration, no federated credential on your side:

<Steps>

### Create a service principal for Alethia's app
`az ad sp create --id <ALETHIA_AZURE_CLIENT_ID>` creates a service principal for Alethia's multi-tenant
app in **your** tenant. The app itself lives in Alethia's tenant; you're just consenting to it.

### Assign RBAC roles
Grant that service principal appropriate roles on your subscription:
- Contributor (or more granular roles for specific resource types)
- User Access Administrator (for AKS AAD integration)

### Enter details in Alethia
Paste back the two values the script prints:
- **Tenant ID** — your Entra directory ID
- **Subscription ID** — the target Azure subscription

(The Client ID is Alethia's own app id — fixed, not something you provide.)

</Steps>

## Verification

Same as other providers — Alethia verifies access **instantly, server-side** (no runner):

1. The console mints an OIDC assertion for Alethia's app and presents it to your tenant
2. Azure validates it against the app's federated credential and returns an ARM access token
3. A small ARM read confirms provisioning access → `Connected` / `Degraded` / `Disconnected`

Existing VNets, subnets, and DNS zones are discovered separately by the server-side
[asset inventory](/console/connectors/health-and-inventory).

## What's Stored

- **Tenant ID**
- **Subscription ID**
- The platform **Client ID** (Alethia's app id — the same for every connection)

**No client secrets, ever.** Revoke access by deleting the service principal (or its role assignment)
in your tenant.
