# Azure Connector

Connecting Azure registers a **cloud identity** (`cloud_identities`) that is **keyless** and
**customer-side**. You create a **user-assigned managed identity** in your own subscription (a plain ARM
resource — no App Registration, no directory-admin rights) with a Federated Identity Credential that
trusts the Alethia OIDC issuer. Both the console (verification) and the runner (provisioning) authenticate
**as that identity** by presenting a short-lived OIDC assertion; Azure validates it and issues an ARM
access token. **You store no client secret**, and there is **no platform Entra app** — parity with the GCP
Workload Identity Federation model. See [Security Architecture](/concepts/security).

<Callout type="info">
**Is this safe?** You create the identity in **your own** subscription and grant it a least-privilege role.
Alethia signs in with a short-lived token (≤10 min) that Entra ID verifies and swaps for a ~1-hour
credential. The only things stored are your **tenant, subscription, and managed-identity client id** (all
public). **Revoke anytime** by deleting the managed identity.
</Callout>

## Setup

The connect dialog gives you a one-line `az` command (or an OpenTofu module). It runs entirely in **your**
subscription:

<Steps>

### Create the managed identity + federated credential
`az identity create` provisions a user-assigned managed identity, and
`az identity federated-credential create` adds a credential trusting the Alethia issuer (subject
`alethia-connector`, audience `api://AzureADTokenExchange`). No App Registration, no secret.

### Assign the least-privilege role
The script creates and assigns the **Alethia Provisioner** custom role — scoped to exactly the services
the project templates create, with an ABAC condition that limits role-assignment writes to the single
DNS Zone Contributor role (no path to self-escalation).

### Enter details in Alethia
Paste back the three values the script prints:
- **Tenant ID** — your Entra directory ID
- **Client ID** — the managed identity's application id
- **Subscription ID** — the target Azure subscription

</Steps>

## Verification

Same as other providers — Alethia verifies access **instantly, server-side** (no runner):

1. The console mints an OIDC assertion for your managed identity and presents it to your tenant
2. Azure validates it against the identity's federated credential and returns an ARM access token
3. A small ARM read confirms provisioning access → `Connected` / `Degraded` / `Disconnected`

Existing VNets, subnets, and DNS zones are discovered separately by the server-side
[asset inventory](/console/connectors/health-and-inventory).

## What's Stored

- **Tenant ID**
- **Client ID** (your managed identity's application id)
- **Subscription ID**

**No client secrets, ever.** Revoke access by deleting the managed identity (or its role assignment)
in your subscription.
