# Alibaba Connector

Connecting Alibaba registers a **cloud identity** (`cloud_identities`) that is **keyless and account-free**. Alethia's control plane is its own OIDC issuer;
you register a **RAM OIDC provider** that trusts it and a **RAM role** that trusts that provider. Alethia
assumes the role by presenting a short-lived minted assertion (STS `AssumeRoleWithOIDC`) ‚Äî Alethia holds
no Alibaba account, generates no AccessKey, and stores nothing but the role ARN. See
[Security Architecture](/concepts/security).

<Callout type="info">
**Is this safe?** You create the trust entirely in **your own** Alibaba account ‚Äî Alethia holds no Alibaba
account and generates no AccessKey. It signs in with a short-lived token (‚â§10 min) that Alibaba STS verifies
and swaps for a ~1-hour credential. The only thing stored is the **RAM role ARN** (a public identifier).
**Revoke anytime** by deleting the role.
</Callout>

## Setup

The connect dialog gives you two equivalent options ‚Äî a one-line `aliyun` command for the
**Alibaba Cloud Shell** (nothing to install), or the **OpenTofu module** for teams that manage
infrastructure as code. Both create everything in **your** Alibaba account:

<Steps>

### Create the RAM OIDC provider
A RAM OIDC provider named `alethia` trusting Alethia's issuer (`https://alethialabs.io/api/oidc`), with
client id `sts.aliyuncs.com`. The provider name is fixed ‚Äî Alethia derives its ARN from your role ARN.
The provider pins the issuer's **CA certificate fingerprints** (the stable intermediate/root, not the
rotating Cloudflare edge leaf); if the issuer's cert chain ever rotates its CA, re-run the setup to refresh
them.

### Create the RAM role
A role whose trust policy allows `AssumeRoleWithOIDC` from that provider, scoped to Alethia's workload
identity (`oidc:sub = alethia-connector`, `oidc:aud = sts.aliyuncs.com`) ‚Äî a wrong subject/audience is
rejected.

### Attach provisioning permissions
The setup attaches **4 enumerated custom RAM policies** ‚Äî `ComputeCluster` (ACK + its transitive node-pool
ECS + API-server/Service SLB), `Network` (VPC/vSwitch/NAT/SNAT + EIP), `Data` (RDS/KVStore/OSS/Tablestore/
KMS), and `EdgeReg` (Container Registry/AliDNS/MNS/WAF) ‚Äî each listing the **specific
create/read/modify/delete/tag actions** the templates issue, never a `service:*` wildcard or an admin
policy. Service-linked roles ACK/NAT need are covered by a narrow `ram:CreateServiceLinkedRole`-only policy.

### Paste the role ARN into Alethia
Run `tofu output` and paste `role_arn` ‚Äî it looks like `acs:ram::<account>:role/<name>`. That's the
only value Alethia needs (the OIDC-provider ARN is derived from it).

</Steps>

## Verification

Alethia verifies the cloud identity **instantly, server-side** (no runner):

1. The console mints an OIDC assertion (audience `sts.aliyuncs.com`) and calls Alibaba STS
   `AssumeRoleWithOIDC` with your role + provider ARNs
2. Alibaba validates the assertion against the RAM OIDC provider and returns temporary credentials
3. A successful assume is the proof of access ‚Üí `Connected` / `Disconnected`

## What's Stored

- The **RAM role ARN**
- The derived **RAM OIDC provider ARN** (`acs:ram::<account>:oidc-provider/alethia`)
- The **account id** (parsed from the role ARN)

**No credentials, ever.** Revoke access by deleting the RAM role (or the OIDC provider) in your account.
