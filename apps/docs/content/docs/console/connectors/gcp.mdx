# GCP Connector

GCP integration uses **Workload Identity Federation (WIF)** with **direct OIDC** — your project's WIF pool
trusts Alethia's issuer directly (no AWS hop), and Alethia federates in with a short-lived minted JWT. No
service account JSON keys are created or stored. See [Security Architecture](/concepts/security).

The connect sheet gives you two one-shot paths that create everything for you and print a **credential
configuration JSON** — you don't hand-build the WIF resources. Both pin the same values the console mints:
issuer `https://alethialabs.io/api/oidc`, audience `alethia-gcp-wif`, subject `alethia-connector`.

    ### Run the setup script
    From the connect sheet, open **Google Cloud Shell** (or run locally with `gcloud`) and run the provided
    `alethia-gcp-setup.sh <PROJECT_ID>`. It enables the required APIs, creates the provisioner service
    account, a **Workload Identity Pool + OIDC provider** trusting the Alethia issuer, and binds the fixed
    subject `alethia-connector` to the service account.

    ### Copy the credential config
    The script prints the WIF credential configuration between `--- START CONFIG ---` and `--- END CONFIG ---`.

    ### Paste into Alethia
    Paste that block into the connect sheet and click "Connect."

## Verification

Alethia verifies the connection **instantly, server-side** (no runner):

1. The console presents an OIDC token to GCP's Security Token Service (Workload Identity Federation)
2. GCP validates the token against the registered issuer/subject and returns a short-lived access token
3. A small Compute read confirms provisioning access → `Connected` / `Degraded` / `Disconnected`

Existing resources (VPC networks, subnets, DNS zones) are discovered separately by the server-side
[asset inventory](/console/connectors/health-and-inventory).

## What's Stored

- **GCP Project ID**
- **WIF configuration** — the credential config JSON you paste (pool + provider audience, the service
  account impersonation URL, and the token-file credential source). The provisioner service account is
  referenced by that impersonation URL; there's no separate key field.

No service account JSON keys. Access is revoked by deleting the WIF pool or unbinding the service account.
