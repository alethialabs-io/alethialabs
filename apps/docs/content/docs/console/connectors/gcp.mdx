# GCP Connector

Connecting GCP registers a **cloud identity** (`cloud_identities`) that uses **Workload Identity Federation (WIF)** with **direct OIDC** — your project's WIF pool
trusts Alethia's issuer directly (no AWS hop), and Alethia federates in with a short-lived minted JWT. No
service account JSON keys are created or stored. See [Security Architecture](/concepts/security).

<Callout type="info">
**Is this safe?** You create the trust entirely in **your own** GCP project — no service account key is
ever generated. Alethia signs in with a short-lived token (≤10 min) that GCP verifies and swaps for a
~1-hour access token. The only thing stored is the **WIF trust configuration** (no secret). **Revoke
anytime** by deleting the pool or unbinding the service account.
</Callout>

The connect sheet gives you two one-shot paths that create everything for you. When they finish, you enter
just **two values** — your **Project ID** and **Project Number** — and Alethia assembles the WIF credential
configuration for you (there is no JSON to copy). Both paths pin the same values the console mints: issuer
`https://alethialabs.io/api/oidc`, audience `alethia-gcp-wif`, subject `alethia-connector`, and the fixed
pool `alethia-pool` / provider `alethia-oidc-provider` / service account `alethia-provisioner`.

  <Tab value="Setup script (Cloud Shell)">
    <Steps>

    ### Run the setup script
    From the connect sheet, open **Google Cloud Shell** (or run locally with `gcloud`) and run the provided
    `alethia-gcp-setup.sh <PROJECT_ID>`. It enables the required APIs, creates the provisioner service
    account, a **Workload Identity Pool + OIDC provider** trusting the Alethia issuer, and binds the fixed
    subject `alethia-connector` to the service account.

    ### Enter the two values it prints
    When it finishes, the script prints your **Project ID** and **Project Number**. Enter both in the connect
    sheet — Alethia assembles the credential configuration from them and the fixed pool/provider/SA names.

    ### Connect
    Click **Connect**. Alethia verifies the federation instantly, server-side (see below).

    </Steps>
  </Tab>

<Callout type="info">
**Why a Project Number and not just the ID?** The WIF audience is keyed on your project's **numeric project
number** (`//iam.googleapis.com/projects/<NUMBER>/locations/global/workloadIdentityPools/...`), not the
project ID — and it can't be derived from the ID without already being authenticated (it's needed to build
the token-exchange request). The project ID separately forms the service-account email. Both are printed by
the setup script, so it's a two-value copy.
</Callout>

## Verification

Alethia verifies the cloud identity **instantly, server-side** (no runner):

1. The console presents an OIDC token to GCP's Security Token Service (Workload Identity Federation)
2. GCP validates the token against the registered issuer/subject and returns a short-lived access token
3. A small Compute read confirms provisioning access → `Connected` / `Degraded` / `Disconnected`

Existing resources (VPC networks, subnets, DNS zones) are discovered separately by the server-side
[asset inventory](/console/connectors/health-and-inventory).

## What's Stored

- **GCP Project ID** and **Project Number**
- **WIF configuration** — assembled by the console from those two IDs: the pool + provider audience, the
  service account impersonation URL, and the token-file credential source. The provisioner service account
  is referenced by that impersonation URL; there's no separate key field.

No service account JSON keys. Access is revoked by deleting the WIF pool or unbinding the service account.
