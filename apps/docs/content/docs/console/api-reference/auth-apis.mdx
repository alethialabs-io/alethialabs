# Auth APIs

These endpoints implement the **Device Authorization Grant** (RFC 8628) for CLI authentication. See [Authentication Flows](/concepts/authentication) for the complete flow.

## Device Code Flow

### `POST /api/auth/cli/generate`

Called by the **browser** (authenticated user) to approve a CLI login attempt.

**Requires:** Active browser session (Better Auth session cookie).

**Body:**
```json
{
  "device_code": "uuid-from-cli"
}
```

**Effect:**
- Upserts a `cli_logins` record keyed by the device code, storing the user's profile ID (which signifies approval) and the user's first linked git provider access token (held in the `verification_code` column for the CLI to pick up on exchange)
- Returns `{ "success": true }`

### `POST /api/auth/cli/exchange`

Called by the **CLI** to exchange the approved device code for tokens.

**No authentication required** — the browser must have already approved the device code (its `cli_logins` record must have a `profile_id` set).

**Body:**
```json
{
  "device_code": "uuid-from-cli"
}
```

**Validation:**
- Looks up the `cli_logins` record by device code
- Verifies that `profile_id` is set (meaning the browser approved the login)

**Response:**
```json
{
  "access_token": "jwt...",
  "refresh_token": "jwt...",
  "provider_token": "git-oauth-token",
  "user_email": "user@example.com"
}
```

**Token Details:**
- **Access token**: 1-hour TTL, HS256 signed with `CLI_JWT_SECRET`
- **Refresh token**: 90-day TTL, same signing

JWT payload:
```json
{
  "sub": "user-uuid",
  "email": "user@example.com",
  "type": "access",
  "iss": "urn:example:issuer",
  "aud": "urn:example:audience"
}
```

### `POST /api/auth/cli/refresh`

Called by the CLI when the access token expires.

**Body:**
```json
{
  "refresh_token": "jwt..."
}
```

**Response:**
```json
{
  "access_token": "new-jwt..."
}
```

Only a new access token is issued. The refresh token is not rotated or invalidated — the CLI keeps reusing the same one until it expires (90-day TTL).

## OAuth Callback

### `GET /api/auth/[...all]`

Better Auth's catch-all handler owns the OAuth callback at `/api/auth/[...all]` after provider authentication, along with email-OTP verify, session, account linking, and sign-out.

Better Auth persists the provider's access token to the `account` table. The CLI `generate` endpoint later reads it from there (via `auth.api.getAccessToken`) to hand the git provider token to the CLI.
