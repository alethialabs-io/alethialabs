# 08 — Security Review & Known Gaps

## Critical Security Issues

### 1. Git tokens stored as plaintext

**Problem:** `vine_repositories` stores `gitops_argocd_token` and `apps_token` as plain TEXT columns. Anyone who can query the vine (via RLS) can read these tokens. They grant write access to Git repos.

**Fix:** Don't store tokens in the vine schema at all. Use one of:

- **Option A: Supabase Vault** — Supabase has `vault.secrets` for encrypted storage. Store a reference ID in `vine_repositories`, actual token in the vault. The worker fetches it at provision time via an RPC.

- **Option B: AWS Secrets Manager** — store tokens alongside the worker's own secrets. The worker reads them at job execution time. The vine table only stores a reference key, never the token.

- **Option C: OAuth app tokens via Git integrations** — the user already connects GitHub/GitLab via OAuth on the Integrations page. Use short-lived OAuth tokens generated at provision time from the stored provider identity. No long-lived tokens stored at all. **This is the modern approach.**

**Recommendation:** Option C for user repos (they've already connected their Git account). For the ArgoCD internal repos, Option B (Secrets Manager) since those are platform-level credentials.

**Schema change:**
```sql
-- Remove from vine_repositories:
--   gitops_argocd_token TEXT,
--   apps_token TEXT,

-- Replace with:
CREATE TABLE public.vine_git_credentials (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  vine_id UUID NOT NULL REFERENCES public.vines(id) ON DELETE CASCADE,
  
  purpose TEXT NOT NULL CHECK (purpose IN ('argocd', 'applications', 'infrastructure')),
  method TEXT NOT NULL CHECK (method IN ('oauth', 'pat', 'deploy_key')),
  
  -- For OAuth: reference to the user's linked Git identity
  provider_identity_id UUID,
  
  -- For PAT/deploy key: reference to encrypted storage (never the actual token)
  secret_ref TEXT,  -- vault key or Secrets Manager ARN
  
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

### 2. No audit trail

**Problem:** No history of who changed what. If someone changes VPC CIDR on a live vine or deletes a database component, there's no record.

**Fix:** Add an audit log table:

```sql
CREATE TABLE public.vine_audit_log (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  vine_id UUID NOT NULL REFERENCES public.vines(id) ON DELETE CASCADE,
  user_id UUID NOT NULL DEFAULT auth.uid(),
  
  action TEXT NOT NULL CHECK (action IN (
    'CREATED', 'UPDATED', 'DELETED', 'PROVISIONED', 'DESTROYED',
    'COMPONENT_ADDED', 'COMPONENT_UPDATED', 'COMPONENT_REMOVED',
    'STATUS_CHANGED'
  )),
  component_type TEXT,        -- 'vpc', 'eks', 'database', etc.
  component_id UUID,
  
  changes JSONB,              -- { field: { old: x, new: y } }
  
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

ALTER TABLE public.vine_audit_log ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users view own audit logs" ON public.vine_audit_log
  FOR SELECT USING (vine_id IN (SELECT id FROM public.vines WHERE user_id = auth.uid()));

-- Worker can insert via service role or a SECURITY DEFINER function
```

**Implementation:** Server actions write an audit log entry alongside every mutation. The worker writes status change entries via the component status API.

### 3. Component status API lacks vine-level authorization

**Problem:** The `PUT /api/vines/{id}/components` route verifies the worker token but doesn't check that the worker is working on a job that belongs to this vine. Any authenticated worker could update any vine.

**Fix:** The route must verify that the worker has an active job (`CLAIMED` or `PROCESSING`) whose `configuration_id` matches the vine_id:

```typescript
// Verify worker has an active job for this vine
const { data: activeJob } = await supabase
  .from("provision_jobs")
  .select("id")
  .eq("worker_id", workerId)
  .eq("configuration_id", vineId)
  .in("status", ["CLAIMED", "PROCESSING"])
  .maybeSingle();

if (!activeJob) {
  return NextResponse.json({ error: "No active job for this vine" }, { status: 403 });
}
```

### 4. No transaction safety on vine creation

**Problem:** `createVine` does `Promise.all` for component inserts. If the vine insert succeeds but a component insert fails, the vine exists without all its components.

**Fix:** Use a Supabase RPC (database function) for atomic vine creation:

```sql
CREATE OR REPLACE FUNCTION public.create_vine_with_components(
  p_vine JSONB,
  p_vpc JSONB,
  p_eks JSONB,
  p_dns JSONB,
  p_repos JSONB,
  p_databases JSONB DEFAULT '[]',
  p_caches JSONB DEFAULT '[]',
  p_queues JSONB DEFAULT '[]'
)
RETURNS UUID
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_vine_id UUID;
BEGIN
  -- All inserts happen in one transaction
  INSERT INTO vines (...) VALUES (...) RETURNING id INTO v_vine_id;
  INSERT INTO vine_vpc (vine_id, ...) VALUES (v_vine_id, ...);
  INSERT INTO vine_eks (vine_id, ...) VALUES (v_vine_id, ...);
  -- etc.
  RETURN v_vine_id;
END;
$$;
```

## Schema Gaps

### 5. Missing `updated_at` on component tables

All component tables only have `created_at`. Add `updated_at` with a trigger:

```sql
-- Add to every component table:
updated_at TIMESTAMPTZ NOT NULL DEFAULT now()

-- Trigger to auto-update:
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Apply to each table:
CREATE TRIGGER set_updated_at BEFORE UPDATE ON vine_vpc
  FOR EACH ROW EXECUTE FUNCTION update_updated_at();
-- (repeat for all component tables)
```

### 6. No uniqueness constraint on component names

Two databases both named "app-db" in the same vine would be confusing.

```sql
-- Add to all 1:N component tables:
ALTER TABLE vine_databases ADD CONSTRAINT vine_databases_name_unique UNIQUE (vine_id, name);
ALTER TABLE vine_caches ADD CONSTRAINT vine_caches_name_unique UNIQUE (vine_id, name);
ALTER TABLE vine_queues ADD CONSTRAINT vine_queues_name_unique UNIQUE (vine_id, name);

ALTER TABLE vine_ecr_repos ADD CONSTRAINT vine_ecr_name_unique UNIQUE (vine_id, name);
ALTER TABLE vine_secrets ADD CONSTRAINT vine_secrets_name_unique UNIQUE (vine_id, name);
```

### 7. `vine_full` view doesn't work for 1:N components

The backward compat view JOINs 1:1 tables fine, but for `vine_databases` (1:N), a JOIN produces multiple rows per vine. The current view ignores 1:N components and uses `EXISTS()` subqueries for booleans only.

**Impact:** The Go provisioner via `config_snapshot` gets a flat row without the individual database/queue/topic details. For the provisioner to handle multiple databases, it needs a separate field in the config_snapshot — e.g. `databases: [...]` as a JSONB array.

**Fix for config_snapshot creation:**
```typescript
// When creating a provision_jobs entry, build the snapshot differently:
const snapshot = {
  ...vineFull,  // flat fields from view (1:1 components)
  databases: await supabase.from("vine_databases").select("*").eq("vine_id", vineId),
  caches: await supabase.from("vine_caches").select("*").eq("vine_id", vineId),
  queues: await supabase.from("vine_queues").select("*").eq("vine_id", vineId),
  topics: await supabase.from("vine_topics").select("*").eq("vine_id", vineId),
  dynamodb_tables: await supabase.from("vine_dynamodb_tables").select("*").eq("vine_id", vineId),
  ecr_repos: await supabase.from("vine_ecr_repos").select("*").eq("vine_id", vineId),
  secrets: await supabase.from("vine_secrets").select("*").eq("vine_id", vineId),
};
```

### 8. Soft delete convention

When a user removes a database from their config, should we hard delete the row or keep it as `DESTROYED`?

**Convention:**
- **Before provisioning (status=PENDING):** hard delete — it was never created in AWS
- **After provisioning (status=ACTIVE):** set to `DESTROYING` → worker tears it down → set to `DESTROYED`. Keep the row for audit purposes. A cleanup job can delete `DESTROYED` rows after 30 days.

## Provisioner Gaps

### 10. Partial Terraform failure

**Problem:** Terraform applies all resources in one `terraform apply`. If VPC succeeds but EKS fails, the worker currently marks the entire job as FAILED. But the VPC exists in AWS.

**Fix:** After a failed `terraform apply`, run `terraform output` to check what was created. Parse the outputs and mark individual components accordingly:
- Output `eks_cluster_name` exists → EKS is ACTIVE
- Output `rds_cluster_endpoint` exists → Database is ACTIVE
- No output for a component → it FAILED

The error message from Terraform usually names the failing resource, which can be mapped to a component.

### 11. No timeout handling

**Problem:** If Terraform hangs (e.g. waiting for an EKS cluster that never becomes ready), the job runs forever.

**Fix:** Add a context timeout in the worker:
```go
ctx, cancel := context.WithTimeout(ctx, 45*time.Minute)
defer cancel()
```

If the timeout fires, mark the job as FAILED with "Provisioning timed out after 45 minutes". The `recover_stale_jobs()` RPC also catches this after 15 minutes of no heartbeat.

### 12. `vine_id` not in `provision_jobs` table

The current `provision_jobs` table has `configuration_id` (referencing the old `configurations` table). With the new schema, this should reference `vines.id`.

**Fix:** Add `vine_id` to provision_jobs (or rename `configuration_id` to `vine_id`).

## Infracost Gaps

### 13. No rate limiting / caching for pricing API

**Problem:** Rapid form changes could flood the Infracost pricing API.

**Fix:**
- **Debounce** in the form — don't call the pricing action on every keystroke, wait 500ms after the user stops typing
- **Server-side cache** — cache pricing data per (resource_type, region) for 1 hour. AWS prices change very rarely.

### 14. Cost API route not documented

The `PUT /api/vines/{id}/cost` route needs the same worker auth as the component status route, plus validation that the worker has an active job for this vine.

## Template Gaps

### 15. Docker build context

The Dockerfile currently builds from `apps/grape/` as the context. If we embed `packages/templates/`, the build context needs to be the repo root:

```dockerfile
# Build from repo root:
docker build -f apps/grape/Dockerfile -t grape:latest .
```

Or add a pre-build step in the Dockerfile:
```dockerfile
COPY packages/templates/ /app/internal/assets/terraform/templates/
COPY packages/modules/ /app/internal/assets/terraform/modules/
```

### 16. Public Terraform modules need internet

The templates reference public modules (`terraform-aws-modules/vpc/aws`, `terraform-aws-modules/iam/aws`, etc.). `terraform init` downloads these from the Terraform registry. The Fargate container has outbound internet access, so this works — but it means `terraform init` is not fully offline even with embedded templates.

For full offline support (future): use `terraform init` with a local mirror or bundle `.terraform/` in the image.

## Migration Gaps

### 17. Table name inconsistencies

The migration script (03) references `vine_database` (singular) and `vine_redis` (singular), but the schema (02) uses `vine_databases` (plural) and `vine_caches`. Need to pick one convention.

**Convention:** Use **plural for 1:N tables**, **singular for 1:1 tables**:
- `vine_vpc` (1:1)
- `vine_eks` (1:1)
- `vine_dns` (1:1)
- `vine_repositories` (1:1, plural because it groups multiple repo configs)
- `vine_databases` (1:N)
- `vine_caches` (1:N)
- `vine_queues` (1:N)
- `vine_topics` (1:N)
- `vine_dynamodb_tables` (1:N)
- `vine_ecr_repos` (1:N)
- `vine_secrets` (1:N)

### 18. No rollback plan

If the migration breaks production:
1. The old `configurations` table is untouched — all existing functionality keeps working
2. Drop the new tables and view
3. Revert the Trellis code to use `configurations` (git revert)

Since we're running parallel tables, rollback is just "delete the new stuff."
