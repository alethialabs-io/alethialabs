Еволюция

От Tendril Agent към Runner + GitOps

Преди: Tendril Agent (v1)

  • In-cluster polling agent — изисква работещ клъстер
  • Проблем „кокошка и яйце“ — не може да има агент без клъстер
  • Локално изпълнение на Terraform — зависимости на машината
  • Ограничена скалируемост и видимост

Сега: Fargate Runner + ArgoCD (v2)

  • Централен Runner в Fargate — не зависи от потребителски клъстер
  • Cross-account IAM Role — провизира в чужд AWS акаунт
  • ArgoCD за Day 2+ — GitOps непрекъснатост след bootstrap
  • Job Queue с атомарно claim_next_job()

Ключова промяна: потребителят не инсталира нищо локално. Свързва AWS акаунт, създава конфигурация, и Runner провизира автоматично.

Control Plane

Trellis — Централизирано управление

Интеграции и идентичности

  • Integrations — единна страница за Git и Cloud доставчици
  • Multi-provider OAuth — GitHub, GitLab, Bitbucket едновременно
  • AWS CloudFormation — Cross-Account IAM Role с External ID
  • integrations каталог — метаданни в базата данни

Провизиране и мониторинг

  • provision_jobs — опашка с config snapshot и атомарен claim
  • Runners — страница за статус, heartbeat и job history
  • Realtime Logs — WebSocket стрийминг на Terraform изход
  • Vineyards — Estate Map визуализация на топология

RLS: дори SELECT * FROM provision_jobs връща само записите на текущия потребител. Сигурността е в базата данни.

provider_tokens: персистентен „портфейл“ за Git токени с trigger-базирана синхронизация от auth.identities.

Runner система

Fargate Runner — Облачно провизиране

Жизнен цикъл

  • REGISTERED → DEPLOYING → ONLINE
  • Heartbeat на всеки 30 секунди
  • Poll за задачи на всеки 10 секунди
  • Атомарен claim_next_job() — без дублиране
  • Автоматично възстановяване на „осиротели“ задачи

Инфраструктура на Runner

  • ECS Fargate — безсървърен контейнер
  • ECR — Docker image registry
  • IAM Roles — task + execution + cross-account
  • S3 — Terraform Remote State
  • CloudWatch — лог групи за отстраняване на дефекти

Cross-Account: Runner работи в акаунта на платформата, но поема IAM роля в акаунта на потребителя чрез STS AssumeRole с External ID. Потребителят запазва пълен контрол.

Сигурност

Zero-Key архитектура

Платформата не съхранява постоянни AWS ключове. Всеки достъп е временен, одитируем и отменяем.

Слой 1

BYOC модел

Cross-Account IAM Role с External ID. Потребителят контролира достъпа и може да го отнеме по всяко време.

Слой 2

Device Flow

RFC 8628 за CLI автентикация. Refresh Token криптиран локално. Без пароли в терминала.

Слой 3

Row Level Security

Политики в PostgreSQL, не в приложния код. Изолация между потребителите на ниво база данни.

Слой 4

Временни кредити

STS AssumeRole за всяко действие. Без дълготрайни Access Keys. IRSA за Pod-level достъп.

Демонстрация

End-to-End сценарий

1

grape login

Device Flow — отваря браузър, потребителят се автентикира в Trellis, CLI получава JWT токен.

2

runner register

CLI деплойва Fargate Runner — ECR image push, ECS задача, IAM роли, heartbeat потвърждение.

3

Trellis конфигурация

Потребителят създава Vine — избира регион, VPC, EKS параметри, Git хранилище.

4

grape harvest

CLI създава provision_job с config snapshot. Задачата влиза в опашката и чака Runner.

5

Runner провизира

Runner поема задачата чрез claim_next_job(), AssumeRole в клиентския акаунт, изпълнява Terraform.

6

Realtime логове

Terraform stdout/stderr се стриймват в Trellis чрез Supabase Realtime. Потребителят вижда прогреса.

Подобрения над Legacy

  • Без зависимости — Go бинарен файл вместо Python + venv
  • Runner модел — няма нужда от локален Terraform
  • Интерактивен TUI — вместо 30+ CLI флага
  • Централен портал — видимост за всички проекти

Принцип: Legacy CLI е поведенчески контракт, не финална архитектура. Модернизираме, не портваме 1:1.
