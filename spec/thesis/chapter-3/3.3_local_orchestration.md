# 3.3. Равнина на Изпълнение А: Локална Оркестрация (Grape CLI)

Докато контролният панел (Trellis) управлява "намерението" (Intent), Grape CLI е инструментът, който превръща това намерение в реалност, особено в началните фази на проекта. Написан на Go, той предоставя мощен и преносим интерфейс за взаимодействие с платформата, без да изисква сложна инсталация на зависимости.

## 3.3.1. Архитектура на CLI инструмента

Grape CLI е изграден върху утвърдената библиотека **Cobra**, която предоставя структурата на командите и подкомандите (напр. `grape deploy`, `grape config create`). Въпреки това, ключовата иновация е в потребителското изживяване (UX).

### Интерактивен TUI (Terminal User Interface)
Вместо да задължаваме потребителя да чете дълги страници с документация за флагове (Flag Hell), Grape внедрява интерактивни съветници (Wizards), базирани на **Charmbracelet/Huh**.

*   **Динамични Форми:** При стартиране на `grape bootstrap`, CLI-то сканира AWS акаунта на потребителя и предлага падащо меню с налични VPC мрежи, вместо да иска ръчно въвеждане на ID.
*   **Визуална обратна връзка:** Използването на **Bubble Tea** (Elm Architecture за Go) позволява анимирани спинъри и прогрес барове по време на дълги операции, създавайки усещане за модерно десктоп приложение в терминала.

## 3.3.2. Командата `bootstrap` (The Seed)

Най-критичният момент за всяка платформа е "Ден 0" – преходът от празен AWS акаунт към управлявана Kubernetes среда. Този процес се управлява от командата `grape bootstrap`.

### Защо локално изпълнение?
За разлика от последващите деплоймънти, които се управляват от агента, първоначалното "засяване" (Seeding) се извършва локално на машината на разработчика.
1.  **Избягване на "Кокошката и Яйцето":** Нямаме агент в облака, защото все още нямаме клъстер, където да работи този агент.
2.  **Compute & Timeout Limits:** Провизирането на EKS клъстер отнема 15-20 минути. Това е твърде дълго за стандартна HTTP заявка или Serverless функция. Локалният процес е устойчив и не струва пари за изчислителна мощ на доставчика на платформата.

### Механизъм на Вградените Ресурси (Embedded Assets)
За да работи CLI-то "out of the box", всички необходими Terraform шаблони и Helm чартове са компилирани директно в бинарния файл чрез `embed` пакета на Go.

```go
// apps/cli/internal/assets/embed.go
//go:embed all:terraform/seed
var SeedTerraform fs.FS

//go:embed all:helm/tendril
var TendrilChart fs.FS
```

По време на изпълнение, `bootstrap` командата:
1.  Разархивира виртуалната файлова система в работна директория (`~/.grape/workspaces/project-env`).
2.  Генерира `terraform.tfvars` файл с избраните от потребителя параметри.
3.  Изтегля автоматично (ако липсва) Terraform бинарния файл от HashiCorp.
4.  Изпълнява `init` и `apply`.

## 3.3.3. Инсталиране на Агента

След като EKS клъстерът е активен, CLI-то преминава към фазата на свързване.

1.  **Registration:** Изпраща заявка към API на Trellis за регистриране на новия клъстер. Сървърът връща `cluster_id` и еднократен `agent_token`.
2.  **Helm Install:** Използвайки локално конфигурирания `kubectl` context, CLI-то инсталира Tendril агента чрез вградения Helm чарт.
3.  **Secrets Injection:** Токенът се подава като Secret към Helm релийза, гарантирайки, че той никога не се записва на диск в чист вид.

```bash
# Псевдо-код на изпълняваната команда
helm upgrade --install tendril ./charts/tendril \
  --set config.clusterId="uuid..." \
  --set config.apiToken="secret-token..."
```

4.  **Handover:** CLI-то започва да "полва" (poll) API-то за статус на клъстера. В момента, в който агентът стартира и изпрати първия си Heartbeat (`ONLINE`), CLI-то докладва успех и приключва своята работа. Оттук нататък управлението се поема от Tendril.
