# 3.2. Контролен Панел "Trellis": Управление на Данни и Идентичност

Контролният панел (Control Plane) е централната нервна система на Grape. Неговата роля е да оркестрира процесите, да управлява състоянието и да осигурява сигурна среда за взаимодействие между потребителите и техните отдалечени клъстери. Тази секция разглежда как базата данни е моделирана, за да поддържа асинхронна комуникация и стриктна изолация на достъпа.

## 3.2.1. Моделиране на Данните (Database Schema)

Базата данни (PostgreSQL) не е просто хранилище на информация, а активен участник в логиката на приложението чрез използване на опашки и събития.

### Изолация на Идентичността
В традиционните системи, идентификацията и конфигурацията често са смесени. Trellis въвежда ясно разграничение:
*   **`auth.users` (Supabase):** Съдържа потребителския профил (Email, UUID).
*   **`cloud_identities` (Trellis):** Нова таблица, която съхранява връзката с облачния доставчик (AWS/GCP/Azure). Тя позволява на един потребител да управлява множество облачни акаунти (Many-to-Many връзка).

```sql
CREATE TABLE cloud_identities (
    id UUID PRIMARY KEY,
    user_id UUID REFERENCES auth.users(id),
    provider TEXT CHECK (provider IN ('aws', 'azure', 'gcp')),
    credentials JSONB -- Съхранява Role ARN и External ID
);
```

### Провизиране като Опашка (The Provisions Queue)
Комуникацията между уеб интерфейса и агента е асинхронна. Таблицата `provisions` действа като опашка (Queue) за задачи.
*   **Config Snapshot:** Когато потребителят натисне "Deploy", системата не просто препраща текущата конфигурация. Тя създава "снимка" (`config_snapshot`) на състоянието в този момент. Това гарантира, че историческите логове винаги ще отговарят на конфигурацията, с която са били генерирани, дори ако потребителят промени настройките по-късно.
*   **Status Machine:** Полето `status` управлява жизнения цикъл: `QUEUED` -> `PROCESSING` -> `SUCCESS` / `FAILED`.

### Проблемът с OAuth Токените (Multi-Provider Auth)
Едно от ключовите предизвикателства беше работата с множество Git доставчици. Стандартният OAuth поток в Supabase замества `provider_token` в сесията при всяко влизане. Това означава, че ако потребителят влезе с GitHub, той губи достъп до своите GitLab хранилища.

За да решим това, създадохме таблица `provider_tokens`, която действа като персистентен "портфейл" за токени:
1.  При успешно влизане (Callback), токенът се прихваща и записва/обновява в `provider_tokens`.
2.  При нужда от достъп до хранилище, системата първо проверява тази таблица, вместо да разчита само на активната сесия.

## 3.2.2. Сигурност и RLS (Row Level Security)

Сигурността е вградена в нивото на данните чрез **Row Level Security (RLS)**. Това предотвратява възможността грешка в API слоя (Next.js) да доведе до изтичане на данни между потребителите (Multi-tenancy isolation).

```sql
-- Пример: Потребител вижда само логовете от своите клъстери
CREATE POLICY "Users can view logs for their provisions" 
    ON public.provision_logs FOR SELECT 
    USING (
        EXISTS (
            SELECT 1 FROM public.provisions
            JOIN public.clusters ON clusters.id = provisions.cluster_id
            WHERE provisions.id = provision_logs.provision_id
            AND clusters.user_id = auth.uid()
        )
    );
```

Тази политика гарантира, че дори SQL заявка `SELECT * FROM provision_logs` ще върне само записите, принадлежащи на текущо логнатия потребител.

## 3.2.3. AWS Onboarding (BYOC Flow)

Моделът "Bring Your Own Cloud" (BYOC) изисква сигурен начин за делегиране на права без споделяне на постоянни ключове (Access Keys).

### Cross-Account IAM Roles
Trellis използва индустриалния стандарт за делегиране чрез **IAM Roles**.
1.  **Generate:** Платформата генерира уникален `ExternalId` за всяка сесия.
2.  **Trust:** Потребителят създава роля в своя AWS акаунт, която има доверие (Trust Policy) към AWS акаунта на Trellis, *само* ако заявката съдържа правилния `ExternalId`.
3.  **Assume:** Агентът или CLI инструментът използват `sts:AssumeRole`, за да получат временни кредиеншъли с ограничени права.

Този подход елиминира риска от компрометиране на дълготрайни тайни и позволява на потребителя да отнеме достъпа по всяко време, просто изтривайки ролята в своя акаунт.

### CLI Аутентикация (Device Flow)
Тъй като CLI инструментът няма браузър, имплементирахме **Device Authorization Grant** (RFC 8628).
1.  CLI генерира код и кара потребителя да го отвори в браузъра.
2.  След потвърждение, сървърът сигурно прехвърля Refresh Token към CLI клиента.
3.  Този токен се съхранява криптиран локално и се използва за подписване на всички последващи заявки.
