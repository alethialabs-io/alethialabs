# 3.2.4. Фронтенд Архитектура и Потребителско Изживяване (Trellis UI)

Потребителският интерфейс на Trellis (`apps/web`) е изграден върху **Next.js 16**, използвайки най-новите парадигми в React екосистемата за постигане на висока производителност и интерактивност.

### React Server Components & Server Actions
Вместо традиционния REST API модел, Trellis разчита на **Server Actions** за мутации на данни.
*   **Директен достъп:** Сървърните компоненти четат директно от базата данни, елиминирайки нуждата от `useEffect` и зареждащи състояния (spinners) за първоначалното визуализиране на съдържанието.
*   **Type Safety:** Типовете на данните се споделят автоматично между клиента и сървъра, гарантирайки, че промените в схемата на базата данни се отразяват незабавно във фронтенда.

### Машина за Конфигурации (Configuration Wizard)
Един от най-сложните компоненти е "Съветникът за конфигурация" (`apps/web/components/configuration-form.tsx`).
*   **State Management:** Използва `react-hook-form` за управление на състоянието на сложни форми с десетки полета и зависимости.
*   **Validation:** Схемата **Zod** (`publicConfigurationsInsertSchema`) валидира входа в реално време, предотвратявайки изпращането на невалидни конфигурации към бекенда.
*   **Dynamic Inputs:** Формата реагира на контекста – например, изборът на "AI Workloads Ready" шаблон автоматично настройва `node_groups` с GPU инстанции в генерирания Terraform код.

### Генериране на Артефакти в Движение (On-the-fly Generation)
За разлика от статичните файлове, "Grape" генерира инфраструктурните пакети динамично при поискване.
*   **ZIP Streaming:** Route Handler-ът `/api/download/zip` използва библиотеката `jszip`, за да комбинира Terraform шаблоните с потребителската конфигурация (генерирана като `terraform.tfvars`) и инсталационните скриптове.
*   **Memory Efficiency:** Целият процес се случва в паметта на сървъра (Edge/Serverless function), без да се записват временни файлове на диск, което подобрява сигурността и скоростта.

### Реално Време (Real-time Feedback Loop)
За да се осигури усещане за "жива" система по време на деплоймънт:
1.  Клиентът се абонира за промени в таблицата `deployment_logs` чрез **Supabase Realtime** (WebSockets).
2.  Когато агентът (Tendril) запише нов ред в лога, базата данни изпраща събитие към клиента.
3.  React компонентът (`log-streamer.tsx`) добавя новия ред към терминалния прозорец, създавайки ефект на стрийминг без постоянно опресняване на страницата.
