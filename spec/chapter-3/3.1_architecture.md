# 3.1. Архитектура и Организация на Кода (The Foundation)

Основата на платформата "Grape" е изградена върху принципите на модерното софтуерно инженерство, целящи мащабируемост, поддръжка и бърз цикъл на разработка. За да се постигне това в среда, която комбинира различни технологии (Polyglot environment), е избран подходът на монохранилище (Monorepo), управлявано от инструменти от ново поколение.

## 3.1.1. Монохранилище (Monorepo Strategy)

Всички компоненти на системата – от потребителския интерфейс до системните агенти и инфраструктурните шаблони – се съхраняват в едно Git хранилище. Това решение не е просто организационно, а стратегическо, позволяващо:
1.  **Atomic Commits:** Промени, засягащи едновременно API договора и неговите консуматори (CLI, Web), се извършват в една транзакция.
2.  **Shared Logic:** Директно споделяне на TypeScript дефиниции, UI компоненти и конфигурации без нужда от публикуване на външни пакети.
3.  **Unified Versioning:** Цялата платформа се движи заедно, елиминирайки проблемите с несъвместими версии на микросървисите.

### Управление на Зависимостите с Turborepo и PNPM
За управление на този сложен граф от зависимости се използва **Turborepo** в комбинация с **PNPM Workspaces**.

*   **Pnpm Workspaces:** Осигурява ефикасно инсталиране на пакети чрез симлинкове (symlinks), спестявайки дисково пространство и време.
*   **Turborepo Pipeline:** Дефинира "интелигентен" процес на изграждане. Чрез файла `turbo.json`, системата разбира зависимостите между задачите. Например, задачата `build` зависи от успешното приключване на `build` на всички пакети, от които зависи даденият компонент.

```json
// turbo.json (съкратен)
{
  "pipeline": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": [".next/**", "dist/**"]
    },
    "dev": {
      "cache": false,
      "persistent": true
    }
  }
}
```

Тази конфигурация позволява **Remote Caching**. Ако разработчик А вече е изградил библиотека `@repo/ui`, разработчик Б (или CI сървърът) ще изтегли готовия артефакт от кеша за части от секундата, вместо да го компилира наново.

### Структура на Проекта
Йерархията е разделена на приложения (`apps`) и споделени пакети (`packages`):

*   **`apps/web`**: Контролният панел (Next.js).
*   **`apps/cli`**: Инструментът за команден ред (Go).
*   **`apps/tendril`**: Агентът за изпълнение (Go).
*   **`packages/ui`**: Библиотека с React компоненти (ShadCN/UI), използвана от уеб приложението и документацията.
*   **`packages/typescript-config`**: Централизирани TS конфигурации (`base.json`, `nextjs.json`), гарантиращи строга типизация навсякъде.
*   **`packages/templates`**: "Златните шаблони" за Terraform, които се консумират от CLI и Tendril.

## 3.1.2. Технологичен Стек

Изборът на технологии е продиктуван от специфичните изисквания на всеки слой от архитектурата.

### Frontend & API: Next.js 16
За уеб слоя се използва **Next.js 16** с архитектурата **App Router** и **React Server Components (RSC)**.
*   **Защо RSC?** Позволява директен достъп до базата данни в сървърните компоненти, елиминирайки нуждата от междинен API слой за прости операции за четене. Това намалява латентността и опростява кода.
*   **UI Framework:** Tailwind CSS и Radix UI (чрез ShadCN) осигуряват достъпен и лек интерфейс.
*   **Validation:** Библиотеката **Zod** се използва за дефиниране на схеми, които валидират данните както на входа (форми), така и на изхода (API).

### Database: PostgreSQL & Supabase
Данните се съхраняват в PostgreSQL, управляван от Supabase.
*   **Auth:** Вградената authentication система (GoTrue) се грижи за управлението на потребители и OAuth потоци.
*   **Realtime:** Supabase Realtime (базиран на PostgreSQL Replication logs) е критичен компонент за "живите" логове на платформата, позволявайки на уеб клиента да получава събития (напр. нов лог от Terraform) мигновено чрез WebSocket.

### Agent & CLI: Go (Golang)
За компонентите, които работят върху инфраструктурата на потребителя, е избран езикът **Go**.
*   **Статично компилиране:** Go създава единичен бинарен файл без външни зависимости (като Python interpreter или Node.js runtime). Това е критично за CLI инструмента и агента, които трябва да работят надеждно на всякакви Linux/macOS машини.
*   **Concurrency:** Вградените goroutines позволяват на Tendril агента да обработва стрийминг на логове, heartbeat сигнали и изпълнение на процеси паралелно с минимален разход на ресурси.
*   **Екосистема:** Използват се библиотеки като **Cobra** (за CLI команди), **Viper** (за конфигурация) и **Charmbracelet** (за модерен Terminal User Interface).

### Infrastructure as Code: Terraform
Въпреки наличието на по-нови инструменти (като Pulumi), **Terraform** е избран като стандарт заради своята зрялост и дефакто статут в индустрията. Платформата генерира динамично `.tfvars` файлове, но самите шаблони са чист HCL, което улеснява одита и разширяването от страна на крайния потребител.
