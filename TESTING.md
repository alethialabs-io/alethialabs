# Testing guide

How we test Alethia, what each layer is for, and how to run and write tests. We follow the
**Testing Trophy** — the model best suited to a full-stack app whose business logic lives in
server actions over a database (vs the classic pyramid, which assumes a logic-heavy monolith).

```
            ┌───────────────────────┐
            │        E2E            │  few — only critical user journeys (Playwright)
          ┌─┴───────────────────────┴─┐
          │      Integration          │  the bulk — server actions + real-Postgres
          │   (actions, SQL, RLS)     │  queries/RLS; highest confidence per test
        ┌─┴───────────────────────────┴─┐
        │           Unit               │  pure logic — fast, deterministic, no I/O
      ┌─┴───────────────────────────────┴─┐
      │      Static analysis              │  tsc --noEmit + Biome/ESLint (already gating)
      └───────────────────────────────────┘
```

Why the trophy and not the pyramid: most of our logic is orchestration over Postgres + Stripe
+ auth. A test that exercises a real query or a whole server action catches far more than a
hundred mocked unit tests, so **integration is the widest band**. E2E stays deliberately small
(slow, flaky-prone) and covers only journeys a user would notice breaking.

## Layers at a glance

| Layer | Runner | Lives in | Mocks? | Runs in CI |
|---|---|---|---|---|
| **Static** | `tsc`, Biome/ESLint | whole repo | — | yes (gate) |
| **Unit** | Vitest (jsdom) | `apps/console/tests/{lib,validations}/**` | no I/O at all | yes |
| **Component** | Vitest + React Testing Library | `apps/console/tests/components/**` | server actions mocked | yes |
| **Action (mocked)** | Vitest | `apps/console/tests/actions/**` | DB/Stripe/auth mocked at the module boundary | yes |
| **Integration (real DB)** | Vitest (`vitest.integration.config.ts`) | `apps/console/tests/integration/**` | real Postgres, nothing else | gated (needs Postgres) |
| **Shared package** | Vitest | `packages/<pkg>/tests/**` (e.g. `@repo/ui`, `@repo/plan-catalog`) | per-package | yes (turbo fan-out) |
| **E2E** | Playwright | `apps/console/e2e/**` | real app, dev stack | local-only (for now) |
| **Go** | `go test ./...` | `apps/{cli,runner}`, `packages/core` | `httptest` | yes (matrix) |

## Running tests

```bash
# Every project's test suite, with coverage (the CI gate; fans out via turbo).
pnpm exec turbo run test

# A single project (console app, or a shared package)
pnpm -F console test
pnpm -F @repo/ui test
pnpm -F @repo/plan-catalog test

# Same, watch mode, no coverage (fast feedback loop)
pnpm -F console test:watch

# Integration suite — needs Postgres up first
pnpm db:up                       # Postgres + migrations (from repo root)
pnpm -F console test:integration

# End-to-end (browser). One-time: install the Playwright browsers.
pnpm -F console exec playwright install chromium
# Needs the dev stack running (console on :3000, SES unconfigured so the OTP is logged):
pnpm dev:up
pnpm -F console test:e2e         # or test:e2e:ui for the Playwright UI

# Go
cd apps/runner && go test ./...
```

> **Note on the legacy specs.** `e2e/{dashboard,sidebar,runners,jobs,connectors,
> account-settings,design-project}.spec.ts` predate the C2 `/{org}/…` routing and target the
> retired `/dashboard/*` paths — they need a selector/route revalidation pass against the running
> app. The auth fixture (`e2e/fixtures/auth.ts`) and `e2e/usage.spec.ts` are on the current flow.

## Coverage

- Generated by the unit run (`pnpm -F console test` → `vitest run --coverage`, v8 provider).
- Reports: terminal table, `apps/console/coverage/index.html` (browse locally),
  `coverage/lcov.info`, `coverage/coverage-summary.json`.
- Scope: our business **logic** — `lib/**` + `app/server/actions/**` (generated migrations/seeds
  and tests excluded; see `vitest.config.ts`). Presentational `components/**` are deliberately
  **not** counted here — UI is covered by the focused `@repo/ui` component tests + e2e, so the
  badge reflects "is our logic tested" rather than being dragged down by untested view code.
- **Self-hosted badge (no third-party service):** the `coverage-badge` CI job (push-to-main)
  runs the suites, merges the per-project `coverage-summary.json` line totals via
  `scripts/coverage-badge.mjs`, and commits `.github/badges/coverage.json` — the shields.io
  endpoint the README badge reads. Regenerate locally with `pnpm exec turbo run test &&
  node scripts/coverage-badge.mjs`.
- **Thresholds are report-only today and ratcheted up as suites land.** When a layer is
  meaningfully covered, raise the floor in `vitest.config.ts → test.coverage.thresholds` so it
  can't regress.

### What the two badges actually count (be honest)

| Badge | Counts | Does **not** count |
|---|---|---|
| **coverage** | console **logic** (`lib/**`, `app/server/actions/**`) + `@repo/ui` src + `@repo/plan-catalog` | console `components/**` (presentational), the marketing/docs/blog apps, `@repo/{email,brand}` |
| **go coverage** | aggregate of `apps/cli` + `apps/runner` + `packages/core` (`go test -coverprofile`, via `scripts/go-coverage-badge.mjs`) | — |

Both are **honest, not cherry-picked-high** — they climb as more logic gets tested. View/content code is covered by component tests + e2e, not these badges. Regenerate the Go badge locally with: `for m in apps/cli apps/runner packages/core; do (cd $m && go test -coverprofile=cover.out ./...); done && node scripts/go-coverage-badge.mjs`.

## Writing tests — recipes

### Unit (pure logic)
No mocks, no I/O. Import the function and assert. Pass any "now"/clock as an argument so tests
are deterministic. Model: `tests/lib/billing/usage.test.ts`, `tests/filters/range.test.ts`.

```ts
import { describe, expect, it } from "vitest";
import { resolvePlanEntitlements } from "@/lib/billing/plan";

it("falls back to community when the subscription isn't live", () => {
  expect(resolvePlanEntitlements("team", "canceled").organizations).toBe(false);
});
```

### Action (mocked boundary)
Mock the I/O modules — DB, Stripe, auth — and assert the action's orchestration and branching.
Pattern: `tests/lib/cli/providers.test.ts` (uses `vi.mock`).

```ts
vi.mock("@/lib/authz/guard", () => ({ currentActor: vi.fn() }));
vi.mock("@/lib/billing/queries", () => ({ getOrgBilling: vi.fn() }));
// ...arrange the mocks, call the action, assert it returned/branched correctly.
```

### Integration (real Postgres)
Lives in `tests/integration/**` and runs via `vitest.integration.config.ts`. Use the shared
`tests/integration/setup.ts` to get a migrated, isolated database; seed rows; call the real
query builder; assert the SQL result. This is the only layer that verifies SQL, the day-bucket
/ clamp math, and **RLS isolation** (one org can't read another's rows).

### Component (RTL)
Render with `@testing-library/react`, drive with `@testing-library/user-event`, mock the
server actions the component imports. Form components: reuse
`tests/components/design-project/test-utils.tsx` (`renderWithForm`).

### E2E (Playwright)
Keep them few and high-value. Auth goes through the real email-OTP flow — the shared fixture
in `e2e/fixtures/auth.ts` requests a code and reads it from the dev server log. Gate anything
needing Stripe behind `test.skip(!process.env.STRIPE_SECRET_KEY)`.

## Test organization — the standard (one rule, no drift)

**The owning workspace project owns its tests, in its own `tests/` directory.** `turbo run
test` fans out across every project that defines a `test` script, and the `coverage-badge` job
merges each project's `coverage-summary.json` into one public number — the same fan-out model
the repo already uses for `lint`/`check-types`.

- **Each project has a `tests/` dir** + a `test` script: `apps/console`, `packages/ui`,
  `packages/plan-catalog`, and any future package. A package's logic and **its tests live
  together** — e.g. `@repo/ui/range` and the filter components are tested in
  `packages/ui/tests/`, not from the console.
- **Tests go in `tests/`, never co-located in `src/`.** For `@repo/ui` a `src/*.test.tsx` would
  be matched by its `"./*": "./src/*.tsx"` export glob and scanned by the apps' Tailwind
  `@source packages/ui/src` + `transpilePackages` — polluting exports/build. A package-level
  `tests/` dir avoids this and keeps the rule uniform with the app.
- **Within a project, mirror the source path** under `tests/` (`lib/billing/plan.ts` →
  `tests/lib/billing/plan.test.ts`; `@repo/ui/src/range.ts` → `packages/ui/tests/range.test.ts`).
- **Naming:** `*.test.ts(x)` = Vitest; `*.spec.ts` = Playwright e2e (`apps/console/e2e/**`).
  The real-DB integration tests are gated by directory (`apps/console/tests/integration/**`) +
  their own config.
- **Go** stays co-located `*_test.go` (the language idiom).

There are **no inline tests next to app source** and no "half here, half there" — every TS test
lives under a `tests/` directory of its owning project.

## Conventions

- One behavior per `it`; name it as the expectation ("rejects a taken slug").
- Deterministic: inject clocks/ids; never assert on `Date.now()` or random output directly.
- Every new pure module ships with a test in the same PR, in its project's `tests/` (mirroring
  the source path).
